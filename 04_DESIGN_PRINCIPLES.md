# 04 — 核心设计原则

> 所属文档集：[00_MASTER_INDEX.md](00_MASTER_INDEX.md)
> 版本：v1.0（2026-06-11）
> 地位：**全员必读**。本文档的 14 条原则（P-01 ~ P-14）是全部后续设计的公理层；任何设计文档、RFC、代码评审都应以编号引用这些原则。原则的修改属于 Full RFC 级变更（[10](10_GOVERNANCE.md) §3）。

---

每条原则包含：陈述、含义边界、证据（来自调研 r1–r12 或上一项目交接 H01–H07）、以及它**否决**什么。

---

## P-01 strict-only：不存在静默降级

**陈述**：任何编译路径的失败（lowering、codegen、能力缺失）都必须是带结构化错误码的编译错误或显式诊断。语言与工具链层面不存在静默 fallback、静默近似、静默跳过。宽容行为只能以显式 `unsafe` 块或显式 feature gate 的形式存在。

**证据**：上一项目的模板 SPIR-V fallback 靠"显式、可诊断、从未删除"三原则与全程纪律压制才未酿成事故（H02 §2）；其严格化只能靠白名单逐站点收缩（5 个站点分两批耗时两个里程碑）。H06 §2 的结论：新语言从第一天就是 strict-only，省掉整个"渐进严格化"阶段。

**否决**：任何"先宽容后收紧"的编译器行为提案；任何无错误码的失败路径；任何"自动回退到较慢但能跑的实现"的运行时设计。

---

## P-02 GPU-first：设备语义是类型系统的一等公民

**陈述**：地址空间（global/shared/constant/local）、执行层级（grid/block/warp/thread）、资源归属（context-bound）不是注释、属性或运行时约定，而是参与类型检查的语言构造。

**证据**：上一项目的痛——"Python 宿主无法表达地址空间/所有权，只能用计数器 + 环境变量模拟"（H06 §2 映射表第一行）。Descend 证明此路线可行且性能无损（r5）；Rust-CUDA 的反例证明不为 GPU 设计的类型系统搬上 GPU 的终态是"全部 unsafe"（r5）。

**否决**：OpenCL generic address space / SYCL 式地址空间推断进入 safe core（r5 点名：弱化 provenance）；把执行层级当运行时参数而非类型的设计。

---

## P-03 安全但不阉割：unsafe 是设计的一部分，不是失败

**陈述**：静态安全包络覆盖结构化模式（资源生命周期、地址空间隔离、规则化分区、barrier 合法性）；包络之外的高性能惯用法（弱序原子协议、运行时索引别名、inline PTX）通过 `unsafe` 显式可达，且每个 unsafe 原语附带成文的验证义务（RustBelt 方法论，r5）。语言永远不以"安全"为由让专家用户失去对硬件的控制。

**证据**：r5 的核心结论——"GPU 上的安全借用理论可行，但安全包络不覆盖全部高性能 CUDA 惯用法"；Descend 的 SSSP 案例；Slang 不暴露 local 指针的取舍。

**否决**：试图静态证明一切的类型系统雄心（工程上不可达）；没有 unsafe 通道的"玩具安全"；无验证义务、无审计注释的 unsafe 使用（见 P-13）。

---

## P-04 能力位由真实探测驱动

**陈述**：任何硬件/驱动/工具链能力（SM 版本、driver 版本、Toolkit 组件、WDDM/TCC/MCDM、HAGS）必须由运行时真实探测或官方查询 API（NVML 优先）驱动，配置与假设不得替代探测。编译器内建 target capability 查询，标准库提供探测 API。

**证据**：上一项目 6 个 Vulkan 扩展能力位全部由 features2 真实探测驱动，"无一例配置说支持、运行时炸"（H04 §1 第 6 行）；r6 的 CUDA 探测细节（13.3 文档仍写 `CUDA 13.2.props`、装了 Toolkit ≠ 有合格驱动）证明假设必然漂移。

**否决**：硬编码版本号/路径/能力表；以编译目标假设替代运行时协商。

---

## P-05 显式优于隐式：内存、拷贝、同步、调度

**陈述**：H2D/D2H 拷贝显式、pinned staging 显式、同步点（event/barrier/fence + scope）显式、kernel launch 配置显式。Unified Memory、零拷贝映射等隐式机制以 opt-in 形式存在并标注其平台语义差异。

**证据**：r4——Windows native 无 full managed memory，UM 默认化是 Linux 思维的陷阱；r11——WDDM batch 扭曲计时、隐式同步是基准与生产性能不可预测的主因；Bend 自动并行路线的市场失败（r10）。

**否决**：默认 UM；自动插入的隐式同步；"编译器替你选 launch 配置"的默认行为（可作为显式工具提供）。

---

## P-06 静态编译与可预测性能

**陈述**：Rurix 是 AOT 静态编译语言。产物是标准 PE/COFF（host）与 PTX/cubin（device）。不依赖运行时 JIT 作为语义组成部分（Driver API 的 PTX JIT 是装载机制，不是语言语义）。性能模型可解释：用户能从源码推断生成代码的形状。

**证据**：JIT 抖动是 Triton/Numba 用户的核心痛点之一（r10）；上一项目对 warmup/JIT 首次运行偏差的处理成本（r11 专节）；静态编译是与全部 Python DSL 竞品的根本差异化。

**否决**：trace-based JIT、运行时特化作为核心语义；"先跑起来再优化"的解释器阶段。

---

## P-07 观测先行：可观测性是语言基础设施

**陈述**：编译器每个阶段（parse/lower/typecheck/borrowck/codegen）与运行时每类资源操作（context/module/stream/buffer/launch）从第一天就带原子计数器与计时桶（rustc `-Z self-profile` 式 + CUPTI Activity 式）。任何优化必须先有观测点，再有实现，最后才有阈值。

**证据**：上一项目最有价值的方法论遗产——90+ 计数器使"所有优化争议变成数据问题"，9 个阶段 0 次"优化了但说不清效果"（H04 §1 第 1 行）；H06 §1 明确要求新项目第一天落地。

**否决**：无观测点的性能优化合入；事后补观测的工作排序。配套硬规则见 P-09 与 [14](14_ENGINEERING_DISCIPLINE.md) §5。

---

## P-08 诊断质量是产品力，不是附属品

**陈述**：诊断体系（span/label/note/help/suggestion + 错误码 + `--explain` + JSON 输出 + MachineApplicable 自动修复分级）与 UI golden 测试从第一个诊断开始建设。诊断数据结构是 LSP 的唯一数据源（单一前端，无双引擎）。

**证据**：r1 的反直觉结论第 8 条——"结构化诊断的 ROI 高于 MIR 优化菜单：对系统编程用户，借用/生命周期错误的解释质量决定语言成败"；r9——rust-analyzer/rustc 双引擎是大团队历史产物，4–6 人必须单前端。

**否决**：先实现后补诊断；独立于编译器的 LSP 语义引擎；无 golden 测试覆盖的诊断变更。

---

## P-09 证据通道先于性能里程碑

**陈述**：任何性能型目标在其证据通道（目标硬件上的 `measured_local` 基准链路）建成之前不得宣布达成。占位阈值（`estimated`）存活不得超过 2 个里程碑，逾期自动降级该里程碑状态。第一个月必须建成 RTX 4070 Ti 上 deviceQuery/bandwidthTest/SAXPY 级真实基线。

**证据**：上一项目最大单点教训（H04 §2.2）——严格阈值依赖的 CI runner 从 P10 deferred 到项目终止都没建成，导致 6 个预算 JSON 几乎全部阈值是占位、所有阶段关闭都带"pending backfill"尾巴。H06 §1 第 4 条的硬规则原文照搬。

**否决**：以"证据后补"为由关闭性能里程碑；连续两个纯骨架阶段（每阶段强制包含一个真实硬件证据交付物，H06 §6）。

---

## P-10 公开面冻结纪律

**陈述**：Rurix 的"公开面"= 语言语法、标准库 stable API、C ABI。公开面的任何变化必须走 RFC + feature gate + conformance 测试三件套；nightly/unstable 特性与 stable 面物理隔离（feature gate 机制，r1）。内部实现（IR、运行时内部结构）自由演进。

**证据**：上一项目"公开 API 字节级冻结 + 内部自由演进"是 16 个阶段没有破坏消费方的根本原因（H02 §1）；Rust 的 feature gate / stabilization / edition 机制是经过验证的载体（r7）。

**否决**：实现先行倒逼语义承诺（"实现可运行 ≠ 语义已承诺"，r7）；绕过 gate 直接进 stable 的捷径。

---

## P-11 单一事实源，生成多视图

**陈述**：规范（spec 条款）、API 文档、错误码索引、状态镜像、绑定代码——凡是存在"多处一致"需求的信息，必须有单一结构化事实源，其余视图由工具生成。手工维护的镜像超过两处即为设计错误。

**证据**：上一项目五段式手写镜像管道失控（`GraphicsPipelineStatus` 100+ 字段全手写）与 6 份文档手工同步的文档税（H04 §2.4/§2.5）；H06 §2 映射表——绑定/状态镜像/FFI 全部由编译器属性 + 代码生成完成。

**否决**：任何新的手写镜像层；以 markdown 互相复制为同步手段的文档体系（本文档集自身的维护规则见 [00](00_MASTER_INDEX.md) §6）。

---

## P-12 范围克制：正式地不做

**陈述**：已识别的诱惑方向（多后端、autodiff、kernel fusion、过程宏、registry、Python 嵌入）必须以 spike gating 形式显式登记"不做"及其触发条件，而不是默默搁置。MVP 范围红线见 [11](11_ROADMAP.md) §2，死亡路线见 [03](03_POSITIONING_AND_LANDSCAPE.md) §4。

**证据**：上一项目 spike gating 机制 4 次正式 not_triggered 全部留痕，多次阻止范围蔓延（H04 §1 第 9 行）；H06 §5 列出的六项"做了骨架但永远没做完"前科。

**否决**：“顺手做个 X"式的范围扩张；无触发条件记录的搁置。

---

## P-13 防 AI 幻觉治理：来源可溯、规范领导实现、agent 完全自主

**陈述**：AI 重度参与是本项目的既定工作形态，AI agent 拥有完全自主行动与决策权限。因此：(a) 规范条款编号 ↔ conformance 测试 ↔ PR 引用强制三角；(b) AI 贡献强制 provenance 标注，AI 自主起草/实现/合入 UB/内存模型/FFI ABI 等高敏面（走 Full RFC 作为留档与可追溯手段，无需自主批准）；(c) 验收数字必须来自命令输出；(d) unsafe 块强制审计注释 + 审核记录；(e) 契约 close-out 只追加。**agent 自主决策落在合入、bless、close-out、状态翻转等全部动作**——agent 可干全部事并自行裁决，无 agent 批准门或无卡点。

**证据**：上一项目实证的三类 AI 漂移与全部有效解药（H06 §4）；Linux kernel 与 LLVM 2025 政策先例（r7）；Ferrocene 可测试规范方法（r7）。

**否决**：无 provenance 的实质性 AI 内容合入；"模型说对就是对"的验收。

---

## P-14 Windows 与 WDDM 是一等环境条件

**陈述**：WDDM/TCC/MCDM 驱动模型、TDR、HAGS、DLL 搜索顺序、Authenticode/SmartScreen——这些 Windows 现实是设计输入而不是兼容性补丁。运行时启动时探测并记录环境画像；基准体系按 WDDM 现实设计（锁频、TDR 预算、计时刷队列）；分发链路按 Windows 原生惯例（MSI/winget/签名/SBOM）。

**证据**：上一项目"Windows 驱动现实被低估"的教训（H04 §2.3，21 个 0xC0000409 永久失败）；r4 的 WDDM 专章；r11 的 Windows 基准专节（未锁频差异 50%+、TDR 默认 2s）；r6 的全套 Windows 工具链结论。

**否决**：Linux-first 假设混入设计（如照搬 CUDA Minor Version Compatibility——Windows 不支持，r6）；"在 Linux 上验证、在 Windows 上祈祷"的 CI 结构。

---

## 原则间的张力与裁决顺序

原则冲突时的优先级裁决（设计评审用）：

1. **P-01（strict-only）与 P-03（不阉割）**：不冲突——unsafe 是显式通道，不是静默宽容。任何"为了让代码能编过而放松检查"的诉求走 unsafe，不走削弱诊断。
2. **P-03（控制力）压过人体工学**：当"更安全的默认"与"专家可达性"冲突时，默认安全 + 显式逃生舱，但逃生舱的表达力不打折。
3. **P-09（证据）压过进度**：里程碑宁可延期，不允许无证据关闭。这是上一项目用 16 个阶段买来的最贵教训。
4. **P-12（克制）压过完整性**：宁可 MVP 功能清单难看，不允许多线骨架并行。
