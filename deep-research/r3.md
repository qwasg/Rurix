# R3 MLIR 在 NVIDIA GPU 编译方向的现状调研

## 核心判断

如果把问题聚焦到“**一门已有自研 MIR、当前走 LLVM/NVPTX 直通的语言，是否该在中间层引入 MLIR**”，结论并不是“MLIR 适合一切 GPU 编译器”，而是更具体的三点。

第一，**upstream MLIR 在 NVIDIA 路线上的分层已经相当清楚**：`gpu` 方言负责目标无关的 kernel/host-device 容器与发射边界，`nvgpu` 方言负责把 `vector`/`gpu` 中适合 Tensor Core、TMA、异步拷贝、barrier 的中层模式映射到 NVIDIA 语义，`nvvm` 方言则是 LLVM-IR-based 的 NVIDIA backend dialect，直接建模 NVVM intrinsic、地址空间与 PTX 级功能。官方文档、默认 NVVM pipeline 源码和 LLVM Dev Meeting 演讲三者在这一点上是相互一致的。citeturn44view1turn44view0turn9view0turn12view1

第二，**官方 `gpu`/`nvgpu`/`nvvm` 体系今天已经足够做“后端 sink”**，但还**不是**“你把任意高层 IR 丢进去就自动得到优秀 NVIDIA kernel”的通用 middle-end。官方 `gpu-lower-to-nvvm-pipeline` 明确要求输入已经是“显式并行”的 IR，本身不负责 GPU parallelization；而 `linalg` 的 tile/fuse/vectorize、bufferization、以及 Tensor Core 相关 pattern 选择，仍然需要在更高层的 `linalg`/`vector`/Transform Dialect 或项目自定义方言里完成。换句话说，**sink 已成熟，通用中层尚不统一**。citeturn8search0turn9view0turn7view0turn8search3

第三，**真实项目的共同实践不是“只用官方 gpu 方言”，而是“在官方方言周围自建语义层”**。IREE 在 upstream MLIR 之上加了 `iree_codegen`/`iree_gpu`；Triton 用 `tt`/`ttg`/`ttng`/`nvws` 等自家方言承载分布式布局、TMA、warp specialization；OpenXLA 在 GPU 路线上使用 `xla_gpu`、RuntimeIR 和 TritonIR；torch-mlir 则止步于 Torch→Linalg/TOSA/StableHLO 的桥接层；Mojo/Modular 公开说自己建立在 **MLIR Core** 之上，甚至明确不使用 `linalg`/`affine`/`scf`。所以现实更像是：**官方方言是公共地基，不是现成整栋楼**。citeturn20view0turn20view1turn20view2turn20view3turn26view2turn28view0turn24view0turn31view1turn31view0turn36view0turn39view0turn39view1

## 方言分层与能力矩阵

先说一个总判断：对于 NVIDIA 路线，**`gpu`/`nvgpu`/`nvvm` 不是三个“并列替代品”，而是三个不同抽象高度的责任边界**。`linalg`/`vector`/`memref` 则不是 GPU 专用方言，但在真正可用的 lowering 路径里承担了绝大部分“可优化编译中层”的工作。`linalg` 文档把其核心变换直接写成 tile、promotion、producer-consumer fusion、map-to-hardware、vectorization、lower to special instructions；bufferization 文档也明确说很多变换“更容易或只支持在 tensor land 完成”，因此 bufferization 通常放在较后阶段。citeturn7view0turn8search3

下表按“责任边界”而不是按“是否 upstream”来组织：

| 层 | 主要职责 | 代表能力与操作 | 典型输入 / 输出 | 适合作为什么 | 依据 |
|---|---|---|---|---|---|
| `gpu` | 目标无关的 kernel 容器、launch、host-device 边界、binary 封装 | 官方 pipeline 围绕 `gpu.module`、kernel outlining、`gpu`→目标 backend lowering；LLVM 2024 Hopper 演讲也把它定位为 kernel launch、cluster launch、driver communication 层 | 输入通常还是较高层并行 IR；输出交给 `nvvm`/`rocdl` 等目标方言 | **发射与容器层**；不是 Tensor Core 专用中层 | citeturn2view3turn9view0turn12view1 |
| `nvgpu` | NVIDIA 专属中层桥接层，把高层 `gpu`/`vector` 模式映射到目标语义 | `device_async_copy/create_group/wait`、`ldmatrix`、`mma.sync`、`mma.sp.sync`、`mbarrier.*`、`tma.create.descriptor`、`tma.async.load/store`、`warpgroup.generate.descriptor`、`warpgroup.mma/init.accumulator/store` | 文档明确说它桥接高层 target-agnostic dialects（GPU、Vector）与低层 NVVM，并且用 `memref` 表示内存、`vector` 表示寄存器/target-specific operands | **Tensor Core / TMA / async copy 的中层语义层** | citeturn44view1 |
| `nvvm` | LLVM-IR-based 的 NVIDIA backend dialect；接近 NVVM intrinsic / PTX | 文档把它定义为 LLVM-IR-based、NVIDIA-specific backend dialect；包含地址空间、inline PTX、`wgmma.*`、`cp.async.bulk.*`、`cp.async.bulk.tensor.*`、`tcgen05.*` 等大量低层 op | 输入来自 `gpu`/`nvgpu`/其他 lowerings；再通过 NVPTX/NVVM 工具链到 PTX / cubin | **最终目标相关 sink** | citeturn44view0turn45view0turn45view2 |
| `linalg` | 结构化 tensor/loop 优化层 | 官方文档把关键变换直接列为：tiling、promotion to fast memory、producer-consumer fusion、map to hardware、vectorization、lower to special instructions/intrinsics | 输入多为张量级计算；输出常走 `vector`/loops/bufferized IR | **真正的通用优化中层** | citeturn7view0 |
| `vector` | 向量级规范化与 tensorization 的承接层 | upstream conversion utilities 明确包含把特定 `vector.contract` 变成 `nvgpu.mma.sync`、把特定 `vector.transfer_read`/shared-memory 访问合法化成 `nvgpu.ldmatrix` 的能力 | 输入来自 `linalg` vectorization；输出给 `nvgpu`/LLVM lowerings | **Tensor Core pattern 识别与张量化承接层**，但覆盖是“特定模式”，不是所有向量 op | citeturn5search5turn6view1 |
| `memref` 与 bufferization | 物理内存与 buffer 语义层 | bufferization 文档明确说 tensor→memref 的转换通常放在较后阶段；`nvgpu` 文档又明确把 `memref` 当作 memory operand 承载层 | tensor land 变换后再进入 memref/buffer land | **shared/global/cluster memory 的实际承载层** | citeturn8search3turn44view1 |

把这些层连起来，**官方认可的路径**不是单一路线，而是一个族：LLVM 2024 Hopper 演讲明确给出 `Vector → NVGPU → NVVM`、`Linalg → NVGPU → NVVM`、`Linalg → Vector → NVGPU → NVVM`，并强调“your dialect”也可以在中途接入 NVGPU。这个表述非常关键，因为它说明 upstream 的设计意图并不是让所有语言都先进入 `gpu` 方言再一路向下，而是允许**从你自己的中层方言直接接 upstream sink**。citeturn11view0turn12view1

我的判断是：对你的场景，`gpu` 更像**launch/container ABI 层**，`nvgpu` 更像**NVIDIA feature bridge**，`nvvm` 更像**最终 backend sink**；真正决定“是否值得引入 MLIR”的，反而是你是否要把 **`linalg`/`vector`/bufferization** 带进体系。因为 kernel 融合、layout 重写、向量/张量 lowering 的主战场，在 upstream 里本来就主要不在 `gpu` 方言本身。citeturn7view0turn8search3turn44view1turn44view0

## 官方下降路径与成熟度

官方默认 NVIDIA sink pipeline 的名字就是 **`gpu-lower-to-nvvm-pipeline`**。其注册说明写得很直白：这是把 `arith`、`memref`、`scf`、`vector`、`gpu`、`nvgpu` 降到 NVVM 的默认方式；先处理 GPU code，再处理 host code。源码里可以直接看到主要步骤：先 `ConvertNVGPUToNVVM`，再 kernel outlining，然后 `VectorToSCF`、`SCFToControlFlow`、`ConvertNVVMToLLVM`、`ConvertFuncToLLVM`、`ExpandStridedMetadata`、attach NVVM target、`LowerAffine`、`ArithToLLVM`、`IndexToLLVM`、canonicalize/CSE；对 `gpu.module` 再做 `ConvertGpuOpsToNVVMOps`；最后 host 侧走 `GpuToLLVMConversionPass`、`GpuModuleToBinaryPass`、`MathToLLVM` 等收尾。citeturn9view0

这条官方路径的**成熟点**很明确：如果你的 IR **已经**是显式并行、并且 GPU kernel 内部的高层优化已经完成，那么 upstream 能提供一条结构清晰、源码公开、可复用的 NVIDIA sink。按官方文档，它会生成 NVVM/LLVM dialect code，并能与 `mlir-runner` 或 execution engine 对接；按 pipeline 源码，它也已经覆盖了 kernel outlining、target attach、binary packaging 等后端基础设施。对“从中层到 NVPTX/NVVM”的最后一公里来说，这不是愿景，而是现成代码。citeturn8search0turn9view0

但这条路径的**边界**同样非常明确。官方文档直接写道：该 pipeline **要求显式并行 IR**，并且**不做 GPU parallelization**。这意味着它不是 polyhedral scheduler，不是 fusion engine，也不是“从普通 loop 自动变成优秀 CUDA kernel”的神奇 pass。`linalg` 之所以存在，正是为了在更高层做 tiling、fusion、promotion、vectorization，再在接近 backend 时 bufferize。也就是说，官方 NVIDIA 路线真正“成熟”的，是 **sink 与 feature exposure**，而不是“一体化 GPU middle-end”。citeturn8search0turn7view0turn8search3

从已有 upstream pass 能力看，这个结论还可以再具体一点：`vector` 到 NVIDIA Tensor Core 的 lowering 目前覆盖的是**特定可识别 idiom**。upstream 的 Vector-to-GPU 相关转换明确提到，它会把某些 `vector.contract` 转成 `nvgpu.mma.sync`，把某些 shared-memory 侧的 `vector.transfer_read` 合法化为 `nvgpu.ldmatrix`。这很好用，但它不是 Triton 那种“以分布式布局为一等公民”的完整中层。真实项目之所以还要自建方言，原因恰恰在这儿：**layout、distribution、pipeline scheduling、autotuning state** 这些信息，在通用 upstream sink 里并没有统一承载位置。citeturn5search5turn6view1turn20view0turn28view0

因此，如果要用一句话概括“官方 pass pipeline 的成熟度”，我会写成：**适合作为标准后端汇聚点，不适合作为你第一版 GPU middle-end 的全部语义承载层**。这个判断既符合官方文档的自我定位，也符合 IREE/Triton/OpenXLA 这些一线使用者的实际分工。citeturn8search0turn9view0turn20view0turn31view1turn28view0

## Tensor Core 与 Hopper Blackwell 建模

就 **Tensor Core / Hopper / Blackwell** 这条主线看，MLIR 的 NVIDIA 建模已经不再停留在 Ampere 时代的 `wmma`/`mma.sync`。upstream `nvgpu` 文档今天公开列出的 op，已经覆盖了 `nvgpu.mma.sync`、`nvgpu.mma.sp.sync`、整套 `warpgroup.*`（包括 descriptor、init accumulator、mma、store），以及 H100/Hopper 非常关键的 `tma.create.descriptor`、`tma.async.load/store`、`tma.fence/prefetch`、`mbarrier.*`。这说明在 **中层语义** 上，Hopper 的 WGMMA/TMA/barrier 不是“项目私货”，而是 upstream 公开接口。citeturn44view1

在更低层的 `nvvm` 方言里，覆盖面还要更宽。当前 NVVM 文档明确包含 `nvvm.wgmma.fence.aligned`、`nvvm.wgmma.mma_async`、`nvvm.wgmma.commit.group.sync.aligned`、`nvvm.wgmma.wait.group.sync.aligned`，也包含 `nvvm.cp.async.bulk.tensor.shared.cluster.global` 这类 Hopper TMA 级 op。换言之，**WGMMA 与 Hopper TMA 在 upstream 低层已经是“显式 op”，不是只能靠 inline PTX 硬塞**。citeturn45view0turn45view2turn45view3

LLVM 2024 的 Hopper 演讲则把上下两层关系说得非常清楚：`nvgpu.warpgroup.mma` 是高层抽象，lowering 后会展开成多条 `nvvm.wgmma.mma_async` 配合 `fence/commit/wait`；同一场演讲也明确说 TMA、warp-group tensor core、事务型 barrier 的相关能力已经被实现到 GPU/NVGPU/NVVM 方言中，而且演讲者声称“展示的工作均已 upstream 到 MLIR”。这说明至少到 Hopper 为止，**“高层用 nvgpu，底层用 nvvm”** 的分工已经比较稳定。citeturn10view0turn11view0turn12view0turn12view1

Blackwell 的情况更微妙一些。好消息是：**当前 NVVM 文档已经公开出现 `tcgen05.*` 家族**，包括稀疏、block-scaled、weight-stationary 等变体，并带有 SM 版本约束 trait，说明 low-level 建模已经开始向 Blackwell / 5th-gen Tensor Core 延展。EuroLLVM 2025 的演讲还给出了一些落地状态：公开材料称 Blackwell 为 MLIR/NVVM 增加了约 30 个 op、TMEM 的新地址空间、以及大量新 intrinsic；同时又承认当时仍在把 lowering 从 inline-PTX 迁往 intrinsic-based 方案，并说“除 MMA 外，tcgen05 的其余操作已支持到 NVVM dialect”。把两类资料合起来看，最稳妥的结论是：**Blackwell 的 low-level NVVM 建模已经明显前进，但在我查到的公开 upstream 资料里，还没有像 Hopper `nvgpu.warpgroup.mma` 那样清晰、成型、文档化的上层通用中层故事**。citeturn45view1turn13view0

这对接入方的现实含义是：如果你只想要**最终到 NVPTX 的 feature reach**，NVVM 层已经相当有用；但如果你希望在 **“可优化中层”** 就优雅地承载最新架构的 TMEM / tcgen05 / 更复杂的 descriptor semantics，那么 2026 年的 upstream 公开材料仍然显示这块处于持续演进中，而不是像 `linalg`/`vector` 那样已经形成一个稳定的大一统抽象。citeturn44view0turn45view1turn13view0

对你的问题，这张表最重要的启示不是“哪家更先进”，而是：**只要你的语言有自己独特的中层约束，最终大概率也会走“自研 MIR 或自研 MLIR dialect + upstream sink”的混合路线**，而不是纯粹“拥抱官方 gpu 方言替换一切”。citeturn20view0turn28view0turn31view1turn39view0

## 接入成本与路线选择

对一门**已有自研 MIR 且已经可直通 LLVM/NVPTX** 的语言，最常见的两种接法，不是“引入 MLIR”与“不引入 MLIR”这么粗，而是：

其一，**翻译层方案**：保留自研 MIR 作为主 IR，只在“值得高级优化的 kernel / dispatch 子问题”上翻到 MLIR，再利用 `linalg`/`vector`/`nvgpu`/`nvvm` 或自定义 MLIR dialect 做中层变换，最后掉回 LLVM/NVPTX。其二，**MLIR 直接成为 MIR**：把你的现有 MIR 逐步替换为一个或多个自家 MLIR 方言，让所有中层优化都在 MLIR 里完成，再统一 sink 到 LLVM。结合 IREE、Triton、OpenXLA、torch-mlir、Mojo 的实践，前者更像“低风险增量接入”，后者更像“平台重构”。citeturn20view0turn20view1turn28view0turn31view1turn36view0turn39view0

下面这张表是基于官方方言职责、官方 sink pipeline 边界、以及上述项目实践做的**综合判断表**：

| 维度 | MIR → MLIR → LLVM | MIR → LLVM 直通 |
|---|---|---|
| 首次接入成本 | 高。要定义至少一个翻译边界、pass pipeline、bufferization/ABI 规则；如果直接以 MLIR 为 MIR，成本更高 | 低。沿用现有基础设施即可 |
| kernel 融合 / tile / vector / tensor lowering | 强。`linalg` 的设计目标本来就包含 tiling、fusion、promotion、vectorization、special-instruction lowering | 弱到中。可以做，但通常要在自研 MIR 或 LLVM 上手写更多 pattern 与成本更高的分析 |
| Tensor Core / TMA / 设备特化 | 强。可用 upstream `nvgpu`/`nvvm`，必要时再在其上叠自家方言 | 中。能直接打 intrinsic / inline asm / NVVM IR，但中层可组合性和可迁移性较差 |
| 短期“先跑起来”速度 | 慢于直通 LLVM，尤其在团队尚无 MLIR 经验时 | 最快 |
| 版本追踪与依赖负担 | 显著更高。真实世界里通常要 pin LLVM/MLIR revision，并定期升级 | 也不低，但你已经在承担 LLVM 部分；不额外引入 MLIR API churn |
| 长期可扩展性 | 更好。多后端、图级优化、transform/tuning、更细粒度 feature exposure 更容易组织 | 容易在自研 MIR 与 LLVM 之间堆积一次性 lowering 逻辑 |
| 对团队能力结构的要求 | 需要懂 ODS、dialect conversion、bufferization、pass pipeline、TableGen、MLIR verifier 生态 | 主要是 LLVM IR / backend / target intrinsic 能力 |
| 最适合的阶段 | 语言语义已基本稳定，GPU kernel 规模与复杂度开始逼近 fusion / tensorization 临界点时 | MVP、验证期、或 kernel 仍然比较简单且主要靠库调用/少量手工 intrinsic 时 |

这个取舍之所以不能只看“功能”，是因为 **MLIR 的版本与依赖负担确实真实存在**。Triton 的官方构建说明直接写着“LLVM does not have a stable API”，因此 Triton 不能对任意 LLVM 版本工作；Tenstorrent 的 tt-mlir 文档甚至把 LLVM 依赖更新写成了季度性例行工作；而 2024 年 MLIR 项目治理讨论里也有人把当前生态形容为“Tower of Babel”，并明确提到 upstream dialect 拉扯带来的维护成本。我的判断是：**MLIR API 不稳定并不是“吓人的传说”，而是你必须预算进去的持续性工程成本**。citeturn41search1turn41search9turn41search2

因此，对你的语言路线，我会给出非常具体的建议：**短期不要把 MLIR 变成全局唯一 MIR；优先做“kernel island / dispatch island”式翻译层接入**。也就是先让自研 MIR 保持主导，只把最需要结构化变换的热 kernel 降到你自己的 MLIR dialect 或直接到 `linalg`/`vector`，再接 upstream NVIDIA sink。原因很简单：这条路能把 `linalg`/`vector`/`nvgpu`/`nvvm` 的好处拿到手，同时把 MLIR 版本 churn、全语言重编译、IR 范式切换的组织成本控制在局部范围。等你真正证明“中层结构化优化收益大到足以覆盖接入成本”时，再考虑是否让 MLIR 上升为主 MIR。这个判断与 IREE、OpenXLA、torch-mlir 的“桥接或分层接入”方式是一致的，也与 Mojo、Triton 这类“从一开始就把 MLIR 作为体系骨架”的项目形成鲜明对比。citeturn20view0turn31view1turn36view0turn39view0turn28view0

## 反方观点与引入时机

“反方观点”里最明确、最有价值的不是泛泛而谈的“MLIR 很复杂”，而是**具体项目为什么评估后没有把它作为核心路径**。

一个非常明确的例子是 **rust-gpu**。在 LLVM/HLSL 相关 RFC 讨论里，rust-gpu 参与者直说：他们在项目早期**评估过 MLIR**，但当时认为它“**不够成熟**”，并且“**似乎主要聚焦在 ML workload，而不是 graphics workload**”；随后 rust-gpu 自己实现了 structurizer 和多种 SPIR-V workaround。这个例子虽然来自图形 shader 而不是 CUDA kernel，但它非常能说明一个现实：**如果你的工作负载不契合当前 upstream 主战场，MLIR 的抽象收益可能会被生态不匹配抵消**。citeturn43view0

第二个反方观点来自 **LLPC / shader compiler** 阵营。在同一讨论里，AMD 侧直接说，像 LLPC 这样的 shader compiler 往往需要运行“典型 LLVM IR pass pipeline”，同时还要保留 GPU 特有构造；而 MLIR 与 LLVM IR 使用的是不同 substrate，于是工程上必须在“已有 LLVM 优化丰富度”和“MLIR 表达丰富度”之间取舍，实际中“使用现有 LLVM 优化的丰富度往往赢得这个务实权衡”。这不是说 MLIR 不好，而是说：**如果你高度依赖 LLVM 现成中后端优化并且不想迁移 substrate，继续留在 LLVM IR 世界是完全理性的选择**。citeturn43view0

第三个反方例子是 **DXC / HLSL 的 SPIR-V 路线**。RFC 中维护者承认，当前 SPIR-V path 仍然从 AST 发射，而且重要缺点之一就是与 IR generation code path 共享代码有限；他们确实讨论过 MLIR 的 SPIR-V dialect，但公开讨论显示这仍然是 trade-off，而不是已采纳的主线。也就是说，**即便在 LLVM 生态内部，是否把 MLIR 放进 GPU/graphics codegen 主线，也常常取决于既有产线、生态兼容性和性能风险，而不是抽象是否“更优雅”**。citeturn43view0

放回你的场景，我会把“**什么信号出现时才值得引入 MLIR**”说得非常操作化：

| 信号 | 我的建议 |
|---|---|
| 你开始反复实现 tile/fuse、producer-consumer fusion、shared-memory promotion、vector/tensor lowering，并且这些逻辑越来越难在自研 MIR 或直接 LLVM 上维护 | 这时值得引入 MLIR，尤其是 `linalg`/`vector`/bufferization 组合，因为这正是它们被设计出来解决的问题。citeturn7view0turn8search3 |
| 你要显式使用 Tensor Core、WGMMA、TMA、mbarrier、cluster 等新硬件能力，而且不希望所有逻辑都直接下沉到 NVVM intrinsic / inline PTX | 这时值得引入至少 `nvgpu`/`nvvm` 作为 feature bridge / sink。citeturn44view1turn44view0turn45view0turn45view2 |
| 你准备做多 backend，或者至少希望未来不只盯 NVPTX | 这时 MLIR 的价值上升，因为 `linalg`/`vector`/自定义 dialect 可以把“可优化语义”与最终 target 分开。IREE 与 OpenXLA 的分层实践非常说明问题。citeturn20view0turn20view1turn31view1 |
| 你当前最重要的是尽快把 MVP 跑稳，kernel 还不复杂，大部分性能仍可通过库调用、手工 intrinsic 或少量定制 LLVM pass 获得 | 继续 MIR→LLVM 直通更合理，不该为“可能以后有用”的中层先付出重构成本。citeturn8search0turn43view0 |
| 团队没有余力维护 pinned LLVM/MLIR revision 并跟踪 API churn | 暂时不要把 MLIR 提升为核心基础设施；最多做局部实验接入。真实项目已经反复证明版本追踪成本不可忽视。citeturn41search1turn41search9turn41search2 |

如果必须给一句最终建议，我会写成：**当且仅当你已经清楚地遇到“结构化 GPU 中层”的痛点，而不是只是“听说 MLIR 很先进”时，才值得引入；并且第一步应当是“自研 MIR → 局部 MLIR kernel codegen island → LLVM”，而不是全量 MIR 替换。** 这既是对 upstream 现状的尊重，也是对真实项目实践的尊重。citeturn7view0turn8search3turn20view0turn28view0turn31view1turn41search1
