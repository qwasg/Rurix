# 新语言包管理与供应链安全设计研究

## 结论摘要

如果你的语言目标是“Windows-only、CUDA-first 的系统语言”，而且 MVP 明确只做 `manifest + lockfile + vendor + checksum`、暂时不做中心化 registry，那么最稳妥的路线不是“照抄 Cargo”，而是采用一个**Cargo 的开发体验骨架**、**Go 的完整性模型方向**、**Zig 的最小化抓取语义**、并且**刻意避免 npm/Cargo 的可执行安装脚本风险**的混合设计。最值得保留的 Cargo 思想是一等 manifest、根锁文件、workspace、vendor、offline、`-sys`/`links` 这一套“系统语言友好”的工程结构；最值得避免的，是把“任意代码构建脚本”做成包管理器默认能力，以及把“特性/可选依赖/命名空间”这些会长期影响生态可维护性的点过早固化。citeturn25search1turn25search7turn23view5turn23view4turn23view6turn39view1turn14view0turn31view1turn31view2

对你的语言而言，MVP 的推荐形态可以概括为：**声明式 manifest，强制根 lockfile，`vendor/` 为可提交目录，锁文件记录内容哈希和解析后的完整依赖图，默认支持完全离线重建；依赖来源先只支持 `path`、`git`、`archive` 三类；不提供 Cargo 式“任意 build.rs”，而是优先提供声明式 native/GPU 元数据，必要时再用受限 runner 或 allowlist 机制补充。** 这样能在不引入中心仓库的前提下，把“可复现”和“可审计”提到 MVP 级别，而不是等 registry 出现以后再补救。citeturn23view0turn23view3turn23view4turn23view6turn23view7turn14view0turn14view1turn29view0turn29view1

如果以后要演进到 registry，我的核心建议是：**不要把 registry 当成唯一信任根**。Cargo 的 registry index 会记录 `.crate` 的 SHA-256 校验和，但它仍然主要依赖 registry/index 自身的正确性；Go 则通过全局追加式 checksum database 把“大家拿到的是不是同一份代码”提升到了透明日志层。对于一个新生态，后者明显更适合作为长期方向。citeturn24view0turn24view2turn24view4turn31view1turn31view2

## Cargo 的可复制部分与官方承认的痛点

Cargo 之所以仍然是系统语言包管理的重要参考，不只是因为它有 `Cargo.toml` 和 `Cargo.lock`，而是因为这两者的职责分工非常清楚：manifest 负责“意图”，lockfile 负责“解析结果”。Cargo 官方文档把这一点讲得很直白：`Cargo.toml` 描述广义依赖，`Cargo.lock` 记录**精确**依赖版本；`--locked` 会在锁文件缺失或需要被改写时报错。workspace 则通过**单一根锁文件**来保证整仓库一起解析、尽量共享依赖和最小化版本分裂。对新语言来说，这些都值得直接继承。citeturn25search1turn23view7turn23view5

Cargo 的 resolver 也是重要启发。官方文档与 RFC 明确承认：早期“对同一包做全局 feature union”的设计虽然易理解、也能减少重复编译，但会导致不希望的功能泄漏，典型场景包括 target-specific 依赖、dev-dependencies、build-dependencies/proc-macro 与普通依赖之间的错误统一。因此 `resolver = "2"` 引入了更细化的 feature unification 规则，只在真正参与当前构建的边上合并特性。对你的语言，这意味着一个原则：**manifest 里的“功能开关”必须从一开始就防止意外外溢，不要把“全局 union 一切”的规则写死。**citeturn23view1turn26view0

Cargo 在 native 依赖上的 `links` 与 `*-sys` 约定，也非常适合系统语言借鉴。官方文档说明，`links` 的作用有两层：一是让 Cargo 知道某个包会链接某个原生库；二是把 build script 发现到的元数据通过 `DEP_<LINKS>_*` 传给直接依赖包。Cargo 还要求**同一个 `links` 值在依赖图里最多只能有一个包**，以避免重复符号；而 `foo-sys` 的约定则把“发现/构建 `libfoo`”以及“导出底层 FFI 绑定”的责任集中到一个公共依赖里。这个模式对 CUDA、Windows SDK、cuBLAS、cuDNN 一类依赖尤其有用。citeturn23view0turn23view2turn39view0turn39view2turn39view3

Cargo 的 vendor/offline 组合也值得保留，但要看清边界。`cargo vendor` 会把 crates.io 和 git 依赖物化到本地目录；`--offline` 会强制只使用本地已下载内容，而官方文档也提醒这可能导致与在线解析不同的结果；`--frozen` 则相当于 `--locked + --offline`。对于你的语言，这说明：**vendor 不是“缓存优化”，而应该是 CI 与 release 的一等输入；离线模式必须与锁文件强绑定。**citeturn23view4turn23view6turn23view7

Cargo 的 registry 完整性模型也很有代表性，但不应照搬。registry index 中会为每个版本记录 `.crate` 文件的 SHA-256 校验和 `cksum`，下载 URL 甚至可以把 `{sha256-checksum}` 作为路径模板的一部分；不过 source replacement 明确要求替代源必须是原源的**子集或等价镜像**，而 index 记录中只有 `yanked` 字段是允许后续修改的。换句话说，Cargo 的完整性更偏向“**我信任 index/registry，它给我一个 tarball 校验和**”，而不是 Go 那种“**即使 proxy/registry 被动过手脚，也要让篡改可审计**”。你在 MVP 阶段既然没有中心 registry，就不应该把未来模型设计成“registry 单点信任”。citeturn24view0turn24view2turn24view4turn23view3

你要求必须找到“哪些 Cargo 决策被 Cargo 团队自己后悔/想改”的证据，这一点在官方 RFC 与维护者议题里其实很明确。第一，feature unification 的原始规则，RFC 2957 明说“当前做法虽然易理解，但会引入 unwanted behavior/dependencies/requirements”；RFC 3143 又把 optional dependency feature 与 namespaced features 描述为 “well-known pain points”。第二，Cargo 维护者 ehuss 在 2024 年开出的 workspace `feature-unification` 跟踪议题，已经把“workspace / package 级 feature unification 可配置化”作为继续推进方向，并列出后续扩展。第三，命名空间问题上，RFC 3243 直接承认今天的 `projectname-foo` 并不一定由 `projectname` 官方维护，因而“很难向用户表达这些 crate 是否属于同一项目、是否应被同等信任”。第四，构建脚本安全上，Rust 2024H2 官方 project goal 明确把“探索 sandboxed build scripts”列为目标；到 2024-2026 年的官方 issue 里，讨论仍集中在 allowlist、sandbox runner 等探索方案，说明 Cargo 团队自己也把“无沙箱 build.rs”视为需要改变但尚未稳定落地的问题。citeturn26view0turn26view1turn29view2turn26view2turn28search19turn29view0turn29view1

## 对照系统给出的设计边界

Go modules 给出的最大启发，不是 `go.mod` 本身，而是“**去中心化来源 + 可选 proxy + 全局可验证 checksum log**”这个三层模型。官方文档说明 `GOPROXY` 默认是 `https://proxy.golang.org,direct`，也就是先走 proxy，再按需直连源；`GOSUMDB` 默认指向 `sum.golang.org`；私有模块可以通过 `GOPRIVATE`、`GONOSUMDB`、`GONOPROXY` 排除。Go 官方博客进一步说明，`go.sum` 只能解决“你自己以后重下同一版本是否一致”，而 checksum database 解决的是“**所有人是否都在使用同一份代码**”；sumdb 通过透明日志、inclusion proof、consistency proof 来让不可信 proxy 也可被安全使用。对一个不想在 MVP 做中心化 registry 的新语言，这个方向比 Cargo 的 registry-only 信任模型更先进。citeturn31view0turn31view1turn31view2

Go 还证明了一件事：不是所有生态都需要 Cargo 那样的完整 lockfile 模型。Go 使用 MVS（Minimal Version Selection），官方文档直接写明它用 MVS 选择模块版本，相关博客也强调它偏向选择满足约束的**最小**版本，从而减少“时间因素”对构建结果的影响。对你的语言来说，是否采用 MVS 是开放问题；但如果你已经决定 MVP 做 lockfile，那么大概率不需要去追求“像 Go 一样无锁也稳定”，因为你的语言首先要解决的是系统库、CUDA、二进制变体，而不是纯源码包的最小版本闭包。citeturn32search2turn31view4turn32search7

npm 提供的主要是反例和安全教训。官方文档说明 `package-lock.json` 描述的是“生成出来的**确切**依赖树”，`npm ci` 会在 lock 与 manifest 不一致时报错，并且不会改写锁文件；但 npm 生态长期允许 install-time lifecycle scripts，直到 2025-2026 文档里才明确提供 `ignore-scripts` 与 `allow-scripts` 这样的细粒度控制。GitHub/npm 官方在 2025 年从 Shai-Hulud 事件中总结得很直白：恶意 `post-install` 脚本可以借由被攻陷的维护者账号注入流行包，并通过窃取 token 和其他 secrets 继续自传播，GitHub 为此一次性下架了 500+ 包。对一个新语言生态，这已经足够说明：**绝不能把可执行安装/构建脚本设计成默认、无约束、无审计的能力。**citeturn38view0turn38view1turn35search7turn38view6

Zig 的价值，在于它把“抓取依赖的信息”从“可执行构建逻辑”里分离出来。Zig 0.11 的 release notes 明确说明：官方包管理没有“官方仓库”，依赖就是本地目录或互联网归档；依赖声明放在 `build.zig.zon`，其中记录名称、版本、URL、hash；更关键的是，这个 hash 不是 archive 本身，而是其**内容**。官方还明确解释为什么要把这些信息放在独立文件，而不是 `build.zig` 里：这样包获取与图分析**不需要执行潜在危险的代码**。到 Zig 0.16，又把依赖抓到项目本地 `zig-pkg` 目录，并引入 fingerprint 来区分同名项目的不同版本。这组设计对你的 MVP 几乎就是“直接可用”的素材。citeturn14view0turn14view1turn14view2

vcpkg 和 Conan 则证明了 C/C++ 二进制包管理为什么容易复杂化。vcpkg 的 manifest mode 是官方推荐工作流，版本控制、自定义 registries 都要求 manifest mode；它有 baseline、registry、binary cache 等不错的工程化能力，但本质上仍然大量依赖 recipe 在用户机器上下载源码并构建。Conan 则更进一步：它把 settings、options、依赖版本都纳入 `package_id`，profiles 里还会携带 tool requirements、build env、run env 等整套上下文；官方 lockfile 文档也说明，锁文件是**配置相关**的，通常每个配置都会有一个 lockfile。这对传统 C/C++/多编译器世界是合理的，但对一个新语言的包管理 MVP 来说，意味着**如果你过早把“预编译二进制矩阵”纳入核心模型，复杂度会爆炸**。citeturn36view0turn36view1turn36view2turn36view3turn16search0turn36view4turn36view6

## 原生依赖与 GPU 生态的专门设计

对系统语言来说，最难的不是“纯源码库怎么解依赖”，而是“C/C++/SDK/CUDA 这种原生输入怎么建模”。Cargo 的 `*-sys`/`links` 约定之所以长期有效，原因恰恰在这里：它不试图让包管理器彻底接管外部世界，而是通过一个公共低层包把**发现、链接、元数据传递**集中起来，再让上层安全封装包依赖它。官方文档还支持对 `links` 对应的 build script 做配置覆盖，直接绕过脚本执行并人工注入 link metadata。对你的语言，这个思路可以直接抽象成三种 provider：`system`、`vendored-source`、`vendored-binary`。citeturn39view1turn39view2

因此我建议新语言把“原生依赖”做成 manifest 的一等字段，而不是完全交给脚本发现。最小模型可以是：`native.<name>.provider = "system" | "source" | "binary"`，再附带 `links`、header/lib 路径、所需环境能力（MSVC、Windows SDK、CUDA Toolkit）等。Windows-only 反而是优势：Visual Studio/Build Tools 的发现可以借助 Microsoft 官方支持的 `vswhere` 和 Setup Configuration API，核心 C/C++ 工作负载也有稳定安装入口；因此 Windows SDK / MSVC 更像“主机能力探测”，不应在 MVP 阶段被当作普通包去分发。citeturn22search1turn22search3turn22search0

CUDA-first 设计则必须把 GPU 兼容元数据前置。NVIDIA 官方文档与技术博客给出了几个硬约束。其一，compute capability 决定硬件能力；其二，CUDA 11 以后存在同一 major family 内的 minor version compatibility，但仍要求足够新的驱动，例如 2026 年 CUDA release notes 仍按 major family 给出最低驱动范围；其三，CUDA 代码兼容有两条路径：**cubin 的二进制兼容**与 **PTX 的 JIT 兼容**。cubin 只在同一 major compute capability 内、且目标 GPU minor 不低于编译基线时可直接运行；PTX 则能作为跨更高 compute capability 的前向载体。citeturn20search12turn41view0turn21search9turn41view2

这决定了包管理器必须允许包声明至少四类 GPU 元数据：`cuda.toolkit`、`cuda.min_driver`、`sm` 覆盖范围、以及分发物是 `ptx`、`cubin` 还是 `fatbin`。NVIDIA `nvcc` 官方文档明确支持直接产出 `.ptx`、`.cubin`、`.fatbin`，并且 fatbin 可作为容器同时承载多种 device code；NVIDIA 的 PTX 文章也明确说明 fatbin 可以同时包含多份 cubin 和 PTX。对你来说，推荐默认策略不是“全 PTX”也不是“全 fatbin”，而是**混合策略**：为最常见热路径预编几份 cubin，再附带一个较低 `compute_XX` 基线的 PTX 以保证前向可用性；同时把这些变体及其哈希写入 lockfile。fatbin 中变体越多，体积越大，这一点可以从其容纳多份 cubin/PTX 的结构直接推导出来，因此必须把“目标 SM 集合”做成显式、可裁剪的打包参数。citeturn41view1turn41view2

换句话说，你不应该把 CUDA 看成“又一个系统库”。它更像是“**一组 ABI/ISA/driver/toolkit 兼容约束 + 一组可选预编译代码变体**”。包管理器如果不把这些约束写进 manifest/lockfile，最后就会退化成“README 驱动的兼容性管理”，而这正是很多 C++/CUDA 生态反复踩坑的源头。citeturn41view0turn41view1turn41view2

## 供应链安全基线与 2024-2026 现状

你的 MVP 明确要做 `lockfile + vendor + checksum`，这本身就是最重要的供应链安全基线。锁文件的价值，不只是“避免升级”，而是把“这次成功构建所依赖的世界状态”冻结下来；Cargo FAQ 与 npm 官方都把这一点说得很清楚。对新语言，我建议把应用/工作区构建的默认模式设计成**必须存在 lockfile**，CI 与 release 使用**默认冻结模式**，而不是像部分生态那样把锁文件视为可选。citeturn25search7turn38view0turn38view1

但只靠 lockfile 仍然不够，因为“第一次拿到内容时”仍有 TOFU 问题。Go 的 sumdb 正是为了解决这一层：它把 `go.sum` 条目放进 append-only、可校验的一致性日志，让 proxy 或源站即使作恶，也很难只针对个别受害者投毒而不留下证据。对你的 MVP，短期内可以接受“**lockfile 中写内容哈希 + vendor 目录做审计边界**”；但如果以后做 registry，强烈建议直接朝“**sparse index + checksum transparency log**”演进，而不是停留在 Cargo/npm 那种“registry 提供 checksum/signature，但用户仍默认相信 registry 元数据”的层次。citeturn31view1turn31view2turn24view0turn38view3turn38view4

构建脚本安全是 2024-2026 最值得正视的新进展之一。就本次查到的官方资料看，Rust/Cargo 对 sandboxed build scripts 的讨论已经进入正式 project goal 和持续 issue 阶段，但**尚未形成稳定方案**。官方 project goal 的目标是给 Cargo 用户一个可选的 sandbox 环境，限制文件系统和网络访问；2024-2025 的后续 issue 又继续讨论 allowlist mode 与自定义 runner sandbox。也就是说，业界对“最小权限构建”的共识已经形成，但实现层仍在探索。对你的语言，这不是“以后再说”的功能，而是**MVP 架构就该规避的问题**：最好的安全脚本，是根本不需要脚本。citeturn28search19turn29view0turn29view1

在 provenance / 签名方面，现在最现实的组合是 **SLSA + Sigstore + SBOM**。SLSA 1.2 的 Build Track 明确把 provenance 生成、签名、验证、build platform 隔离强度分层；L2 开始要求托管平台生成并签名 provenance，L3 则强调 hardened build。Sigstore 的核心机制是短命证书、透明日志 Rekor 和可审计签名事件；Cosign 官方同时支持 SPDX、CycloneDX 这两类 SBOM，并支持 in-toto attestations。npm 已经把这条路线产品化：trusted publishing 通过 OIDC 代替长期 token，自动生成 provenance attestation；npm provenance 文档还明确写到这些 attestation 由 Sigstore public good servers 签名并进入公共 transparency ledger。对一个新语言 registry 的未来形态，这其实已经给出了成熟答案。citeturn37view0turn37view1turn37view2turn37view3turn37view4turn37view5turn37view6turn37view7

SBOM 输出最好从“包管理器能天然看到的信息”做起。Cargo 已经有不稳定的 `-Z sbom`，它生成的是 SBOM pre-cursor 文件，里面包含依赖、features、target、编译器等信息，供下游 SBOM 工具补全。这个设计很值得借鉴，因为它避免了“让包管理器自己一次性生成全格式 SBOM”的过度承诺。对你的语言，建议 MVP 不必直接承诺 SPDX/CycloneDX 全量输出，但至少应生成一个**稳定的 precursor JSON**，未来再由官方或第三方工具转成 SPDX 3.0 或 CycloneDX。citeturn40view0turn37view4turn37view5

typosquatting 防御不要等 registry 上线后再想。npm 官方已经把 typosquatting / dependency confusion 列为一类明确威胁，并声称会检测并阻止 typosquat 包发布；Cargo 侧则通过 namespaces RFC 直面“项目族官方性表达不足”的问题。对你的语言，MVP 没有中心 registry，可以暂时把风险收敛到“lockfile + vendor + 内容哈希”；但一旦做 registry，**scopes/namespaces、保留名、相似名审核、来源证明**应该在第一天就进设计，而不是作为补丁功能。citeturn37view8turn26view2

## 新语言 manifest 与 lockfile 草案

下面给出一个更适合你场景的字段草案。设计原则是：**manifest 只描述意图和能力约束，lockfile 才写解析结果、来源证明、内容哈希、GPU/native 变体。** 这种分层直接吸收了 Cargo 的 manifest/lock 分离、Zig 把抓取元数据独立于可执行构建逻辑、以及 Go 把完整性数据单独建模的经验。citeturn25search1turn14view0turn31view1

```toml
manifest-version = 1

[package]
name = "tensor_app"
version = "0.1.0"
language-version = "0.1"
edition = "2026"

[workspace]
members = ["crates/*"]

[resolver]
features = "additive-v1"      # 禁止隐式开启 optional dep
unification = "selected"      # 预留 selected/workspace/package

[vendor]
dir = "vendor"
mode = "locked"               # locked / update

[dependencies.linalg]
git = "https://example.com/linalg.git"
tag = "v1.4.0"
version = "^1.4"
features = ["cuda"]
optional = false

[dependencies.win_sdk_sys]
path = "sys/win_sdk"
native.links = "windows_sdk"
native.provider = "system"    # system / source / binary

[dependencies.cublas_sys]
git = "https://example.com/cublas-sys.git"
rev = "a1b2c3d4"
native.links = "cublas"
native.provider = "system"
native.require = ["cuda-toolkit", "msvc"]

[gpu]
toolkit = "12.8..13"
min-driver = ">=525"
sm = ["80", "86", "89"]
codegen = ["cubin", "ptx"]    # 可选 cubin / ptx / fatbin
ptx-floor = "compute_80"

[build]
model = "declarative"         # declarative / sandboxed-runner
```

这个 manifest 草案有几个刻意的取舍。第一，依赖来源只假定 `path`、`git`、`archive` 风格输入，不引入 registry 专属字段。第二，`native.*` 与 `gpu.*` 是一等概念，而不是塞进任意脚本。第三，feature resolver 与 unification 预留版本位，避免像 Cargo 那样以后通过 RFC 补语义裂缝。第四，`build.model = "declarative"` 把“默认不执行任意脚本”写进体系，而不是靠文档约束。Cargo、Zig、npm 的资料共同说明，这会在长期安全性上省掉大量生态债务。citeturn26view0turn26view1turn14view0turn38view6turn29view0

```toml
version = 1
generated-by = "rpk 0.1.0"
resolver-features = "additive-v1"
resolver-unification = "selected"
checksum-algorithm = "sha256"

[[package]]
id = "linalg@1.4.0"
name = "linalg"
version = "1.4.0"
source = "git+https://example.com/linalg.git?tag=v1.4.0"
resolved-rev = "9f8e7d6c5b4a..."
manifest-digest = "sha256:..."
content-digest = "sha256:..."     # 规范化源码树哈希
vendor-path = "vendor/linalg-1.4.0"
dependencies = ["cublas_sys@0.3.0"]
features = ["cuda"]

[[package]]
id = "cublas_sys@0.3.0"
name = "cublas_sys"
version = "0.3.0"
source = "git+https://example.com/cublas-sys.git?rev=a1b2c3d4"
resolved-rev = "a1b2c3d4..."
manifest-digest = "sha256:..."
content-digest = "sha256:..."
vendor-path = "vendor/cublas_sys-0.3.0"
links = "cublas"
native-provider = "system"
native-metadata = { include = "...", lib = "...", importlib = "cublas.lib" }

[[artifact]]
package = "linalg@1.4.0"
target = "x86_64-pc-windows-msvc"
cuda-toolkit = "12.8"
min-driver = "525"
sm = ["80", "86", "89"]
payloads = [
  { kind = "cubin", arch = "sm_80", digest = "sha256:..." },
  { kind = "cubin", arch = "sm_86", digest = "sha256:..." },
  { kind = "ptx",   arch = "compute_80", digest = "sha256:..." }
]
```

锁文件里我尤其建议新增四类 Cargo 没有系统化做好的字段。第一是 `content-digest`：更接近 Zig 的“内容树哈希”，比仅对下载归档做哈希更适合 vendor 与镜像场景。第二是 `vendor-path`：把 vendoring 从外部约定提升为解析结果的一部分。第三是 `native-metadata`：把 `links` 发现到的结果持久化，尽量减少每次重建重新探测主机环境的需要。第四是 `[[artifact]]`：把 CUDA / 预编译变体视为 lockfile 的一部分，而不是“隐含在某个目录命名规则里”。这些取舍分别对应 Zig 的内容哈希、Cargo 的 `links` / `-sys` 模式、以及 NVIDIA 对 PTX/cubin/fatbin 兼容性的正式建模。citeturn14view0turn39view1turn39view2turn41view1turn41view2

## 从 MVP 到 registry 的演进路线

MVP 阶段，建议只做三件事，并且把它们做“硬”：**根锁文件必需、vendor 为可提交目录、lockfile 记录所有远程内容的规范化内容哈希。** 来源只支持 `path`、`git`、`archive`；构建侧只支持声明式 `native`/`gpu` 元数据；如果确实需要主机探测，就通过受控 provider 与显式 capability check 完成，而不是执行第三方脚本。这一阶段的信任根应该是：**VCS 提交 + lockfile + vendor 目录**。citeturn23view4turn23view6turn14view0turn25search7

第一阶段增强，应补足“最小权限构建”和“可审计产物”两件事。前者可以先做 allowlist/runner 化，而不是一开始就追求跨平台内建沙箱；Cargo 2024-2026 的讨论恰恰说明 runner/allowlist 是务实方向。后者则应引入官方 precursor SBOM JSON，并在 CLI 中提供验证 vendor 目录内容哈希、导出依赖图、列出含 native/GPU 变体的包等能力。citeturn29view0turn29view1turn40view0

第二阶段如果要进入“有 registry 但不把 registry 当唯一信任根”的形态，我建议采用：**source index / mirror 用 sparse 协议，包内容继续由 lockfile 内容哈希校验，新增一个 sumdb-like 透明日志记录 `<source, version, content-digest>` 绑定**。也就是说，registry 负责发现与分发，透明日志负责“大家都在拿同一份内容”的可验证性。这本质上是把 Cargo 的发现体验与 Go 的可验证性结合起来。citeturn24view2turn31view1turn31view2

第三阶段再引入开发者发布侧的签名与 provenance。可以参考 npm 当前路线：trusted publishing 用 OIDC 替代长期 token，自动产生 provenance attestation，并把 attestation 放入 Sigstore 的透明日志。与此同时，registry 需要提供 scopes/namespaces、similar-name 风险控制、恶意包举报与 staged publishing 之类的门槛控制。对于一个新生态，这一阶段才适合把“中心 registry”做成生态入口，而不是在 MVP 就背上全部治理复杂度。citeturn37view6turn37view7turn38view5turn37view8turn26view2

最后给出一句落地判断：**如果只能选一条主线，就选“Cargo 的工程体验 + Zig 的声明式抓取 + Go 的透明完整性 + 明确拒绝 npm/Cargo 式默认任意构建脚本”**。这条路线最符合你的语言定位，也最能把“Windows-only、CUDA-first”这两个现实约束，从一开始就体现在包管理与供应链模型里，而不是等生态长大后再追着修。citeturn23view5turn14view0turn31view2turn38view6turn28search19

## 开放问题与局限

关于 Cargo 构建脚本安全的“2024-2026 最新讨论”，本次查到的官方材料已经足以确认方向：Cargo 团队把 sandboxed build scripts 当成正式目标，且 2024-2026 仍在讨论 allowlist、runner 与沙箱策略，但我没有在这些资料里看到“已经稳定上线的统一内建方案”。因此本报告在这里给出的不是“Cargo 已解决该问题”的结论，而是“**Cargo 官方已经明确承认问题并持续探索，但仍未稳定落地**”。citeturn28search19turn29view0turn29view1

另一个开放问题是版本选择算法的最终形态。Cargo 当前方向是继续把 feature unification 做得更可配置，而 Go 则通过 MVS 降低求解复杂度；对于你的语言，因为还要原生建模 CUDA/Windows/native 变体，所以我更倾向于“**维持 Cargo 式锁文件 + 显式求解**”，而不是在 MVP 追求 Go 式无锁稳定。这一判断是基于本次比较研究做出的设计推断，而不是来自某个现成标准。citeturn29view2turn32search2turn16search0turn41view2
