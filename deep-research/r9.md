# R9 — 编译器诊断与 IDE/LSP 体验工程

## Rust 诊断体系解剖

Rust 编译器的诊断体系设计严谨，主要组成包括错误/警告等级、错误码、错误信息、**span**（源代码位置）、**label**、**note**、**help** 及 **suggestion**（建议修复）。例如，一个错误报告包含主信息、错误码 `[E0001]`，并在代码对应位置加上波浪线指示错误范围，同时可能附带多行 `label`、`note`、`help` 说明。**Error Code 索引**对应错误码，可通过 `rustc --explain E0001` 查看详细解释。诊断输出既面向开发者，也支持机器工具：Rust 提供 JSON 格式输出 (`--error-format=json`)，包含结构化字段和原文渲染信息。其中，**MachineApplicable** 等级的建议可以被 IDE/工具自动应用（如 `cargo fix`），JSON 中用 `suggestion` 字段表示可自动修复的修改示例。Rust 的测试框架使用“UI 测试”，在测试源文件中用注释（如 `//~ ERROR`、`//~ SUGGESTION`）标注预期诊断和建议，以确保每次修改后诊断信息准确。  

对新语言而言，需要实现类似的基础设施，包括：**源代码位置管理**（span/label）系统，用于定位和标注问题；**诊断结构定义**，支持多级消息（Error/Warning/Note/Help/Suggestion）；**可自动应用建议**的标记体系（相当于 MachineApplicable 建议，便于批量修复）；**错误码体系**及文档说明功能（为常见错误分配代码、生成详解）；并建立 **UI 测试框架**（如 Rust 的 compiletest）以便校验诊断输出和修复建议。这些要素构成最小诊断落地需求：正如 Rust 诊断示例所示，每个诊断报告至少应有代码**span**和**消息**，可选附带labels、notes、help、建议等。

## rust-analyzer 架构与进展

Rust-analyzer（RA）是 Rust 语言的独立增量 IDE 后端，其设计目的是实现快速的 IDE 交互体验，与 `rustc` 的一遍性设计相比有明显不同。**分离原因**：Rustc 针对批量编译优化，一次性顺序处理整个包；而 RA 面向编辑器环境，需要支持不完整代码和实时反馈，因而采用增量、按需计算的架构。RA 的**解析器**基于手写递归下降算法，将输入分解为事件流（Salsa 官方博客称之为“增量构建查询系统”）。这些事件通过 `rowan` 库组装成**语法树**（绿/红树结构），实现“无损解析”，可以精确跟踪源文件中的每一个标记。在此基础上，RA 采用 **Salsa**（Rust 增量计算框架）管理查询和语义分析，实现按需计算。每次编辑仅重新计算受影响的查询，从而快速响应 IDE 请求。

RA 提供按需语义分析和诊断，其多数错误诊断来自集成的 `cargo check`（即调用 rustc），同时自身实现了越来越多的语义检查。它利用 Rustc 输出的 JSON 错误以及 Clippy 报告为基础，在此之上叠加自己的分析结果。这意味着 RA 可以显示大部分 Rustc 检查出的编译错误，同时对某些情况（如代码补全或轻量分析）依赖自身引擎。

关于 **rustc 与 rust-analyzer 合流** 的进展：Rust 官方正在朝“查询化前端”（query-driven frontend）方向发展，rustc 自身也引入了查询系统，并计划采用无损语法树等技术。过去几年有多项提议（如共享解析器、Trait 解决器等），但至 2025 年底，`rustc` 和 RA 代码库依然保持分离，仅共享少量组件（例如词法分析器）。官方认可 RA 为 Rust 的主要 LSP 实现，但并未完全合并两者。总体而言，2025-26 年状态是**渐进式整合**：Rustc 在自身架构中逐步借鉴 RA 的理念（查询机制、无损语法树），而 RA 继续保持独立发展，两者暂时平行推进、逐步向兼容方向努力。  

## 对照案例：clangd、TypeScript、Zig

- **Clangd (基于 Clang 的 LSP)**：Clangd 直接复用 Clang C/C++ 编译器，通过编译数据库（compile_commands.json）确定编译选项。它为每个打开文件保持一个独立的 AST，后台多线程解析文件（`TUScheduler` 管理多个编译单元 Worker）。所有诊断由 Clang 编译器生成，因 C/C++ 允许单独编译每个源文件而无需整体解析。Clangd 还维护全局**索引**，离线解析整个工程以支持跨文件查找（定义、引用等）。优点是充分利用已有编译器的强大功能；缺点是增量处理较为困难，需要额外索引机制且内存占用大。

- **TypeScript (编译器即语言服务)**：TypeScript 的 `tsserver` 将编译器和语言服务合为一体。它建立 **语言服务宿主** 管理项目文件和变化，并使用增量**快照**（ScriptSnapshot）记录文件内容变化，仅重新解析修改区段。诊断和补全等操作被设计成按需执行：例如调用 `getSyntacticDiagnostics` 时只解析当前文件而不做绑定/type-check；而 `getCompletionsAtPosition` 只解析与该位置相关的局部作用域。TypeScript 引擎可以**区分语法/语义**错误、逐文件计算诊断。其优点是单一工程、功能完善、增量响应快；缺点是工具逻辑复杂、体积大，需要依赖完整的语言编译基础（但得益于 Microsoft 团队资源支持）。

- **Zig (自建增量编译器 + 语言服务器)**：当前 Zig 语言的官方 LSP 实现 ZLS 由社区开发，功能主要集中在词法和初级语义检查（如语法错误、未使用符号）。复杂的编译时计算（comptime）ZLS 目前无法处理，开发者常常通过“保存时自动运行 build”来获取完整编译诊断。Zig 团队路线图表明未来会实现**驻留编译器**（长驻进程），支持增量编译和 IDE 查询。优点：目前实现简单，可快速启用；缺点：功能不完整（尤其是类型检查、comptime）且依赖外部构建工具。Zig 正在积极向着类似 Rust-analyzer 模式演进。

这些案例展示了不同取舍：Clangd 复用成熟编译器适合 C/C++ 的多源文件模式；TypeScript 单一编译器设计支撑 IDE 功能但需要强大团队；Zig 则处于过渡期，力图用增量编译引擎满足 IDE 需求。

## 新语言架构决策：单一编译器 vs 双实现

对于一个 4-6 人的小团队，维护双重实现（一个用于批量编译，一个专门用于 IDE）工作量巨大。因此通常建议**单一编译器**兼顾批量编译与 IDE 需求，采用增量/查询化架构。正如社区建议的，“如果决定将编译器与 LSP 结合，从架构上要参照 rust-analyzer，它实现了增量且健壮的 Rust 前端，这些特点对于编辑器交互非常重要”。换句话说，可以在同一个代码基础上为批量编译和交互式查询设计，利用**查询系统**（如 Rust 的 Salsa）管理不同级别的编译信息。这避免了双重维护的成本和潜在不一致，同时也利用了共用部分（语法树、类型检查、语义分析等）。当然这种设计一开始实现难度较高，需要良好的模块化和增量能力，但长期成本低于维护两套引擎。

双实现方案（类似 Rust 现有的 rustc + rust-analyzer）在大项目或分工明确的大团队中可行，因为允许 IDE 后端采用不同优化。但对于人力有限的团队风险较高：需要重复造轮子、同步两套语义定义。因此推荐尽量采用单一编译器架构，从设计之初就支持增量查询。如果团队实在要分工作流，可以考虑在编译器中内置一个“工具模式”（tooling mode），即在主编译器中实现增量接口和必要的查询，而不是独立项目。

## 诊断基础设施的最小落地清单

要达到“Rust 级”诊断质量，新语言编译器的最小诊断基础设施至少应包括：

- **Span 管理**：源码位置对象，应支持精确定位字符范围、文件路径等，用于标注错误所在的代码片段。
- **诊断结构**：每条诊断包含消息文本、错误级别（Error/Warning）、可选错误码、主要 span 和额外 span（labels）、以及附加信息（notes、help）。如 Rust 的示例所示，一个诊断可包含多行 label 和 note。
- **建议系统**：支持在诊断中提供 **补丁建议**，包括 `MachineApplicable`（可自动应用）的修改。例如 Rust 在 JSON 输出中使用 `suggestion` 字段给出代码替换建议。这些建议应有适用性标志，工具（IDE 或命令行修复工具）可据此自动修改源码。
- **错误码与文档**：定义错误码体系，把可解释的编号附给常见错误，提供命令（如 `--explain`）查阅详细解释。这种体系帮助用户深入理解错误原因，并统一错误分类。
- **JSON 输出**：支持机器可读的诊断输出（如 JSON），便于 LSP 或外部工具消费。例如 Rustc 输出包含原文渲染和分解字段，可以供 `cargo fix` 等工具使用。
- **UI 测试框架**：类似 Rust 的 compiletest UI 测试，允许在测试文件中用注释标注期望诊断（包括 ERROR、WARNING、NOTE、SUGGESTION 等）。这确保诊断输出的稳定性和准确性。测试框架应支持匹配预计提示的错误信息和建议文本，并验证行号或 span 是否正确。

这些要素构成最小可用诊断系统：即语法树能提供 span，编译逻辑能报出结构化诊断，并通过测试自动验证其质量。

## LSP 能力分期路线图

结合系统语言需求与团队资源，可规划 LSP 特性分阶段实施：

- **MVP 阶段**：优先实现基础功能——**诊断（publishDiagnostics）**、**补全（textDocument/completion）**、**跳转（textDocument/definition & references）**、**高亮**（至少基本语法高亮）、**重命名（textDocument/rename）**。这些功能是编辑器的基本要求，可通过调用编译器内部或构建的索引获取结果。VS 的 LSP 支持列表表明这些功能均已支持。此阶段重点在于连贯的编辑体验和错误提示。

- **中期阶段**：在基础上增加**语言语义增强**功能。例如**签名提示（textDocument/signatureHelp）**、**文档悬停（hover）**、**代码操作（codeAction）**（例如快速修复建议）。同时实现 **语义标记（Semantic Tokens）** 和 **参数/类型内联提示（Inlay Hints）**。语义标记需要编译器能够区分符号类别（如函数、类型、变量）；内联提示需类型信息或参数名推断。此外可加入 **代码导航** 的更多特性，如查找所有引用（textDocument/references）、工作区符号（workspace/symbol）等。中期目的是提升代码阅读和编辑效率，让 IDE 更加智能。

- **远期阶段**：面向高级IDE特性，例如**调用层次（Call Hierarchy）**、**类型层次（Type Hierarchy）**、**语义高亮（在编辑器支持时）**、**代码透视（Code Lens）** 等，这些依赖更深入的静态分析。也可考虑 **重构支持**（如提取函数）和**性能反馈**（例如分析变量使用情况提示）。对于 GPU 特有需求（见下节）也可逐步引入，如计算/显示内存访问模式、并行度等。总之，长期目标是实现与 Rust-analyzer/clangd 类似的全面开发体验，但必须根据团队能力逐步推进。

## GPU 语言特有的 IDE 需求

GPU 语言（如 CUDA、HLSL/Slang）在 IDE 中有一些特殊需求：  
- **Kernel Launch 配置提示**：GPU 核函数通常需要指定网格和线程块维度。IDE 可以提供对 `kernel<<<...>>>` 参数的自动提示或验证，帮助用户正确设置并行度。  
- **地址空间可视化**：GPU 语言存在多种存储空间（`__global__`、`__shared__`、`__local__` 等）。IDE 可以通过颜色、内联标记或工具提示直观显示变量的地址空间，或在代码中突出显示不同空间的使用情况。  
- **寄存器压力/Occupancy 提示**：GPU 性能调优中，寄存器使用量和活跃度（occupancy）十分关键。现代工具（如 NVIDIA Nsight Compute）能对核函数进行分析，给出每行代码的寄存器使用和硬件占用率信息。IDE 可参考这些工具，为关键函数提供资源使用内联提示（比如在函数定义处标注预计使用的寄存器数和可达的多处理器占用率）。  
   
先例方面，**NVIDIA Nsight** 系列（包括 Visual Studio 插件和独立分析器）就是 GPU IDE 功能的典型体现，它集成了内联性能统计和参数计算。对于着色器语言，**Slang** 提供了 Visual Studio/VSCode 扩展，支持语法着色、补全、快速信息、签名帮助、跳转定义、诊断等常规功能，虽然目前未专门涵盖 GPU 性能指标，但表明第三方 GPU 语言在 IDE 中可以获得与一般语言类似的编辑支持。未来 IDE 对 GPU 语言的支持会更多地融入编译器提供的静态分析数据和硬件特定信息，以提升开发效率。

## Visual Studio 集成：可行性与成本

除了 VS Code 外，传统 Visual Studio 对第三方语言的支持路径主要有两种：**LSP 扩展**和**传统语言服务扩展**。从 Visual Studio 2017 15.8 开始，IDE 内置了对 LSP 的支持，开发者可以编写 VSIX 扩展将已有的 LSP 服务器集成进去。该方式使我们能够复用统一的语言服务器（与 VS Code 共用），在 VS 中实现 IntelliSense、诊断、重命名、定义跳转等功能。Visual Studio 文档列出了支持的 LSP 功能：基础请求如 `textDocument/publishDiagnostics`、`completion`、`definition`、`references` 都已支持。开发成本方面，LSP 扩展需要编写 VSIX 项目并嵌入或安装语言服务器，但不必理解 Visual Studio 复杂的内部 API。根据官方文档，这是推荐的方式来为第三方语言上手 VS。

另一种方式是使用 Visual Studio SDK 提供的**原生语言服务接口**（如 C#/VB 的 Roslyn API），这通常需要大量工作且难以维护，尤其对小团队而言成本很高。相比之下，LSP 扩展的学习曲线更平缓：我们只需关注语言服务器（按 LSP 协议通信），然后通过 VSIX 将其包装进去即可。需要注意的是，VSIX 扩展要管理语言服务器的安装（可随 VSIX 打包或通过引导程序安装），以及提供文本着色（可以使用 TextMate 语法文件）。总体来说，对于 4-6 人团队，采用 LSP 方式集成 VS 是可行且成本较低的方案，而完全自研传统语言服务代价过大，通常不推荐。
