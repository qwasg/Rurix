# rustc 编译器工程深度解剖

## 执行摘要

如果把 rustc 的工程经验压缩成一句话，我的结论是：**最值得借鉴的不是 Rust 的全部前端层数本身，而是“先把用户语法变成更稳定、更可分析的内部表示，再把复杂静态语义尽量下沉到 CFG 化的中层 IR 上做”的总体方法**。在 rustc 里，这条链路表现为：源码先被词法化成 token stream，再经 parser 形成 AST；随后经历宏展开、名称解析、AST validation 与早期 lint；再降到 HIR 做类型收集、类型推断、trait solving 与类型检查；再从 HIR 构造 THIR，把类型与更多 desugar 补齐，用于 MIR 构造、穷尽性检查和 unsafety checking；最后进入 MIR 做借用检查、移动/初始化数据流检查、常量求值、MIR 优化与单态化项收集，再降到后端 IR 做代码生成。这个分层不是“学术洁癖”，而是为了解耦：语法友好性、诊断质量、CFG 级安全分析、增量粒度、后端耦合度，各在不同层解决。citeturn19view0turn19view1turn19view2turn19view5turn39view0turn39view1turn39view2turn19view6turn19view7

对一个 **4–6 人的新语言团队**，我不建议照搬 rustc 的“全套体量”；我建议抄的是 **三件事**：第一，保留 **AST → HIR → MIR** 的三层主骨架，把 THIR 压缩成“临时 typed body IR”；第二，从第一天就把语义分析 API 写成“query 风格”的纯函数接口，但**早期只做内存内 memoization，不做全量红绿增量**；第三，把所有权/借用这类最难的静态语义放到 **MIR/CFG 层**，不要在 AST/HIR 上硬做。相反，**Polonius、完整红绿增量、并行前端** 都属于“规模团队+多年演进”产物，不适合作为 MVP 前置目标。citeturn38view0turn38view2turn13view0turn12view3turn12view4turn36view0

## 分层编译流程与为什么需要这么多层

### 文字架构图

可以把 rustc 的主流程整理成下面这张“文字架构图”：

> 源码  
> → **Lexer / TokenStream**  
> → **Parser / AST**  
> → **宏展开 + 名称解析 + AST validation + early lints**  
> → **HIR lowering**  
> → **HIR 上的 type collection / type inference / trait solving / type checking**  
> → **THIR**  
> → **THIR 上的 exhaustiveness / unsafety checking，并作为 MIR builder 输入**  
> → **MIR**  
> → **MIR 上的 borrow check / move & init dataflow / const eval / MIR opts / mono-item collection**  
> → **Codegen IR / LLVM IR**  
> → **对象文件与链接**  

这条链路能直接从 rustc-dev-guide 的总体概览、HIR/THIR/MIR 章节以及后端章节拼起来：lexer 产出 token stream，parser 从 token stream 构 AST；HIR 是 parsing、macro expansion、name resolution 之后的主 IR；THIR 在 type checking 之后生成；MIR 是 CFG 化的中层 IR，用于借用检查、数据流分析、优化和代码生成准备；单态化在 codegen 前后衔接，实际 MIR → LLVM IR lowering 时发生具体 monomorphization。citeturn40search2turn40search6turn19view0turn19view1turn19view2turn19view5turn39view0turn39view1turn39view2turn19view7turn19view8

### 每一层的职责

**Lexer / TokenStream** 的职责是把 `SourceFile` 变成 token stream，给 parser 使用。这一层在 rustc 里不是“单独炫技”的复杂 IR，而是 parser 的直接前置输入；关键工程点在于它保留 span/source map 信息，为后面所有诊断、edition hygiene、feature gating 打基础。citeturn40search2turn40search9turn23view0

**AST** 基本上“接近用户写法”。它适合承接 parser 恢复、宏展开、名称解析前后的结构转换，也适合做语法相关的 validation。rustc 概览明确写到：宏展开、AST validation、名称解析和 early lint 都发生在 lexing / parsing 这一大阶段里；parser 返回的是 `rustc_ast::ast::{Crate, Expr, Pat, ...}` 这类 AST 节点，并配合标准诊断 API 做错误恢复。citeturn19view0turn19view1turn39view3turn41view0

**HIR** 是 rustc 的“主工作 IR”。它来自 AST lowering，是更 compiler-friendly 的表达，但仍然保留了大量接近源代码的结构；同时又做了一部分 desugar，比如 `for` loop 在 HIR 中就已经被转换掉。更重要的是，HIR 的 out-of-band storage、`ItemId`/`HirId`/`DefId` 体系以及 item/body 的分离，让编译器既能高效遍历，也能把依赖追踪和增量边界做得更细。类型收集、类型推断、trait solving、类型检查，主要都发生在 HIR 这一层。citeturn39view0turn39view4turn19view1turn21view0turn21view1

**THIR** 是“typed bodies + 更强 desugar”的临时 IR。它只表示 body，不表示 `struct` / `trait` 这类 item；它在 type checking 完成后生成，自动引用/解引用、方法调用、重载运算符、destruction scope 等都会被显式化。它被用在 MIR construction、穷尽性检查和 unsafety checking 上，而且只临时存在、很快释放，以控制峰值内存。对 rustc 来说，THIR 的价值不只是“多一层”，而是把“typed user body”和“CFG 化 MIR”之间的落差隔开。citeturn39view1turn19view5

**MIR** 是 CFG 形式的、显式类型化的中层 IR。它是 rustc 做所有 **flow-sensitive** 静态语义的主战场：借用检查、未初始化检查、移动分析、常量求值、MIR 级优化、mono-item collection 等都在这里或围绕这里发生。由于 MIR 仍然是泛型态，分析它通常比在单态化之后更便宜。MIR 的根本作用，是把“复杂 Rust 语法”蒸馏成“数据流和控制流问题”。citeturn39view2turn19view2turn19view5turn13view0turn37search8

**Codegen** 层在 rustc 中并不是自己手写机器码，而是把 MIR 降到 LLVM IR 等后端 IR；`rustc_codegen_ssa` 提供 backend-agnostic 接口，LLVM/Cranelift/GCC 后端在其上实现。单态化项收集在 MIR 层发生，而“实际的 monomorphization”是在 MIR → codegen IR lowering 时执行。这个切分让“前端优化/安全检查”和“后端平台细节”分离。citeturn19view6turn19view7turn19view8

### 为什么 rustc 不只保留 AST 和 LLVM IR

rustc 不是简单地把 AST 直接扔给 LLVM，原因有四个。第一，**AST 太贴近表面语法**，对类型检查和数据流分析并不友好；第二，**HIR 仍然太接近“树”，不够 CFG 化**，所以借用/移动/初始化这类检查需要 MIR；第三，**THIR 让 typed body desugar 与 MIR builder 解耦**，否则 pattern、method call、autoderef、unsafety、exhaustiveness 会在 MIR builder 里缠作一团；第四，**多层 IR 带来更稳定的内部边界**，方便查询系统、增量编译和诊断实现。MIR RFC 也明确把 MIR 的定位说成“desugar most of Rust’s surface representation，适合 type-checking 和 translation”，而增量编译 RFC 还指出，相比继续依赖 annotated HIR，使用 MIR 能简化增量方案。citeturn19view2turn19view5turn39view0turn39view1turn31search2turn31search18

### 对新语言团队的抄法建议

如果你是 4–6 人团队，我建议把 rustc 的分层**收缩**成：**AST → HIR → typed-body IR → MIR**。其中 typed-body IR 可以不叫 THIR，也不必做成完整长期存活的数据结构；但请务必保留“类型填充后、MIR 之前”的那道窄门。因为一旦你的语言里有模式匹配、隐式解引用、方法糖、析构/资源释放语义，直接 HIR → MIR 会很快变脏。这个建议属于 **“简化后抄”**。反过来，AST 直接做借用检查、或把所有静态语义都塞到 parser/AST pass 里，则是 rustc 历史已经付出过代价、后来看起来明确不优的路径。citeturn39view4turn39view1turn39view2turn13view0

## Query 系统与增量编译

### on-demand query 的工程模型

rustc 的 query model 可以概括成：**把编译器视为一个按需填充的不可变数据库**。一个 query 至少有四部分：名字、key、result type、provider。query context 初始为空；只有当外部驱动真正“索取”某条信息时，对应 provider 才会运行；provider 只能通过 query context 访问其他 query 或输入数据，且要求 key/result 不可变、provider 对同 key 纯函数。这个限制不是形式主义，而是为了让查询结果可 memoize、可重放、可并行。citeturn38view0

query 的第二根支柱是 **memoization**。同一 query key 的结果一旦算出，就会缓存在内部表里；下次再请求，直接命中缓存而不是重跑 provider。rustc-dev-guide 明确指出：没有 memoization，query 系统仍然“正确”，但会不断重复计算；也正因此，provider 必须是纯函数，否则缓存就不可信。citeturn38view0

如果你想把这个模型翻译成适合新语言团队的工程动作，最重要的是：**先把语义 API 写成 query 风格，再决定底层是否真的做增量与持久化**。也就是说，哪怕第一版仍是“串行、单次编译、无磁盘缓存”，`type_of(def_id)`、`mir_borrowck(body_id)`、`layout_of(ty)` 这类接口也应该长成 query 的样子。这样你以后要引入缓存、增量、并行时，不必推翻前端抽象。这个思路对小团队属于 **“直接抄思路”**。citeturn38view0turn37search3

### 红绿算法与查询缓存

rustc 的增量编译是 query system 的扩展，而不是额外补丁。其核心算法是 **red-green**：编译结束后把 query 结果和 query DAG 存下来；下次编译时，当某个 query `Q` 被请求，系统先尝试证明它还是 **green**，只有无法证明时才把它重算并标成 **red**。是否能继续复用，并不是“文件时间戳”那种粗粒度判断，而是沿着旧依赖图递归尝试 `try-mark-green`。citeturn37search16turn38view2

真正让跨编译会话复用成立的，是 **稳定标识与稳定哈希**。rustc 不把跨会话会变的 `DefId` 直接存盘，而是用 `DefPathHash` 等稳定等价物；query key 和 query result 都会计算稳定的 `Fingerprint`，并把这些 fingerprint 跟依赖图一起存盘。这样，下一次编译就能把“本次 query key”映射到“上次 dep-node”，并用 fingerprint 比较结果有没有变化。citeturn38view2

这一设计的代价也被官方文档直接点明了：**fingerprint 计算很贵，而且它正是增量编译有时会比非增量更慢的主要原因**。另外，虽然 128-bit 指纹配上高质量哈希把碰撞风险压到“实践上可忽略”，但理论上碰撞并非零。对小团队来说，这意味着：**红绿算法不是“加个缓存目录”那么简单，而是需要一整套稳定身份、稳定哈希、磁盘格式、序列化边界与 dep-graph 设计**。因此它属于 **“简化后抄”**，不属于 MVP 必做项。citeturn38view2

### 失效粒度与实际效果上限

rustc 的前端增量粒度，本质上是 **“每个 query key / dep-node”**；HIR 之所以做 item/body 分离，也是为了避免“拿到一个模块就顺便看见整个函数体”，从而让依赖边界更细。这个方向本身非常正确。citeturn39view0turn38view2

但 rustc 的**实际效果上限**不是“只改一行就全部瞬时返回”。官方文档给出了两个硬边界。第一，后端并**不是完全由 query 构成**；LLVM 那部分只是“手工接入”增量，粒度主要落在 **CGU（codegen unit）** 上：如果某个 CGU 对应 dep-node 不能标绿，就需要整 CGU 重新编译。第二，即便前端大量命中缓存，fingerprint/stable-hash 的开销、缓存加载、链接等不可跳过工作仍然存在。也就是说，增量编译的收益上限，取决于“这次改动把多少前端 dep-node 和多少 CGU 撞红”，而不是抽象意义上的“文件没怎么变”。citeturn38view2turn19view2turn19view7

从 4–6 人团队角度，我建议的策略是：**先做“进程内 query 缓存”与“模块/函数级失效”，再考虑跨会话红绿；先让后端天然吃到更粗的 cache，再追求更细的前端持久化**。如果你的目标平台是 Windows-only、NVIDIA GPU-only、CUDA-oriented，这个顺序尤其重要，因为你很可能还要同时处理 NVVM/PTX/驱动链路本身的缓存与重编译边界，前端增量不该抢走过多早期预算。这个结论是工程判断，但它直接建立在 rustc 对红绿成本和 CGU 粒度边界的公开实现之上。citeturn38view2turn19view8

## MIR 层借用检查、NLL 与 Polonius

### 为什么 borrow check 放在 MIR 层

rustc-dev-guide 对这个问题给出的答案非常直接：**旧借用检查在 HIR 上做过，但 MIR 上做有明显优势**。第一，MIR 比 HIR 简单得多，激进 desugar 能减少 borrow checker 的 bug 面；第二，更关键的是，MIR 让借用检查可以基于 **控制流图** 推导 non-lexical lifetimes，而不是被语法块边界绑死。换句话说，借用检查下沉到 MIR，不只是“实现细节”，而是 NLL 成立的前提之一。citeturn13view0turn21view2

MIR borrow check 的主要阶段也公开写得很清楚：先为 MIR 做一份本地副本；把其中 region 替换为新的推断变量；运行 move/初始化等数据流分析；再做一次 MIR type check 来收集 region constraints；然后做 region inference；随后计算每个点上“在作用域内的 borrows”；最后再次 walk MIR，根据前面分析结果发出错误。这个流程说明了为什么 borrow check 最适合待在 MIR：它本质上就是“**MIR 数据流 + MIR 类型/区域约束 + MIR 级报错 walk**”。citeturn13view0turn21view2

所以，对一门想“全面借鉴 Rust 前端方法”的新系统语言，**MIR 层 borrow checker 是非常值得抄的**，评级是 **“简化后抄”**：简化的是规则覆盖面与诊断精细度，不要简化掉“CFG 中层 IR”这件事本身。citeturn13view0turn39view2

### NLL 的演进

NLL 是 Rust 借用检查器的“第二代”。2018 edition 时代，NLL 已经面向 2018 代码启用，但 2015 代码还在 migration mode：编译器同时跑旧 checker 和新 checker，用 warning 平滑迁移。到 Rust 1.63，旧的 lexical borrow checker 已经从 rustc 中完全移除，**所有 edition 都统一使用 NLL**。citeturn17view0turn17view1

这也解释了一个工程现实：**MIR-based borrow checker/NLL 不是“一次性切换”完成的，而是经历了长时间的双轨运行、诊断对齐与兼容收敛**。官方致谢里列出了一整串贡献者与 NLL working group 成员，这说明该子系统的 productization 成本远高于“把算法跑通”。对小团队而言，这意味着“先做正确性、后做诊断对齐”的优先级更合理。citeturn17view0

### Polonius 到 2026 的真实状态

截至 **2026-06-10**，Polonius 仍然**没有在 stable 上落地**。官方 Inside Rust 在 2023 年已经明确说：历史上的 datalog 版 Polonius 原型不会直接成为当前计划；当前方向是把从 datalog 实验中学到的东西，**重新在 rustc 内部实现**。2025H1 的官方目标页进一步点明：旧 prototype 因为 **scalability issues** 没有稳定化路径，因此需要“improved architecture and implementation”。citeturn13view1turn12view2

到 2025H2，官方目标已经从“scalable nightly support”推进到“**stabilizable Polonius support on nightly**”。这时的表述非常关键：当前选定的是一个 **alpha analysis**，它“没有最初想要的那样强”，但它**能覆盖最重要的案例**、通过了 in-tree tests、perf run 和 crater，虽然“still too slow”。这说明 Rust 团队已经把目标从“完整理想版 Polonius”收敛为“先把价值最高且能优化的子集产品化”。citeturn12view3

2026 年被接受的项目目标则更进一步：目标是 **stabilize Polonius alpha**，并且页面直说还有 **一个已知 soundness issue** 尚未解决，同时还需要扩测试、做 a-mir-formality 形式化建模、验证真实代码性能并准备 stabilization report。换句话说，**截至 2026 年中，Polonius 的状态是：nightly 上已有功能性原型，方向明确，正在冲刺稳定化，但仍未稳定，且 correctness/perf/documentation 三条线都没完全收尾。**citeturn12view4turn12view5

代码层面的最新状态也能从 nightly 源码里看到：`parse_polonius` 当前已经支持 `legacy` 和 `next` 两种取值，说明新旧两套路径仍在并存试验。对外部语言团队的含义很明确：**不要把 Polonius 当成你第一代借用检查器的起跑线**。这个子系统对 4–6 人团队评级应是 **“不要抄”**；你可以关注其思想，但不应把它变成首发实现目标。citeturn15view0turn12view4

## 诊断体系、feature gate 与 edition 共存

### 诊断体系的骨架

rustc 的诊断体系不是“字符串拼接艺术”，而是一套结构化框架。`rustc_errors` 提供核心设施；新诊断推荐实现成 `Diagnostic` trait 的类型，复杂度较低时可以直接 `derive(Diagnostic)`；`DiagCtxt` 会创建 `Diag`，你在其上追加 label、note、help、suggestion，最后显式 `.emit()`。官方文档还特别提醒：拿到 `Diag` 却既不 `emit` 也不 `cancel`，会触发 ICE。citeturn27view0turn26view0

这套结构化 API 对工具生态也非常友好。`Diag` 的 suggestion API 不只是为了终端美观；当使用 `--error-format json` 时，结构化 suggestion 会被机器读取，供 `rustfix` 之类工具消费。与此同时，suggestion 还带有 `Applicability`，从 `MachineApplicable` 到 `MaybeIncorrect` 分级，告诉工具哪些建议能机械应用、哪些只能提示用户。citeturn27view0turn27view2

对新语言团队，我对“结构化诊断 + fix-it suggestion”的评级是 **“直接抄思路”**。这是 rustc 最值得借鉴、且 ROI 最高的工程投资之一；尤其你如果面向系统编程与 CUDA 生态，用户会大量面对 ABI、借用、资源生命周期和目标特性错误，诊断质量会直接影响语言成败。citeturn27view0turn27view2

### 错误码、可翻译诊断与 UI golden tests

rustc 的错误码统一登记在 `compiler/rustc_error_codes`；新增错误码要在 `src/lib.rs` 的 `error_codes!` 宏里分配编号，并在对应 `E####.md` 中补充长说明。官方文档甚至把“找下一个未占用号码、补长说明、用 `struct_span_code_err!` 发错”写成了标准流程。citeturn26view2turn26view3

可翻译诊断方面，rustc 采用 **Fluent**。诊断消息不是任意字符串，而是 Fluent message identifier；诊断所需的变量必须作为参数显式提供，这样不同语言才能在不丢信息的情况下重排句子。`#[derive(Diagnostic)]` 还会在编译 rustc 本身时校验 Fluent 资源，防止重复 identifier 或无效消息在运行时炸掉。citeturn26view0turn26view1

UI golden tests 则是 rustc 诊断体系的另一半。官方做法是在测试旁边直接放 `.stderr` / `.stdout` snapshot，用 `--bless` 生成后再人工检查；编译器还会启用 `-Z ui-testing`，把某些 line number、路径风格、CRLF 等噪声规范化，必要时还能写 `normalize-*` 指令。Compare mode 还能为 Polonius 等模式单独维护 `foo.polonius.stderr` 之类的期望输出。citeturn27view3turn27view4

这套 UI golden test 的**维护成本**也很真实：snapshot 需要 `--bless` 和人工复核，小改动会造成大量期望文件更新，所以官方才要做 `-Z ui-testing`、路径/哈希规范化来压 diff 噪声；compare mode 虽然有用，但文档还写明“currently none of the compare modes are checked in CI for UI tests”，也就是说维护面越宽，成本越高。这部分我给的评级是 **“简化后抄”**：小团队应优先覆盖“解析错误、类型错误、借用错误、目标后端错误”四类黄金路径，而不是一开始就铺满所有 compare mode。这个成本判断是基于官方测试组织方式得出的工程推论。citeturn27view3turn27view4

值得补充的是，诊断渲染器本身在 2025–2026 也在演进：Rust 2025H1 的官方项目目标是把 rustc 输出切到 `annotate-snippets`，理由是减少 Cargo/rustc/其他工具间的重复实现与总体维护负担；到 2026 年 1 月，Inside Rust 已经写到：nightly rustc 切换到 `annotate-snippets` 的实验进展顺利，并已被批准在 rustc 中通用。这说明诊断“呈现层”也仍在重构中。citeturn29view0turn29view1

### feature gate / unstable 机制与 edition 共存

rustc 的 feature gate 实现是非常“工程化”而不是“语义上玄学”的。官方 feature-gate-checking 章节明确写到：所有 gate 定义都在 `rustc_feature` crate 中；**unstable** 在 `unstable.rs`，**accepted** 在 `accepted.rs`，**removed** 在 `removed.rs`，内建 **attribute/cfg** gate 在 `builtin_attrs.rs`。crate 级 `#![feature(...)]` 会在 AST validation/expansion 之前就被收集，构造出一份 `Features` 集合；随后 parser-gated spans、post-expansion AST visitor、attribute 检查、cfg 检查分别消费它。citeturn41view0turn23view1turn23view2turn23view3turn24view0

这套机制有两个很值得新语言抄的点。第一，**gate 定义表与检查点分离**：定义集中，检查分散到 parser、AST pass、attribute/cfg pass 中。第二，**稳定/已稳定/已移除** 三种状态在代码里就是三张表，而不是靠文档口头约定。再加上 `RUSTC_BOOTSTRAP`、`Span::allows_unstable`、`#[allow_internal_unstable]` 等通道/内部宏机制，Rust 的不稳定功能治理是“制度即代码”。对新语言团队，这部分评级是 **“直接抄思路”**。citeturn41view0turn24view0

edition 的内部共存也很有代表性。官方 editions 指南写得很清楚：**edition-specific parsing 很少直接在 parser 里做**；大多数 edition 行为变化，是通过 **migration lint** 在旧 edition 上给出迁移提示、在新 edition 上变成 hard error。更重要的是，**span 自带 edition**，所以 edition 检查通常应该看 token/span 的 edition，而不是全局 session edition；这样跨 crate 宏展开时，edition 语义才不会错。换句话说，Rust 的 edition 共存不是“编译器里跑多个 parser”，而是“**同一前端主干 + span-based edition hygiene + lint/error 双轨迁移**”。这部分对新语言团队评级也是 **“直接抄思路”**。citeturn23view0

## 性能观测、编译时间统计与并行前端近况

### 自我观测基建

rustc 自己的性能观测分三层。**第一层**是 `-Z self-profile`：它会输出三份二进制 profile 文件（`.events`、`.string_data`、`.string_index`），再由 `measureme` 工具链做 summarize、flamegraph 或 Chromium trace 转换。**第二层**是 `rustc-perf`：官方 perf 基建会在大量真实 crate 上比较两版编译器的 fresh build、incremental build 等配置，并把变化持续展示在 `perf.rust-lang.org`。**第三层**是更用户向的 `cargo --timings`，适合看 crate graph 层面的时间分布。citeturn8view0turn8view1turn8view2

这意味着 rustc 的“性能治理”并不是靠 contributors 主观体感，而是靠**自 profile + 批量基准 + PR 回归监控**进行制度化管理。2025H2 的官方 `rustc-perf improvements` 目标还说明：现有 rustc-perf 在多机器、多配置分布式 benchmark 上还在升级中，说明这套基础设施本身也仍在持续演进。对小团队而言，**`-Z self-profile` 式的 query-level profiling 非常值得尽早引入；完整的 rustc-perf SaaS 则只需简化版**。前者评级 **“直接抄思路”**，后者评级 **“简化后抄”**。citeturn8view3turn8view0turn8view1turn8view2

### 并行前端到 2026 的真实状态

你在验收标准里特别要求 2024 年后的变化必须给真实状态。这里最重要的一条是：**并行前端截至 2026-06-10 仍在稳定化路上，尚未稳定。** 2023 年官方博客宣布 nightly 已能用 `-Z threads=8` 体验 parallel front-end，并当时希望 2024 稳定；但 2024H2/2025H1/2025H2/2026 的连续官方目标页都还在处理 deadlock、ICE、测试、Cargo 接入、incremental 条件下的 race，以及性能提升问题。citeturn34view5turn34view0turn34view1turn34view2turn36view0

到 2026 目标页，已完成的里程碑包括：解决一批 deadlock 与 ICE、把 `rustc-rayon` 纳入工作树、在 bootstrap 中启用 parallel front-end，并修复了一个与 **incremental compilation 并行执行**相关的重要 race 问题；但官方仍把“构建专门测试套件、启用 Cargo、补充 rustc-perf benchmark、写稳定化报告送 FCP”列为后续工作。对 4–6 人团队，这一子系统的评级应是 **“不要抄”**：你可以设计 query/context API 时保留并行安全约束，但不要把并行前端本身列为第一阶段目标。citeturn36view0turn38view1

## 团队演进、可抄评级与最小可行前端

### 团队与人年估算

先说一个边界：**我没有找到 Rust 官方公开的“子系统工时账本”**。所以“多少人年”只能做**粗略、显式标注为推算**的估计，而不能伪装成官方数字。下面的范围，是根据公开时间线、工作组、项目目标和官方点名贡献者数量推出来的。citeturn31search1turn17view0turn12view2turn12view4turn30search0turn32search5

**MIR 主骨架**大致可以看作 **中个位数人年** 的工程。理由是：它在 2015 路线图里已被列为 compiler restructuring 的关键项目，2016 年公开发布介绍，随后在 1.12 成为新 backend 的基础；这是一轮明确的体系重构，但时间窗比 NLL/Polonius 短、参与面也相对集中。citeturn31search1turn31search0turn30search5turn31search12

**Query + 增量编译**我会估成 **中个位数到接近十人年**。理由是：2015 已有增量路线图，2016 发布增量编译说明与 RFC，2017 仍在做 finer-grained incremental refactoring；而 dev-guide 今天展示的 stable hash、fingerprint、old/new dep-graph、cache promotion、backend CGU integration，明显不是一两个季度能补全的“缓存功能”。citeturn32search12turn32search1turn31search18turn32search8turn38view2

**MIR borrow check / NLL** 我会估成 **十人年以上**。理由非常强：它从 MIR 引入后的早期工作一路演进到 2018 edition 的 migration mode，再到 2022 年 1.63 彻底移除旧 checker；官方致谢里单独列出了多位稳定化推动者和一个完整的 NLL working group。这个跨度、参与面和诊断对齐成本，都显示它已远超“中个位数人年”。citeturn13view0turn17view0turn17view1

**Polonius** 的累计投入，我会保守估成 **中个位数到十人年且仍在继续**。原因是它在 2018 左右从 NLL 主线中“分叉”为未来工作，2023 有新的官方路线图，2025H1/2025H2/2026 连续三轮项目目标持续推进，而且还伴随 Amanda Stjerna 的 PhD 研究和 a-mir-formality 形式化建模。这个子系统本质上还没收官。citeturn17view0turn13view1turn12view2turn12view3turn12view4

**诊断体系** 我更愿意称它为 **长期持续投入、难以封箱计数**。官方多年目标与文档都表明：diagnostic structs、Fluent 翻译、UI testing、渲染器重构、annotate-snippets 迁移，都是叠加式演进，不像 MIR/NLL 那样有清晰“完工点”。因此它更像**常年维护的基础设施曲线**，而不是一个封闭项目。citeturn27view0turn26view0turn26view1turn27view3turn29view0turn29view1

### 哪些子系统后来被重写或显著重构

**借用检查器**是最典型的“后来重写”。旧实现跑在 HIR/AST 风格层面；新实现迁到 MIR，理由是 MIR 更简单、bug 面更小，而且只有在 MIR/CFG 上才能做 NLL。到 1.63，旧 lexical checker 已被彻底移除。citeturn13view0turn17view0turn17view1

**Polonius 实现路线**也发生了显著转向。2023 官方博客明确说：历史上的 datalog 项目不再是当前计划的直接落地方向；现在是把其中思想重新在 rustc 里实现。2025–2026 的目标又进一步把目标收缩为“可稳定化的 alpha analysis”，这是一次非常典型的“从研究原型回退到可产品化子集”的重构。citeturn13view1turn12view2turn12view3turn12view4

**诊断渲染层**正在重构到 annotate-snippets。官方给出的动机是减少 Cargo/rustc/其他工具间维护重复，并统一输出风格。到 2026 年初，这项迁移已经被批准在 rustc 中普遍使用。citeturn29view0turn29view1

**codegen 接口**也经历过平台化重构：`rustc_codegen_ssa` 现在是 LLVM/Cranelift/GCC 的公共后端接口，而不是把所有逻辑都锁死在 `rustc_codegen_llvm` 里。这个重构不是“推倒重来”，但确实是后端架构从单后端实现向多后端抽象层的演进。citeturn18search1turn19view6

### 对 4–6 人新语言团队的可抄程度评级

**分层 IR 体系：简化后抄。** 你最应该保留的是 AST、HIR、MIR 三个长期层，以及一个临时 typed-body IR；不必一开始就把 THIR 做到 rustc 那么完整，但不要省掉 typed-body → CFG 这道过渡。原因是 rustc 已经公开展示了每层各自承载的检查与内存/增量角色。citeturn39view0turn39view1turn39view2turn39view4

**Query 模型：直接抄思路。** 纯函数 provider、不可变 key/result、按需填充数据库、memoization，这一整套抽象对小团队也立刻有价值，而且不要求你第一天就做磁盘增量。citeturn38view0

**完整红绿增量：简化后抄。** 这是正确方向，但 stable hash、fingerprint、旧/新 dep-graph、cache promotion、CGU 级后端接入都很贵；你应先做进程内缓存和粗粒度失效。citeturn38view2

**MIR 层 borrow checker：简化后抄。** 如果你的语言真要学 Rust 的所有权模型，这一项应当进入第一代设计；但规则、诊断和 corner case 覆盖可以先缩减。citeturn13view0turn21view2

**Polonius：不要抄。** 连 rustc 到 2026 仍在为稳定化、soundness 和性能收尾，小团队没必要把未来版 checker 当首发门槛。citeturn12view4turn15view0

**结构化诊断 + fix-it：直接抄思路。** 这是语言 UX 的杠杆点，且有明确 API 形态可学。citeturn27view0turn27view2

**可翻译诊断：简化后抄。** 如果你首发只做中文/英文，完全可以先不做多 locale；但“消息 key + 参数化变量 + 编译期校验资源”这套骨架值得及早奠定。citeturn26view0turn26view1

**UI golden tests：简化后抄。** 一定要做 snapshot 化的诊断测试，但先只覆盖几条核心错误链路，不要一上来复制 rustc 那么大的 compare-mode 与 normalization 面。citeturn27view3turn27view4

**Feature gate / unstable 机制：直接抄思路。** 如果你计划有 nightly/experimental 轨道，这几乎是最成熟的治理模型之一。citeturn41view0

**Edition 机制：直接抄思路。** 特别是“span 带 edition + migration lint + 新 edition hard error”的组合，非常适合长期演进语言。citeturn23view0

**self-profile / perf 基建：简化后抄。** 自 profile 应尽早有；大规模 perf SaaS 可后置。citeturn8view0turn8view1turn8view2

**并行前端：不要抄。** 直到 2026 仍在稳定化；小团队应把预算优先给 IR 边界、语义正确性、诊断与后端联调。citeturn36view0turn34view5

### 最小可行 rustc 式前端的子系统清单与依赖顺序

下面这份清单，是我认为最适合你这种“新语言 + 小团队 + 静态编译 + 最终面向 CUDA”的 **最小可行 rustc 式前端**：

- **公共基础设施先行**：`Span/SourceMap`、symbol interner、`DiagCtxt`、错误码/消息 key 体系。这些必须在 lexer 之前落地，因为 parser、edition、feature gate、后续所有诊断都要依赖它们。citeturn27view0turn26view1turn23view0

- **Lexer / TokenStream**：只做稳定 token 化与 span 保真，不要在 lexer 里做过多语义。citeturn40search2turn40search6

- **Parser / AST**：保证错误恢复和源代码保真。此时 AST 要尽量“像用户写的代码”。citeturn19view0turn19view1

- **Feature 收集 + 早期 gate + 宏前处理**：先把 crate-level feature 与最基本 parser gate 跑起来；如果你的语言也有宏，应在 AST/HIR 之间完成展开与第一阶段名称解析。citeturn41view0turn39view3

- **AST → HIR lowering**：这里开始做语法糖消解、稳定 ID、item/body 分离，这是后续 query 与增量的地基。citeturn39view4turn39view0

- **HIR type collection / inference / type checking**：先跑通签名、item、表达式体的基本类型系统。trait solver 的雄心可以收缩，但 HIR 层 typeck 的边界不要动。citeturn21view0turn21view1turn19view1

- **typed-body IR**：做一个“迷你 THIR”，哪怕只覆盖 body、pattern、方法糖、autoderef、drop scope，也值回票价。citeturn39view1

- **MIR builder + MIR 数据流**：先搭 CFG、basic blocks、显式 temporaries、drops，再实现 move/init analysis。citeturn39view2turn13view0

- **MIR borrow checker**：如果你真要借 Rust 的所有权方法，这是首发必选；但先做 NLL 风格核心规则，不碰 Polonius。citeturn13view0turn17view0turn12view4

- **最小 MIR 优化 + const eval**：只做对后端明显减压的少数 pass；不要过早做漫长的中端优化菜单。const eval 则应建立在 borrow-checked MIR 之后。citeturn19view2turn37search8

- **后端接口层**：尽早把前端与后端隔开，哪怕你最终不是走 LLVM 通路，也建议保留“backend-agnostic facade + target-specific lowering”这个结构。citeturn19view6turn19view7

- **第二阶段再加**：query 持久化、红绿增量、translatable diagnostics、parallel front-end、Polonius。它们都不该卡住 MVP。citeturn38view2turn26view1turn36view0turn12view4
