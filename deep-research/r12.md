# R12 — 标准库设计与 CUDA 生态库绑定调研报告

**副标题**：静态类型、Rust 风格、CUDA-first 新语言的标准库绑定层设计依据

**调研范围**：数值核心类型 / Buffer 内存抽象 / cuBLAS-cuDNN 绑定 / C ABI 导出 / Python 扩展 / 数学函数

**交付物**：类型层次草案 / cuBLAS 绑定方案 / Python 互操作分期方案 / 全部出处

---

## 1. 执行摘要与核心决策概览

本报告围绕一种新设计的静态类型、Rust 风格、CUDA-first 系统编程语言的标准库与 NVIDIA 生态库绑定层展开系统性调研。调研覆盖数值核心类型设计、Buffer 内存抽象、cuBLAS/cuDNN 绑定工程、C ABI 导出、Python 扩展生成以及设备端数学函数六个领域，每一项 API 设计建议均基于至少一个真实项目的生产实践先例。

**核心结论概括如下**：

- **数值类型**：标准库应提供固定维度的 `Vec2/Vec3/Vec4` 和 `Mat2/Mat3/Mat4` 作为语言核心类型（precedent：Slang/HLSL/GLSL 的内建向量矩阵类型），动态维度 ndarray 留给外部库（precedent：Rust 核心库不包含 vector/matrix）。采用列主序（column-major）作为默认矩阵布局（precedent：Eigen 默认、glam 默认、cuBLAS 接口惯例）。
- **Buffer 抽象**：标准库应定义统一的 `Buffer<T>` trait，并由四种具体类型实现：`DeviceBuffer<T>`、`PinnedHostBuffer<T>`、`MappedHostBuffer<T>`、`ManagedBuffer<T>`（precedent：RustaCUDA 的 DeviceBuffer/UnifiedBuffer、RAPIDS 的 Buffer 类型区分）。采用流序（stream-ordered）语义（precedent：RAPIDS Buffer、CUDA 11.2+ MemoryAsyncPool）。
- **cuBLAS 绑定**：推荐手写薄封装层（shim）而非自动生成工具绑定 CUDA 头文件。每个 CUDA stream 对应一个 cuBLAS handle，workspace 在 handle 创建时预分配并在销毁时释放（precedent：CuPy 的 handle pool 设计、cuDNN workspace 预分配模式）。
- **C ABI 导出**：采用不透明句柄（opaque handle）模式暴露 C API，所有结构体使用 `repr(C)` 定义，通过 cbindgen 自动生成头文件（precedent：Rust FFI 最佳实践、VibeTensor 的稳定 C plugin ABI）。
- **Python 扩展**：MVP 阶段使用 nanobind + scikit-build-core 构建 CUDA-aware Python 扩展，仅暴露 `__cuda_array_interface__` 和 DLPack 导出能力（precedent：VibeTensor 使用 nanobind、pytorch-nanobind-cuda-example 项目）。
- **数学函数**：设备端数学函数链接 NVIDIA libdevice bitcode 库，通过 NVVMReflect pass 控制 fast-math / precise-math 模式切换（precedent：LLVM NVPTX 后端官方文档）。

以下各章将逐一展开上述结论的详细调研过程、设计草案与先例依据。

## 2. 数值核心类型设计先例研究

### 2.1 Rust 核心库的"无 vector/matrix"设计哲学

Rust 的标准库（std）和核心库（core）刻意不包含线性代数向量或矩阵类型，这一设计决策源于 Rust 语言团队对标准库最小化原则的长期坚持。Rust 的 `core` 库甚至不进行堆分配，因此诸如 `Vec`、`HashMap` 等需要堆分配的数据结构都不属于 core 范畴。在线性代数领域，Rust 生态将这一职责完全交由外部库承担，其中最成熟的两个 crate 是 `ndarray`（专注于 N 维数组容器）和 `nalgebra`（专注于线性代数运算）。

`ndarray` 的设计深受 NumPy 影响，其核心类型 `ArrayBase<S, D>` 使用存储（Storage）和维度（Dimension）两个泛型参数，支持拥有所有权的数组、数组视图（view）和可变视图（mutable view）等多种形态。它实现了复杂的切片、广播和跨维度操作，但其 API 复杂度较高，需要用户理解 strides、layout、contiguity 等概念。`glam` 则走了一条截然不同的道路：它专门针对游戏和图形学场景优化，公共 API 不使用泛型（no generics in public API），所有类型都是具体类型如 `Vec3`、`Mat4`，这使得编译错误信息极为简洁，同时也避免了单态化带来的代码膨胀。glam 还支持 `cuda` feature gate，当启用时会强制类型的内存对齐方式匹配 CUDA 的要求。

这一领域的 precedents 给我们的启示是：标准库的定位应当清晰——是提供最小公分母的基础类型，还是提供完整的线性代数能力？Rust 的选择是前者（仅提供原始标量和数组切片），将高级类型完全外置。但对于一种 CUDA-first 的语言，如果 GPU 编程中固定维度的向量和矩阵是最基础的操作单元（如同 GLSL/HLSL 中的 `float4` 和 `float4x4`），则将这些类型纳入标准库或语言核心具有充分的生产率正当性。

### 2.2 Mojo 的 SIMD 类型设计

Mojo 语言由 Modular 公司开发，由 LLVM 和 Swift 的创始人 Chris Lattner 主导设计，其类型系统中的一个核心特征是 `SIMD[DType, Width]` 类型。这是一个参数化类型，同时接受数据类型（如 `DType.float32`）和 SIMD 宽度（如 4、8、16）两个参数。Mojo 的博客文章展示了 `SIMD` 类型如何同时服务于 CPU 向量化（SSE/AVX/NEON）和 GPU 编程场景。

值得注意的是，Mojo 在矩阵布局方面支持行主序（row-major）和列主序（column-major）两种模式，并提供不同的算法实现以适配每种布局的最优访问模式。在行主序模式下，访问同一列的连续元素需要使用 `simd_strided_load`（跨步加载）；在列主序模式下，同一列的元素在内存中连续排布，可以直接使用普通的连续加载指令。基准测试显示，当矩阵布局与 SIMD 加载模式匹配时，性能差距可达数倍。

Mojo 的设计给我们的启示是：如果语言核心提供向量/矩阵类型，应当考虑将 SIMD 宽度作为类型参数的一部分（类似 `f32x4` 这种形式），并在矩阵类型中明确编码布局信息作为类型级属性，使得编译器可以根据布局选择最优的代码生成策略。

### 2.3 Slang/HLSL 的向量矩阵内建类型

Slang 是由 NVIDIA 开发的着色语言，旨在为图形和计算工作负载提供统一的编程模型。Slang 将标量、向量（如 `float3`、`int4`）和矩阵（如 `float4x4`）作为语言的**内建类型（built-in types）**，而非库类型。这些类型拥有特殊的语法支持（如 `.xyz` swizzle、矩阵乘法运算符 `*`），并且编译器对其内存布局和运算语义有内建的深度理解。

Slang 社区的一个讨论帖（"Consider exposing a host-side library matching built-in Slang Vector/Matrix Math Intrinsics"）揭示了一个长期存在的痛点：Slang 的设备端有完善的向量矩阵内建函数，但主机端（C++）缺乏对应的官方数学库。开发者通常被迫选择第三方库如 `linalg`（sgorsten/linalg）或从 NVIDIA Falcor 框架中复制代码，但 `linalg` 采用列主序矩阵存储，而图形 API（Vulkan/DirectX）通常期望行主序的实例变换矩阵，这导致主机到设备传输时的微妙 bug——正交矩阵的转置等于其逆，因此错误在视觉上往往不易察觉。Falcor 的数学库则采用行主序以匹配图形 API 的期望。

这一讨论的结论是：主机端和设备端应**保持相同的向量解释和矩阵布局约定**，以避免在主机-设备边界处的隐式转置。Slang 文档进一步分析指出，最佳性能、一致性和跨平台独立性的组合方案是**行主序矩阵布局 + 行向量解释（row-major matrix layout + row vector interpretation）**。

### 2.4 Eigen/glam 的布局决策：行主/列主、对齐、SoA/AoS

Eigen 是 C++ 领域最广泛使用的线性代数库之一，其设计决策对 GPU 语言同样具有参考价值。Eigen 的 `Matrix<Scalar, Rows, Cols, Options, MaxRows, MaxCols>` 模板有六个参数，其中 `Options` 参数允许显式指定 `RowMajor` 或 `ColMajor`，**默认值为列主序（ColMajor）**。Eigen 的选择基于与 Fortran/LAPACK 传统的兼容性，因为 LAPACK 例程期望列主序输入。Eigen 还提供了 `AutoAlign` / `DontAlign` 选项控制内存对齐，SIMD 向量化要求数据按特定边界对齐（如 SSE 需要 16 字节对齐，AVX 需要 32 字节对齐）。

glam 作为 Rust 生态中面向游戏/图形学的库，同样选择了**列主序（column-major）和列向量（column vectors）**解释，即变换应用时矩阵在左、向量在右（`v' = Mv`）。glam 的设计哲学明确追求"无泛型公共 API"的简洁性，与 nalgebra 的泛型密集设计形成对比。glam 的 `Vec3A` 类型使用 SIMD 友好的 16 字节布局（padding 的 `(x, y, z, w)`），即使在仅使用三个分量的场景下也保证 SIMD 效率。

在 SoA（Structure of Arrays）vs AoS（Array of Structures）的维度上，GPU 编程的普遍共识是**批量处理场景下 SoA 显著优于 AoS**。SoA 布局下，同一属性的所有值在内存中连续排布，一个 warp（32 线程）的内存访问可以被硬件合并（coalesce）为 1-2 次事务；而 AoS 布局下，32 个线程访问 32 个分散的内存位置，需要 32 次独立事务，严重破坏并行效率。因此，对于粒子系统、顶点属性等 GPU 批量处理场景，SoA 是事实标准。

### 2.5 内建类型 vs 库类型的取舍证据

综合以上 precedents，我们可以建立一个决策框架：

| 维度 | 内建类型（如 Slang/HLSL） | 标准库类型（如 Rust glam） | 外部库类型（如 Rust ndarray） |
|------|------------------------|------------------------|--------------------------|
| 固定小维度（2/3/4） | 强烈推荐：GPU 着色器中最频繁使用的类型，运算符重载和 swizzle 需要语法级支持 | 可行：glam 证明库类型也能提供良好体验 | 不推荐：固定小维度不值得泛型开销 |
| 动态维度 ndarray | 不推荐：编译器不应过度膨胀 | 部分可行：标准库可提供基础 Buffer 和切片 | 推荐：ndarray/cupy 证明生态多样性有价值 |
| SIMD 友好性 | 最优：编译器直接映射到 GPU 寄存器 | 良好：通过 repr(simd) 和 align 实现 | 依赖实现：通常需要显式 SIMD 优化 |
| 跨语言互操作 | 需要明确定义 ABI | 通过 C ABI 导出可互操作 | 通常通过 C FFI 或 DLPack |
| 布局灵活性 | 可通过类型参数编码 | 通过泛型参数或不同类型实现 | 最灵活：运行时决定 |

对于目标语言（CUDA-first 系统语言），我们建议采用**分层策略**：固定维度 2/3/4 的向量和矩阵作为语言内建类型或标准库核心类型（享受运算符重载、swizzle、SIMD 指令直接映射），动态维度张量/数组作为标准库中的泛型容器（类似 `Buffer<T>` + `Tensor<T, D>`），更高级的线性代数运算通过外部 crate 提供。这一策略的 precedent 是 Slang 内建小维度类型 + 标准库提供通用 buffer 的组合模式。

## 3. Buffer / 内存抽象设计先例

### 3.1 四类资源抽象：Device / PinnedHost / Mapped / Managed

CUDA 编程模型中存在四种本质不同的内存资源类型，每种具有独特的分配语义、访问模式和性能特征：

- **DeviceBuffer**：仅 GPU 可访问的显存（通过 `cudaMalloc` 分配），CPU 不能直接解引用。这是 GPU 计算的主力内存类型。
- **PinnedHostBuffer**（Page-locked）：CPU 侧的页锁定内存（通过 `cudaMallocHost` 分配），可与设备内存之间进行高速异步传输（DMA），但过度使用会降低整个系统的物理内存可用性。
- **MappedHostBuffer**（Zero-copy）：映射到 GPU 地址空间的页锁定主机内存（通过 `cudaHostAlloc` with `cudaHostAllocMapped`），GPU 可以直接通过 PCI-e/NVLink 访问，无需显式拷贝，但访问速度低于本地显存。
- **ManagedBuffer**（Unified Memory）：通过 `cudaMallocManaged` 分配，由 CUDA 运行时自动处理主机-设备间的页面迁移，简化了编程模型但引入了隐式同步和迁移开销。

CuPy 的文档明确指出，其默认内存池管理两种内存：设备内存池（device memory pool）用于 GPU 显存分配，固定内存池（pinned memory pool）用于 CPU-GPU 数据传输期间的主机内存分配。CuPy 还提供了对 CUDA 11.2+ 流序内存分配器（MemoryAsyncPool）的实验性接口，允许内存按流顺序异步分配和释放。

### 3.2 RustaCUDA/cust：DeviceBuffer 与 DeviceCopy trait

RustaCUDA（及其演进版本 cust）是 Rust 生态中最成熟的 CUDA 绑定库。其内存模块定义了完整的类型层次：

- `DeviceBuffer<T>`：拥有所有权的设备内存缓冲区，要求元素类型 `T` 实现 `DeviceCopy` trait（表示类型安全地通过位拷贝在设备间传输）。提供 `from_slice`（从主机数据创建）、`as_device_ptr`（获取设备指针）、`from_raw_parts`/`drop`（不安全构造/析构）等方法。
- `UnifiedBuffer<T>`：统一内存缓冲区，同时可被 CPU 和 GPU 访问，同样要求 `T: DeviceCopy`。
- `LockedBuffer<T>` / `PinnedHostBuffer`：页锁定主机内存，用于优化的 H2D/D2H 传输。
- `DevicePointer<T>` / `UnifiedPointer<T>`：原始指针包装类型，保证内部表示与 `*mut T` 相同，可安全通过 FFI 边界传递给期望原始指针的 C 代码。

RustaCUDA 的一个重要设计决策是：`DeviceBuffer` 和 `UnifiedBuffer` 等高层类型**不保证 FFI 安全**，其内部表示在不同版本间可能变化。需要通过 FFI 边界时，必须显式转换为 `DevicePointer<T>` 和 `len()` 等原始组件，并使用 `mem::forget` 防止 Rust 侧自动释放内存。调用方负责后续通过 `from_raw_parts` 重建缓冲区并正确释放。这一设计体现了 Rust 的显式安全哲学。

### 3.3 CUDA.jl 的 CuArray 与 AbstractArray 接口

CUDA.jl 是 Julia 语言的 CUDA 编程接口，其 `CuArray` 类型是 GPU 数组抽象的典范设计。CuArray 实现了 Julia 的 `AbstractArray` 接口，这意味着所有作用于普通 `Array` 的高阶函数（如 `map`、`reduce`、`broadcast`）都可以无修改地在 CuArray 上工作。构造 `CuArray` 的方式与构造普通数组完全一致，数据通过构造函数或 `copyto!` 在主机和设备间传输。

CUDA.jl 的高阶抽象是其最强大的特性：用户可以通过 `map(sin, a)` 或 `reduce(+, a)` 等元素级/归约操作避免手写 CUDA kernel。这些高阶操作会自动融合（fusion），生成优化的 GPU kernel。值得注意的是，CUDA.jl 要求 `accumulate`、`scan` 等操作的运算符必须是**可结合的（associative）**，因为 GPU 并行执行可能改变非结合操作的求值顺序。

CUDA.jl 的设计启示是：标准库应提供与语言原生数组接口一致的 GPU buffer 类型，使得大量通用代码可以通过 trait/接口的多态在主机数组和 GPU 数组上统一工作。

### 3.4 CuPy 的内存池架构

CuPy 的内存管理架构是 Python 生态中最成熟的参考实现。其核心特征包括：

- **默认内存池**：CuPy 默认使用内存池分配设备内存和固定内存，显著降低重复分配/释放的开销。
- **可插拔分配器**：通过 `cupy.cuda.set_allocator()` 和 `cupy.cuda.set_pinned_memory_allocator()` 可以替换默认分配器。CuPy 提供了 `malloc_managed()`（统一内存）和 `malloc_async()`（流序异步分配）两种内置替代分配器。
- **MemoryAsyncPool**（CUDA 11.2+）：使用 CUDA 驱动内置的流序分配器，允许其他 CUDA 应用共享同一个内存池，实现真正的跨库异步内存管理。
- **统计与诊断**：内存池提供 `used_bytes()`、`total_bytes()`、`free_all_blocks()` 等监控接口，便于调试内存泄漏。

### 3.5 RAPIDS Buffer：流序访问与异构统一抽象

RAPIDS 生态系统（cuDF、cuML、cuGraph）中的 `Buffer` 类代表了一种更高级的内存抽象。一个 Buffer 实例要么持有设备内存（由 `rmm::device_buffer` 支持），要么持有主机内存（由 `HostBuffer` 支持），在构造时通过 `MemoryType`（DEVICE、PINNED_HOST、HOST）决定。Buffer 是**流序的（stream-ordered）**：所有读写操作必须在关联的 CUDA stream 上执行，或与该 stream 同步后才能访问。

RAPIDS Buffer 提供两种访问模式：`write_access(F&& f)` 接受一个可调用对象并在 stream 上执行；`exclusive_data_access()` 获取排他锁并返回原始指针，用于与非 stream-aware 的 API（如 MPI 或阻塞式主机 I/O）交互。Buffer 还支持 `rebind_stream()` 方法将缓冲区重新绑定到新的 stream（自动插入同步事件）。这一设计将 CUDA 的异步执行模型显式编码到类型系统中，是生产级 GPU 库内存管理的先进 precedent。

### 3.6 类型化 vs 字节缓冲、切片/视图所有权语义

在类型系统层面，Buffer 设计面临一个关键抉择：是提供类型化缓冲 `Buffer<T>` 还是无类型字节缓冲 `Buffer<u8>`？RustaCUDA 选择了强类型化路径（`DeviceBuffer<T: DeviceCopy>`），这确保了大量数据在类型层面就是安全的，不会将 `f32` 缓冲区误传给期望 `i32` 的 kernel。CuPy 则走了更 Pythonic 的路线：其底层存储是字节缓冲，类型信息存储在更高级别的 `ndarray` 包装器中。

对于 Rust 风格的静态类型语言，**类型化缓冲是更自然的选择**。切片和视图的所有权语义可以参照 Rust 的标准切片设计：`&[T]` 是共享只读切片，`&mut [T]` 是独占可变切片，这些概念可以自然地扩展到设备内存：`DeviceSlice<T>` 和 `DeviceSliceMut<T>`。RustaCUDA 的 `DeviceBuffer` 已经证明了这种模式在 CUDA 上下文中的可行性。

## 4. cuBLAS / cuDNN 绑定工程研究

### 4.1 句柄与 Descriptor 生命周期建模

cuBLAS 和 cuDNN 是 NVIDIA 的两个核心数学库，它们的 API 设计遵循相似的模式：使用 opaque handle（`cublasHandle_t`、`cudnnHandle_t`）管理库的内部状态，使用 descriptor 对象描述操作数的数据布局。

**cuBLAS Handle 的生命周期**：通过 `cublasCreate(&handle)` 创建，通过 `cublasDestroy(handle)` 销毁。Handle 内部维护流的关联（通过 `cublasSetStream` 设置）和 workspace 指针。一个关键发现来自 CuPy 的 issue #4676：cuBLAS 官方文档指出，多 stream 并行执行时，要么为每个 stream 提供独立的 workspace（通过 `cublasSetWorkspace`），要么为每个 stream 使用独立的 handle。如果多个 stream 共享同一个 handle 而不设置独立 workspace，会导致非确定性行为，因为 cuBLAS 会在内部共享默认 workspace pool。这意味着在面向多 stream 的绑定设计中，**handle-per-stream 是必需的**。

**cuDNN Descriptor 的生命周期**：cuDNN 使用多层 descriptor（`cudnnTensorDescriptor_t`、`cudnnFilterDescriptor_t`、`cudnnConvolutionDescriptor_t` 等）描述张量和操作。这些 descriptor 通常创建一次、多次复用，在层初始化阶段配置好维度、数据类型、内存布局等属性，在前向/反向传播中反复使用。

### 4.2 Workspace 管理策略

Workspace 是 cuBLAS 和 cuDNN 执行算法时需要的临时内存。Workspace 管理的最佳实践在生产级深度学习框架中有详细记录：

- **预分配策略**：Workspace 应在层初始化阶段一次性分配，在执行阶段反复复用，而非每次操作前临时分配。一个 LHCb 实验的 Allen 项目博客详细描述了这一策略：在 70 kHz 的事件处理速率下，重复的 `cudaMalloc` 调用会导致 GPU 堆碎片化和同步开销，因此他们在引擎线程级别预留连续的 workspace chunk，并按层的需求将子范围分配给各层。
- **算法选择与 workspace 大小的权衡**：cuDNN 的 `cudnnFindConvolutionForwardAlgorithm` 会返回多个候选算法及其 workspace 需求。例如 `IMPLICIT_PRECOMP_GEMM` 算法速度快但可能需要 20-50 MB workspace，而 `IMPLICIT_GEMM` 内存需求小但速度较慢。最佳实践是缓存每层的最优算法选择（基于输入形状），仅在输入形状显著变化时重新评估。
- **并发 workspace**：在并行推理配置中，每个活动 stream 需要独立的 workspace 缓冲区。可以使用固定数量的预分配 buffer（简单安全），也可以使用栈分配器从更大的 block 中切分（更节省内存但有锁开销）。

### 4.3 版本兼容与 ABI 稳定性现状

NVIDIA 库的版本兼容性可以从"四轴兼容矩阵"框架理解：**驱动版本、Toolkit 版本、框架构建版本、GPU 计算能力**四个维度各自独立变化，每个维度携带自己的 ABI 和功能约束。cuBLAS 的 SONAME（共享库主版本号）在同一主版本内保持 ABI 向后兼容，这意味着针对 cuBLAS 11.x 编译的应用可以在 cuBLAS 11.y（y >= x）上运行。然而，不同主版本之间（如 11.x 到 12.x）存在 ABI 断裂，需要重新编译。

cuBLAS/cuDNN 的 C API 相对来说是稳定的，但新版本中会持续增加新函数和新枚举值。绑定层应通过动态加载（`dlopen`/`dlsym` 或 Windows 的 `LoadLibrary`/`GetProcAddress`）来处理版本差异，对可选的新 API 进行运行时能力检测。

### 4.4 自动绑定生成 vs 手写

对于 cuBLAS/cuDNN 这类 C API 库，存在两种绑定策略：

- **自动生成绑定**：使用 bindgen 类工具从 CUDA 头文件自动生成 FFI 声明。这种方法适用于 API 表面大、版本更新频繁的场景。但 CUDA 头文件包含大量宏、条件编译和平台特定定义，纯自动化工具难以完美处理。
- **手写薄封装**：手工编写核心 API 的 FFI 声明，并在其之上提供符合语言习惯的安全封装。这种方法工作量大但质量更高，可以精心设计错误处理、生命周期和资源管理。

对于 MVP 阶段，推荐**手写关键 API 的薄封装**（覆盖 GEMM、GEMV 等核心 BLAS Level 2/3 操作），而非尝试自动生成完整绑定。这一选择的 precedent 包括 CuPy（手写 CUDA 绑定）和 RustaCUDA（手写 CUDA Driver/API 绑定）。

### 4.5 CUB/Thrust 模板库的绑定可行性

CUB 和 Thrust 是 NVIDIA 提供的 header-only C++ 模板库，专为 GPU 并行算法设计。由于它们是**模板库**而非 C API 库，直接通过 FFI 绑定在技术上几乎不可能：模板必须在编译时实例化，而 FFI 边界跨越的是编译后的二进制接口。

绑定 CUB/Thrust 的可行策略包括：

- **预实例化 C API 包装**：为常用的类型组合（如 `int`、`float`、`double` 的 reduce、sort、scan）编写 C++ 包装函数，显式实例化模板并导出 C 符号。这是 AMD 的 rocPRIM 库采用的策略。
- **自研等价物**：在目标语言中直接实现相同算法。对于 reduce、scan 等基础并行基元，使用 CUDA kernel 直接实现比依赖 Thrust 更轻量且可控。
- **编译时集成**：如果目标语言的编译器架构允许（如生成 CUDA C++ 代码），可以在编译输出中直接包含 CUB/Thrust 头文件。Slang 和 Julia 的 CUDA 编译器后端采用的就是这种方式。

对于 MVP 阶段，推荐**对必要的并行基元（reduce、scan、sort）进行自研 CUDA kernel 实现**，而非尝试绑定 Thrust/CUB。

## 5. C ABI 导出设计

### 5.1 Rust 的 extern "C" + repr(C) + cbindgen 工作流

Rust 的 C FFI 工作流是系统设计型语言导出 C ABI 的最成熟 precedent。其三个核心技术要素是：

- `extern "C"`：声明函数使用 C 调用约定，确保参数传递顺序、栈清理方式与 C 编译器一致。
- `#[no_mangle]`：禁止编译器对函数名进行 mangling，确保 C 代码可以通过原始函数名链接。
- `#[repr(C)]`：强制结构体/联合体使用与 C 兼容的内存布局，包括字段顺序和对齐方式。

`cbindgen` 是一个从 Rust 代码自动生成 C/C++ 头文件的工具。它扫描 crate 中的 `#[no_mangle] pub extern "C" fn`、`pub const` 和 `#[repr(C)]` 结构体，为它们生成对应的 C 声明。cbindgen 支持 `repr(u8)` 等枚举布局属性、`repr(transparent)` 单字段结构体，以及条件编译和模块系统的处理。与 safer-ffi 相比，cbindgen 不介入 FFI 接口的实现过程，它假定开发者已经正确使用了 `extern "C"` 和 `#[repr(C)]`，仅负责头文件生成，因此更成熟稳定（safer-ffi 仍处于 alpha 阶段）。

VibeTensor 项目（一个完全由 AI 代理生成的深度学习运行时）也采用了稳定的 C plugin ABI 设计，其 C++ 核心暴露 C ABI 接口供动态加载的 operator 插件使用，证明了这种模式在现代 AI 基础设施中的持续有效性。

### 5.2 稳定 ABI 的边界设计：句柄式 API

当通过 C ABI 暴露复杂类型（如 GPU buffer、矩阵、计算图）时，**不透明句柄（opaque handle）**模式是确保 ABI 稳定性的最佳实践。具体设计为：

- 所有复杂类型在 C API 中表示为指向不透明结构体的指针：`typedef struct MyBuffer* my_buffer_t;`
- 创建函数：`my_buffer_t my_buffer_create(size_t size, my_status_t* status);`
- 销毁函数：`void my_buffer_destroy(my_buffer_t buffer);`
- 操作函数接收句柄作为首参数：`void my_buffer_copy(my_buffer_t dst, my_buffer_t src, my_status_t* status);`

这种模式的优势在于：C 侧完全不知道结构的内部布局，允许 Rust（或目标语言）侧在不破坏 ABI 的前提下自由改变内部实现。NVIDIA 的 cuBLAS handle、cuDNN handle、CUDA context 等全部采用相同的模式，是工业界最广泛的 precedent。

### 5.3 错误码跨 FFI 约定

跨 FFI 的错误处理有两种主要模式：

| 模式 | 实现 | 优点 | 缺点 | 典型先例 |
|------|------|------|------|---------|
| 输出参数错误码 | `my_status_t* status` 作为最后一个参数 | C 兼容性好，无需异常机制 | 调用方需显式检查 | cuBLAS（`cublasStatus_t` 返回值）、CUDA Driver API |
| 结构体返回值 | 返回包含结果+错误码的结构体 | 原子性获取结果和状态 | C 中需要解构 | Rust `Result` 的 C 表示 |
| 线程局部错误变量 | `errno` 风格 | 不污染函数签名 | 线程安全问题，不易组合 | C 标准库 errno |
| 回调式错误处理 | 注册错误回调函数 | 可集中处理 | 控制流不直观 | 某些图形 API debug callback |

推荐采用**返回值错误码模式**（与 cuBLAS/cuDNN 一致），每个 C API 函数返回 `my_status_t` 枚举，成功时为 `MY_SUCCESS`，失败时携带具体错误类型。对于需要返回值的函数，使用输出参数指针：`my_status_t my_buffer_get_size(my_buffer_t buf, size_t* out_size);`

## 6. Python 扩展生成研究

### 6.1 PyO3/maturin vs nanobind vs 手写 CPython API

将 Rust/C++ 代码暴露给 Python 有三种主流技术路径，每种在 CUDA 扩展场景中有不同的适用性：

**PyO3 + maturin**：PyO3 是 Rust 的 Python 绑定库，maturin 是对应的构建工具。PyO3 提供了安全的 Rust API 来操作 Python 对象、处理 GIL（全局解释器锁）和转换类型。PyO3-dlpack crate 提供了通过 DLPack 协议与 PyTorch/JAX/CuPy 零拷贝交换张量的能力，支持 CPU、CUDA、ROCm 等多种设备。PyO3 的劣势在于编译产出物较大，且绑定代码的编译时间随项目规模增长。

**nanobind**：由 pybind11 的作者 Wenzel Jakob 开发，定位为更轻量、更快速的替代品。根据官方基准测试，nanobind 相比 pybind11 可实现 **~4 倍更快的编译时间、~5 倍更小的二进制文件、~10 倍更低的运行时开销**。nanobind 支持 DLPack 和 buffer protocol，可实现与 NumPy、PyTorch、TensorFlow、JAX 等框架的零拷贝 CPU/GPU 数组交换。VibeTensor 项目（NVIDIA 研究团队发布的由 AI 代理完全生成的深度学习运行时）选择了 nanobind 作为其 Python 前端绑定工具，证明了 nanobind 在 CUDA 扩展场景中的生产就绪性。

**手写 CPython C API**：直接使用 Python 的 C API 编写扩展模块。这种方式控制权最大、依赖最少，但开发效率极低，需要手动处理引用计数、类型转换和错误状态。对于 MVP 阶段不推荐，但可作为最终 fallback。

**推荐选择：nanobind + scikit-build-core**。理由：(1) nanobind 的编译速度和二进制体积优势在需要链接 CUDA 运行时的场景中尤为重要；(2) scikit-build-core 提供了对 CMake/CUDA 的原生支持；(3) DLPack 支持使得与 PyTorch/CuPy 的互操作 trivial。Precedent：VibeTensor（NVIDIA 研究项目）、pytorch-nanobind-cuda-example（社区最小示例）。

### 6.2 Wheel 打包含 CUDA 依赖的策略

在 Windows 平台上分发包含 CUDA 依赖的 Python wheel 是最具挑战性的环节。PyTorch 和 CuPy 的实践提供了关键 precedents：

**PyTorch 的 Windows DLL 策略**：PyTorch wheel 将所需的 CUDA 运行时 DLL（如 `cudart64_13.dll`、`cublas64_13.dll`、`cublasLt64_13.dll`）直接打包在 `torch/lib/` 目录下。在 `torch/__init__.py` 中，通过 `_load_dll_libraries()` 函数使用 `ctypes.CDLL(dll)` 循环加载这些 DLL。这一机制的核心挑战在于 Windows 的 DLL 搜索路径规则：Python 3.8+ 需要使用 `os.add_dll_directory()` 显式添加搜索路径，而非依赖 `PATH` 环境变量。当 torch wheel 中的 CUDA DLL 版本与系统 CUDA_PATH 指向的版本不一致时（例如 wheel 包含 CUDA 13 而系统安装 CUDA 12），必须确保 torch/lib 目录在搜索路径中优先于系统路径，否则会出现 DLL 加载失败或版本不匹配。

一个典型的 bug 场景是：某工具在启动子进程时只添加了 `CUDA_PATH\bin` 到 PATH，但未添加 `torch\lib`，导致子进程中的 CUDA DLL 加载了错误版本，最终 GPU 后端无法正确初始化。修复方案是显式检查并添加 `<site-packages>/torch/lib` 到 DLL 搜索路径。

此外，Windows 上的 OpenMP 依赖也是一个常见陷阱。PyTorch 的 `fbgemm.dll` 依赖于 `libomp140.x86_64.dll`（来自 Visual Studio 的 OpenMP 运行时），如果目标系统缺少此 DLL，会导致 `OSError: [WinError 126]`。这要求 wheel 打包时要么静态链接 OpenMP，要么将 `libomp140.x86_64.dll` 一并打包。

**CuPy 的 Windows DLL 加载策略**：CuPy 在其 `_environment.py` 中实现了复杂的 DLL 发现逻辑：首先检查是否为 wheel 安装（查找 `cupy/.data/lib`），然后依次尝试 `CUDA_PATH`、conda 环境的 Library/bin 目录、以及系统路径。CuPy 使用 `os.add_dll_directory()` 在 Windows 上注册搜索路径，并通过调试日志（`CUPY_DEBUG_LIBRARY_LOAD`）输出每一步的 DLL 查找过程，这对于诊断加载失败至关重要。

对于新语言的 Python 绑定，Windows DLL 分发策略应遵循以下原则：

1. 在 wheel 中捆绑所需的 CUDA 运行时 DLL（`cudart64_XX.dll` 和 `cublas64_XX.dll`），将其放在包内的 `lib/` 目录中。
2. 在包的 `__init__.py` 中，使用 `os.add_dll_directory()` 注册 `<package>/lib` 目录到 DLL 搜索路径（Python 3.8+）。
3. 对于 OpenMP 依赖，使用 delvewheel 工具将 `libomp140.x86_64.dll` 等依赖打包进 wheel 并修复路径。
4. 提供调试模式（环境变量控制），输出 DLL 加载的详细日志。

### 6.3 __cuda_array_interface__ 与 DLPack 零拷贝协议

实现与 Python GPU 生态的零拷贝互操作需要支持两个关键协议：

**__cuda_array_interface__**：由 Numba 项目发起、被 CuPy/PyTorch/PyCUDA/JAX 等广泛支持的 CUDA 数组交换协议。实现该协议的对象需要暴露一个字典，包含以下必需字段：`data`（设备内存指针和只读标志）、`shape`、`typestr`（NumPy 风格类型描述符）、`version`（通常为 3）。可选字段包括 `strides`、`descr`、`mask`、`stream` 等。协议的关键语义是：**对象生命周期由用户管理**，消费者在使用期间必须确保导出对象未被销毁。CuPy 的文档提供了 CuPy↔PyTorch 零拷贝交换的完整示例：通过检查 `a.__cuda_array_interface__['data'][0] == b.__cuda_array_interface__['data'][0]` 验证底层指针相同。

**DLPack**：一个跨框架的、语言无关的张量数据结构规范。DLPack 定义了 `DLTensor` 结构体（包含数据指针、设备类型/ID、数据类型、维度、形状、步幅）和 `DLManagedTensor`（增加删除回调函数用于生命周期管理）。DLPack 通过 Python 的 `PyCapsule` 机制在框架间传递：导出方将 `DLManagedTensor` 指针封装为 capsule，导入方提取指针并消费。DLPack 相比 `__cuda_array_interface__` 的优势在于：(1) 有显式的生命周期管理（删除回调）；(2) 支持更多设备类型（CPU、CUDA、ROCm、Metal、Vulkan）；(3) 被纳入 Python array API 标准。

对于新语言的标准库 Python 绑定，应**同时实现这两个协议**：`__cuda_array_interface__` 用于与现有 Numba/CuPy 生态的最大兼容性，DLPack 用于与 PyTorch 2.x+/JAX 的现代互操作。nanobind 内置了对两种协议的支持，pyo3-dlpack crate 为 PyO3 提供了 DLPack 的 ergonomic 封装。

## 7. 设备端数学函数来源

### 7.1 libdevice 链接机制与 NVVMReflect

NVIDIA 随 CUDA Toolkit 提供了一个名为 **libdevice** 的 LLVM bitcode 库，位于 `nvvm/libdevice/` 目录下。libdevice 实现了大量常见的数学函数（sin、cos、exp、log、pow 等），可作为任何使用 LLVM NVPTX 后端的编译器的高性能数学库。每个 GPU 计算架构有独立的 libdevice 版本（如 `libdevice.compute_70.10.bc`）。

libdevice 的一个重要特性是其与 **NVVMReflect** 的集成。NVVMReflect 是一个 LLVM IR pass，它处理 libdevice 代码中的条件编译。具体机制是：libdevice 中的数学函数内部调用 `@__nvvm_reflect("FASTMATH")`，NVVMReflect pass 在执行早期将这些调用替换为常量（0 或 1），基于编译器传入的反射参数。这实现了同一份 bitcode 在不同编译配置下的不同行为：

```
float my_function(float a) {
  if (__nvvm_reflect("FASTMATH"))
    return my_function_fast(a);    // 低精度快速实现
  else
    return my_function_precise(a); // IEEE-754 精确实现
}
```

推荐的编译流程是：(1) 保存模块的外部函数列表；(2) 将用户模块与 libdevice bitcode 链接；(3) 将非外部函数 internalize；(4) 删除未使用的内部函数；(5) 运行 NVVMReflect pass；(6) 运行标准优化流水线。NVVMReflect 应在优化流水线早期执行（链接后立即），以便后续的 dead code elimination 消除不可达分支。

### 7.2 Fast Math 精度模式的 API 暴露先例

Fast math 是 GPU 编程中常见的编译选项（`--use_fast_math`），它启用一系列可能牺牲精度换取性能的优化：将除法转换为乘以倒数、使用近似内在函数、允许非规格化数刷新为零等。在 API 设计中，fast math 模式的暴露有多种 precedents：

- **glam 的 fast-math feature gate**：glam 提供了 `fast-math` 特性，默认关闭以确保所有平台上逐位一致的结果。文档明确警告"中间库不应使用此特性，应将决定推迟给最终二进制构建"。这是一种保守但安全的策略。
- **CUDA 编译器标志**：在 CUDA C++ 中，fast math 通过 `nvcc --use_fast_math` 全局启用，粒度是整个编译单元。用户无法对单个函数或操作选择精度模式。
- **LLVM NVVMReflect**：通过编译器选项控制 libdevice 的精度模式，允许更细粒度的控制。

对于新语言，建议在标准库中提供**两级精度控制**：(1) 编译器级别的 `--fast-math` 标志，通过 NVVMReflect 影响 libdevice 链接；(2) 标准库级别的显式函数变体（如 `sin` vs `sin_fast`），允许用户在代码中对关键路径选择精度模式。这与 Rust 的 `f32::sqrt`（硬件指令，快速）和 `f32::powf`（软件实现，精确）的区分一致。

## 8. 标准库类型层次草案

### 8.1 核心 trait 层次

基于上述调研，建议标准库定义以下核心 trait 层次：

```rust
// 所有可安全按位拷贝到 GPU 的类型
pub trait DeviceCopy: Copy + 'static {}

// 内存资源核心 trait
pub trait Buffer<T: DeviceCopy> {
    fn len(&self) -> usize;
    fn is_empty(&self) -> bool;
    fn device_ptr(&self) -> DevicePointer<T>;
    fn memory_type(&self) -> MemoryType;
}

// 流序操作支持
pub trait StreamOrdered {
    fn stream(&self) -> &Stream;
    fn on_stream(&self, stream: &Stream) -> &Self;
}

// 主机-设备传输
pub trait HostDeviceCopy<T: DeviceCopy> {
    fn copy_from_host(&mut self, src: &[T]) -> Result<()>;
    fn copy_to_host(&self, dst: &mut [T]) -> Result<()>;
}

// 统一内存（主机和 GPU 均可直接访问）
pub trait UnifiedAccess<T> {
    fn as_host_slice(&self) -> &[T];
    fn as_host_slice_mut(&mut self) -> &mut [T];
}
```

### 8.2 Buffer 类型族设计

```rust
// 设备显存（仅 GPU 可访问）
pub struct DeviceBuffer<T: DeviceCopy> { /* opaque */ }

// 页锁定主机内存（优化 H2D/D2H 传输）
pub struct PinnedHostBuffer<T: DeviceCopy> { /* opaque */ }

// 映射主机内存（GPU 可通过 PCI-e 直接访问）
pub struct MappedHostBuffer<T: DeviceCopy> { /* opaque */ }

// 统一内存（自动页面迁移）
pub struct ManagedBuffer<T: DeviceCopy> { /* opaque */ }

// 设备切片（类似 &[T]，但指向设备内存）
pub struct DeviceSlice<T: DeviceCopy> { /* opaque */ }
pub struct DeviceSliceMut<T: DeviceCopy> { /* opaque */ }
```

### 8.3 Vector/Matrix 类型设计

```rust
// 固定维度向量（列向量解释）
pub struct Vec2<T: DeviceCopy> { pub x: T, pub y: T }
pub struct Vec3<T: DeviceCopy> { pub x: T, pub y: T, pub z: T }
pub struct Vec4<T: DeviceCopy> { pub x: T, pub y: T, pub z: T, pub w: T }

// SIMD 友好别名
type f32x2 = Vec2<f32>;
type f32x4 = Vec4<f32>;
type i32x4 = Vec4<i32>;

// 列主序矩阵（默认）
// Mat4 存储为 4 个列向量
pub struct Mat2<T: DeviceCopy> { pub cols: [Vec2<T>; 2] }
pub struct Mat3<T: DeviceCopy> { pub cols: [Vec3<T>; 3] }
pub struct Mat4<T: DeviceCopy> { pub cols: [Vec4<T>; 4] }

// 行主序变体（显式类型名）
pub struct Mat4RowMajor<T: DeviceCopy> { pub rows: [Vec4<T>; 4] }
```

### 8.4 每个决策的先例依据汇总

| 决策 | 选择 | 先例依据 |
|------|------|---------|
| 固定维度向量/矩阵 | 语言核心/标准库内置 | Slang/HLSL/GLSL 的内建类型；GPU 编程中 2/3/4 维度是最基础操作单元 |
| 动态维度 ndarray | 标准库泛型容器，非核心内置 | Rust 核心库不包含 matrix；ndarray/cupy 证明生态多样性有价值 |
| 矩阵默认布局 | 列主序（column-major） | Eigen 默认 ColMajor；glam 默认；cuBLAS 接口惯例；与 Fortran/LAPACK 兼容 |
| 向量解释 | 列向量（column vector） | glam 列向量；Eigen 列向量；OpenGL 传统；v' = Mv 形式 |
| Buffer 类型化 | `Buffer<T: DeviceCopy>` 泛型 | RustaCUDA `DeviceBuffer<T: DeviceCopy>`；类型安全防止误用 |
| 流序语义 | Buffer 关联 Stream，操作异步 | RAPIDS Buffer 流序设计；CUDA 11.2+ MemoryAsyncPool |
| SIMD 友好对齐 | Vec4 保证 16 字节对齐 | glam Vec3A（16 字节对齐的 Vec3）；SSE/AVX 对齐要求 |
| SoA/AoS | 标准库主要提供 SoA 布局，AoS 用于小结构 | GPU coalesced access 要求 SoA；仅固定小维度（2/3/4）用 AoS |

## 9. cuBLAS 绑定推荐工程方案

### 9.1 整体架构

cuBLAS 绑定采用三层架构：

1. **底层 FFI 层（`sys`）**：手写核心 cuBLAS C API 的 `extern "C"` 声明，覆盖 handle 管理、stream 设置、workspace 配置、GEMM/GEMV 等 Level 2/3 操作。该层使用 `repr(C)` 结构体映射 cuBLAS 的数据类型（如 `cublasOperation_t`），并保持 API 签名与 C 头文件一致以便对照验证。
2. **安全封装层（`safe`）**：为每个 opaque handle 提供 RAII 包装（`CudaBlasHandle`），实现 `Send`/`Sync` trait（如果底层线程安全），提供符合语言习惯的泛型 API（如 `gemm<T: GemmSupported>(alpha: T, a: &DeviceSlice<T>, b: &DeviceSlice<T>, beta: T, c: &mut DeviceSliceMut<T>)`）。
3. **高层 API 层（`api`）**：提供更 ergonomic 的矩阵乘法接口（如运算符重载、方法链式调用），类似 `c.matmul(a, b)` 的形式。

### 9.2 生命周期建模示意

```
cuBLAS Handle 生命周期管理
───────────────────────────

struct CudaBlasHandle {
    raw: cublasHandle_t,
    workspace: DeviceBuffer<u8>,
    stream: Stream,  // 绑定的 stream
}

impl CudaBlasHandle {
    // 创建 handle 并预分配 workspace
    fn new(stream: &Stream, workspace_size: usize) -> Result<Self> {
        let mut raw = ptr::null_mut();
        unsafe { cublasCreate(&mut raw)?; }
        let workspace = DeviceBuffer::new(workspace_size)?;
        unsafe {
            cublasSetWorkspace(raw, workspace.as_ptr(), workspace_size);
            cublasSetStream(raw, stream.as_raw());
        }
        Ok(Self { raw, workspace, stream: stream.clone() })
    }
}

impl Drop for CudaBlasHandle {
    fn drop(&mut self) {
        unsafe { cublasDestroy(self.raw); }
        // workspace 通过 DeviceBuffer 的 Drop 自动释放
    }
}
```

**Handle Pool 设计**：对于多 stream 场景，维护一个 handle pool。每次 GEMM 调用前从 pool 获取与目标 stream 匹配的 handle（如果不存在则创建），调用后返回 pool。这避免了频繁的 create/destroy 开销，同时保证每个 stream 有独立的 handle 和 workspace。

### 9.3 Workspace 管理实现

```
Workspace 预分配与复用
─────────────────────

struct WorkspacePool {
    // 按大小分级缓存
    buckets: HashMap<usize, Vec<DeviceBuffer<u8>>>,
    max_size: usize,
}

impl WorkspacePool {
    fn acquire(&mut self, size: usize, stream: &Stream) -> DeviceBuffer<u8> {
        let size = size.next_power_of_two();
        if let Some(buf) = self.buckets.get_mut(&size).and_then(|v| v.pop()) {
            return buf;
        }
        DeviceBuffer::new(size).expect("workspace alloc failed")
    }

    fn release(&mut self, buf: DeviceBuffer<u8>) {
        let size = buf.capacity();
        self.buckets.entry(size).or_default().push(buf);
    }
}

// 在 handle pool 的 get_handle 中：
fn get_handle(&mut self, stream: &Stream, workspace_size: usize) -> CudaBlasHandle {
    let ws = self.workspace_pool.acquire(workspace_size, stream);
    // ... 创建 handle，绑定 workspace 和 stream
}
```

## 10. Python 互操作分期方案

### 10.1 MVP 阶段（只做这些）

MVP 阶段的 Python 互操作目标是让新语言的 GPU buffer 能被 PyTorch 和 CuPy 零拷贝消费，反之亦然。具体交付物：

1. **nanobind + scikit-build-core 构建的 Python 扩展模块**，暴露标准库的核心 Buffer 类型。
2. **`__cuda_array_interface__` 协议支持**：Buffer 对象导出该属性字典，包含 `data`（设备指针）、`shape`、`typestr`、`version`(3)、`stream`（可选）。
3. **DLPack 导出**：`buffer.__dlpack__()` 方法返回 `PyCapsule` 包装的 `DLManagedTensor`。这是与 PyTorch 2.x 互操作的现代标准方式。
4. **DLPack 导入**：`Buffer::from_dlpack(cap: &PyCapsule) -> Buffer<T>` 构造函数，消费其他框架导出的 DLPack capsule。
5. **Windows DLL 加载**：Python 包的 `__init__.py` 中使用 `os.add_dll_directory()` 注册 bundled CUDA DLL 路径。

**明确不在 MVP 阶段做的**：完整的 NumPy 风格 ndarray API、自动梯度（autograd）、Python 层 kernel 编译/启动、多 GPU 分布式支持。

### 10.2 Phase 2 扩展

- **Buffer protocol** 支持（与 NumPy 的零拷贝 CPU 数组交换）。
- **Python 上下文管理器**：自动释放临时 buffer（`with buffer.stream_scope()`）。
- **类型存根（type stubs）**：通过 nanobind 的 stub 生成为 Python 侧提供类型提示。

### 10.3 Phase 3 完整支持

- **Python 装饰器式 kernel 编译**：类似 Numba CUDA 的 `@cuda.jit` 体验（如果语言编译器支持 JIT）。
- **与 PyTorch 的内存池共享**：参考 pytorch-pfn-extras 的 `use_torch_mempool_in_cupy()` 模式。
- **多流和 CUDA Graph 捕获**的 Python API。
