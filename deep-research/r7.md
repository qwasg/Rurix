# 新编程语言的治理骨架与防 AI 幻觉工程

## 结论摘要

对一个只有 4–6 人、却要重度使用 AI agent 的新语言项目，最值得借鉴的不是 Rust 的“全量组织规模”，而是它那套把“语义承诺”锁进流程的硬约束：重大语义变更先过提案流程；实现先放在 feature gate 后面；每个不稳定特性都挂 tracking issue；稳定化前必须补齐规范、测试与稳定化报告；真正的兼容性破坏尽量打包进 edition。Rust 的 Leadership Council 负责跨团队协调和授权，但大量决策仍委托给各团队，这说明“中央治理”可以很薄，而“规范—实现—测试—发布”的闭环必须很硬。citeturn37view0turn37view2turn39search12turn40search0turn40search1turn40search6turn37view3turn38search0turn38search12

如果把你的核心风险定义为“语义漂移”，那么最有效的工程反制不是更多口头评审，而是把**规范写成可测试文本**，再把**测试变成唯一的验收边界**。Ferrocene Language Specification 展示了一个成熟方法：把条文拆成 Syntax、Legality Rules、Dynamic Semantics、Undefined Behavior、Implementation Requirements 等段落；给规范条款编号；再用测试注释和 traceability matrix 把测试追溯回规范条款。WebAssembly 则把规范、参考实现和官方测试套件放在同一治理闭环里。对小团队而言，这比追求“先有完美大部头规范”更现实。citeturn15search0turn14search7turn14search17turn18search1turn18search7

进入 AI 时代后，新增机制也已经有真实项目先例。Linux kernel 已发布两份官方文档：一份要求 AI 工具遵循正常内核开发流程、禁止 AI 代替人类签署 DCO，并建议为 AI 贡献加 `Assisted-by` 标记；另一份要求对“工具生成内容”做来源透明披露，包括用过什么工具、提示词摘要、哪些片段受工具影响、如何测试。LLVM 的正式政策则要求“human in the loop”、作者对所有 AI 生成内容负责、实质性 AI 内容要标注来源，而且明确禁止无人类审批的 agent 在 LLVM 的数字空间自动发言或发 PR。也就是说，**人类负责、可追溯来源、额外审查、不得把维护者当成免费验证器**，已经是大型项目正在成形的共同底线。citeturn29view1turn29view0turn30view0

基于这些先例，我建议你的项目采用一套“薄治理、厚约束”的骨架：组织上只保留 3 人治理小组；流程上只保留三档变更门；仓库上把 `spec/`、`rfcs/`、`conformance/`、`ui/`、`unsafe-audit/` 作为一等公民；AI 上则强制要求 provenance、强制加载仓库规则文件、强制跑验证命令、强制把规范条款号写进 PR。这样做既保留了 Rust 的抗漂移能力，又把组织和文档负担裁剪到小团队可以长期执行的程度。citeturn37view0turn40search0turn29view0turn30view0turn32view0turn32view1turn32view3turn35view0turn35view1turn35view2

## Rust 的可借鉴内核

Rust 当前的顶层治理不是单一“核心团队拍板”，而是由 Leadership Council 统筹项目整体成功、识别无人负责的工作、创建或调整团队、确保顶层团队问责；Council 由各 top-level team 的代表组成，并把大量权力继续委托给团队自治。对小团队来说，这说明你不必复制 Rust 的组织数量，但可以复制它的**治理分层原则**：顶层只管方向、归属和卡点升级，具体语义与实现仍由对应责任域拍板。citeturn37view0

Rust 的 RFC 流程本质上是一条“先设计、后实现、再稳定”的长链条。RFC Book 规定，凡是“substantial” 的语言或标准库变化，必须先把 RFC 合入 `rust-lang/rfcs`，RFC 合并后才进入“active”状态，可以开始实现；而每个被接受的 RFC 又会在主仓库里配套一个 tracking issue 来追踪实施进度。实践上，在正式 RFC 之前，Rust 团队通常还会先做 pre-RFC 式讨论，例如先在 Zulip、Internals 或团队仓库里征求意见，再决定是否进入正式提案。citeturn37view2turn39search12turn37view1

Rust 的“最后决断”机制不是简单投票，而是 FCP。以编译器团队文档为例，FCP 需要由团队成员发起，并由对应 reviewer 子团队完成 sign-off，随后进入十天等待期，以便让异议浮现。对于编译器层面的提案，Rust 还区分 RFC、MCP、PR 三种重量级别：RFC 用于重大变化，MCP 用于多数非面向终端用户的中等变化，PR 则覆盖较轻量、可直接伴随补丁的提案。这个分层非常适合小团队裁剪：你完全可以保留“RFC / mini-RFC / direct PR”三档，而不需要复制 Rust 全部的团队专用流程。citeturn37view1

Rust 的 feature gate 和稳定化要求，是防漂移最值得照抄的部分。`rustc-dev-guide` 明确要求：每个 feature gate 都要有 tracking issue；不稳定语言特性在 nightly 上通过显式 `#![feature(...)]` 光学隔离；实现时要在 `unstable.rs` 登记 gate；稳定后再移到 `accepted.rs`；稳定化前必须有完整的 Reference PR，而且要经过 `lang-docs` 审阅批准；同时还要提交 stabilization report，说明行为、与 RFC 的偏差、受影响的 edition、关键测试等，最后再通过 lang team 的 FCP。换句话说，**“实现可运行”远远不等于“语义已承诺”**。citeturn40search1turn40search2turn40search6turn40search0turn40search4

Rust 的 edition 机制提供了一个很好的“破坏性变更缓冲层”。Edition Guide 明确说，edition 用来承载那些否则无法保持向后兼容的语言变化；迁移依赖 `cargo fix --edition`、测试和手工收尾；Edition Releases 文档则显示，Leadership Council 会为 edition 组建项目组，推荐 2–3 名负责人，并在正式发布前大约一年的时间里推进一系列明确截止期。`rustc-dev-guide` 还说明，实验特性可以同时受 feature gate 与 edition 条件控制，等被确认纳入某个 edition 后，再逐步切换成 edition 判定。对于你的新语言，这意味着“把破坏性语义修复绑进年度 edition 窗口”，会比零散放出更可控。citeturn38search0turn38search12turn37view3turn40search5

## 规格工程、测试治理与 unsafe 治理

Ferrocene Language Specification 最有价值的地方，不是“它已经完美定义了 Rust”，而是它展示了**怎样把语言规范写成可审计、可追踪、可验证的条文**。FLS 在开篇就把规范分成 Syntax、Legality Rules、Dynamic Semantics、Undefined Behavior、Implementation Requirements，并且显式区分 normative 与 informative 内容。它还有条款编号、术语表、未定义行为附录，以及与测试关联的 traceability matrix。Ferrocene 的资格认证文档还说明，测试目录和单个测试都可以带“ferrocene annotation”标签，把测试追踪到 FLS 章节。对你的语言而言，这几乎就是现成模板：别写“导读型文档”，要写“能驱动测试、能引用条款号、能标注 UB”的规范。citeturn15search0turn14search7turn14search17

但也要看清 FLS 的边界。FLS 明说它“更倾向于记录 rustc 的行为”，并非为多实现之间的 conformance 而写，也不是语言演化讨论场。对一个从零开始的新语言，你应该反过来做：让 `spec/` 成为**项目唯一的规范源**，实现服从规范，而不是规范事后追认实现。否则 AI 最擅长的“从已有代码归纳合理语义”，会把偶然实现细节默默升级为规范承诺。citeturn15search0

在 conformance 组织上，WebAssembly 的结构比 Rust 更接近你需要的模型。官方 `WebAssembly/spec` 仓库把规范、参考实现和官方测试套件放在一起；而所有显著语义变化又先在 design 仓库讨论，再回流到 spec 仓库。这种结构的优点是：规范、测试、参考行为不会散落在多个“解释层”里。Rust 的 Crater 则补上了另一层：在潜在破坏性变化或发布前，对生态进行大规模构建与测试比对，以发现回归和兼容性影响。对小团队来说，前者应当成为日常机制，后者则适合作为里程碑机制。citeturn18search1turn18search7turn17search0turn17search1turn17search20

测试治理上，Rust 的 `compiletest` 与 `tests/ui` 给了一个很实用的模式：UI tests 主要验证编译器终端输出，也可按指令运行程序验证行为，预期输出保存在相邻的 `.stderr`/`.stdout` 文件里，并支持 `--bless` 更新快照。对语言项目而言，UI/golden 测试的最大价值，是把“错误信息、span、建议修复、告警等级”的回归显式化，防止 AI 在“只要能编过就行”的心态下悄悄放宽验收标准；它的代价则是快照更新频繁，文案微调也会造成噪声。citeturn19search0turn19search1turn19search2turn19search4

行为 conformance 测试则是语义主战场。它应该直接引用规范条款号，覆盖 parse/typecheck/runtime/FFI/diagnostics 四类观察点，并把“未定义行为”“实现定义行为”“保留未来扩展”的边界分别编码。只有这样，AI 生成实现时才有可能被“反向约束”：它不是去猜作者想要什么，而是必须满足一组已经编号的、可执行的要求。GitHub 的 Copilot 文档也把“先跑自动化测试和静态分析”列为审查 AI 生成代码的第一步；OpenAI 自己多个开源仓库的 `AGENTS.md` 还把 `$code-change-verification`、变化集验证和“改变用户可见行为前先做 implementation strategy”写成强制规则。这说明“把验证流程前置并显式写入 agent 上下文”，已经不是理论，而是现成可用的实践。citeturn33search3turn35view1turn35view2turn35view0

模糊测试和差分测试不适合做日常唯一防线，但很适合做“守住实现正确性底线”的第二道墙。Csmith 的核心思想是生成**无 UB 的随机 C 程序**，用差分测试找编译器 bug；2024 年的 Rustlantis 则通过生成 MIR 来对 rustc 做随机差分测试，并发现了新的正确性问题。对你这样的新语言，小团队最现实的做法不是一开始写完整 Csmith 级生成器，而是先做两个轻量版本：一是语法/类型层的 grammar-based 生成器；二是少量 IR/字节码层的差分测试。它们不需要每个 PR 都跑，但应当在周级或里程碑级执行。citeturn20search1turn20search16

unsafe 治理方面，Rust 的经验也很明确。Unsafe Code Guidelines 这个项目本身目前“基本处于停止状态”，但其 glossary 仍活跃维护，这提醒我们：**unsafe 语义一定要谨慎承诺**，不要在内存模型未清楚前写下过于武断的规范。与此同时，Clippy 已经沉淀出三条非常实用的可执行约束：`missing_safety_doc` 要求 public unsafe function 必须写 `# Safety` 段；`undocumented_unsafe_blocks` 要求 `unsafe` block/impl 前必须有 `// SAFETY:` 注释；`multiple_unsafe_ops_per_block` 进一步鼓励把 unsafe 缩到最小作用域。再配合 `cargo-geiger` 对 crate 与依赖中的 unsafe 使用做盘点，以及 Miri 对未定义行为进行运行期检测，你就有了一套“成本不高但极其抗漂移”的 unsafe 最小治理。citeturn21search0turn24view0turn24view1turn22search1turn21search2turn21search3

## AI 时代新增机制与真实先例

Linux kernel 目前已经把 AI 贡献治理写进正式开发文档。第一份文档《AI Coding Assistants》要求 AI 工具遵循标准内核开发流程，明确指出 AI agent **不得**添加 `Signed-off-by`，因为只有人类才能就 DCO 做法律认证；同时建议在 AI 参与的贡献中加入 `Assisted-by: AGENT_NAME:MODEL_VERSION ...` 形式的归因标签。第二份《Kernel Guidelines for Tool-Generated Content》则更进一步，把“来源透明”写成项目期望：提交者应说明用了什么工具、输入是什么、哪些内容受工具影响、如何测试；维护者可以把这类贡献当普通贡献、额外审查、要求更多测试，甚至直接拒绝。对你的项目来说，这两份文档几乎就是 provenance 与审查强度分级的现成蓝本。citeturn29view1turn29view0

LLVM 在 2025 年落地的 AI Tool Use Policy 也同样值得直接借鉴。其核心条款有四个：一是允许使用 AI 工具，但必须“human in the loop”；二是贡献者要阅读并审查所有 LLM 生成的代码和文本，对内容完全负责；三是实质性 tool-generated content 需要透明标注，可放在 PR 描述、commit message 或 `Assisted-by:` trailer；四是**禁止**没有人类批准的 agent 在 LLVM 的数字空间自动行动，包括自动发 PR 评论的 agent。LLVM 还把“reviewer attention 是稀缺资源”写成政策理由，用“extractive contribution”概念来描述那种把维护者当免费验证器的 AI 垃圾贡献。这个表述对你的项目非常重要，因为它正好对应你说的“验收标准被悄悄放宽”：如果作者自己都没跑完验证，就等于把验证成本外包给 reviewer。citeturn30view0

在“规范作为 AI 上下文”的工程化方面，2025–2026 年其实已经出现了三个很强的官方工具先例。GitHub Copilot 支持仓库级和路径级 custom instructions，文档直接说这些文件是为了给 Copilot“理解项目以及如何 build、test、validate 改动”提供额外上下文。Anthropic 的 Claude Code 用 `CLAUDE.md`、`.claude/rules/` 和 auto memory 来承载项目规则、构建命令、项目架构与路径级规则，而且支持组织级、用户级、项目级分层加载。OpenAI Codex 则把 `AGENTS.md` 作为“自动载入上下文”的开放格式，建议把 repo 布局、build/test/lint、PR 预期、完成定义等写进去。也就是说，**把规范、约束和验证命令放进仓库，让 agent 每次会话自动读取**，已经是工具官方支持的做法，而不是自创约定。citeturn32view0turn32view1turn32view3

更进一步，OpenAI 自己的开源仓库已经把这种“spec-as-context + verification-as-gate”写成真实仓库规则。`openai/codex` 的 `AGENTS.md` 要求改动完成后自动运行 `just fmt` 和相关测试；改依赖时必须同步更新锁文件；涉及共享 crate 时再决定是否跑完整测试。`openai-agents-js` 与 `openai-agents-python` 的 `AGENTS.md` 甚至把 `$code-change-verification`、`$changeset-validation`、`$implementation-strategy` 作为 mandatory skill：一旦改动运行时代码、测试、构建行为、导出 API、持久化模式、协议或用户可见行为，就必须先做策略判断，再跑验证步骤。这个先例非常贴合你的场景：**不是让 AI 想起要验证，而是让仓库规则把验证变成 agent 的工作定义。**citeturn35view0turn35view1turn35view2

因此，针对你的新语言，AI 治理最有力的组合拳应该是：用 Linux/LLVM 的 provenance 与 human-in-the-loop 作为底线，用 GitHub/Claude/Codex 的规则文件作为上下文承载层，用 OpenAI 开源仓库那种“完成前必须跑验证技能”的 repo 规则来把 conformance test、UI test、unsafe audit 变成 agent 的强制步骤。这样做的收益是：AI 不再靠“似是而非的内部统计相关性”补语义，而是被仓库里的规范、测试和流程文件持续拉回到项目真实边界内。citeturn29view1turn29view0turn30view0turn32view0turn32view1turn32view3turn35view0turn35view1turn35view2

## 新语言治理骨架 v0

### 组织与决策层

对 4–6 人团队，我建议只设一个**轻量治理三人组**，不设大委员会。三个角色分别是：语言负责人、实现负责人、质量与发布负责人。三人组的职责只保留四类：定义变更归类、指定 owner、处理跨域冲突、批准稳定化与 edition。除此之外，所有日常工作都下放给 RFC owner、实现 owner 和 spec editor。这个设计借的是 Rust Council 的“顶层只做协调与授权”而不是“顶层亲自做所有决定”。citeturn37view0

决策规则建议采用 **FCP-lite**：语义、unsafe、FFI、edition、破坏性变化，必须由三人组中至少两人同意，其中至少一人必须是语言负责人；并设置一个固定公开等待窗，例如 5–7 天。普通实现优化和机械重构则只需代码 owner + 另一名 reviewer。这样既保留了 Rust FCP 的“异议浮现窗口”，又避免了大团队那种 reviewer 子组负担。citeturn37view1

### 仓库结构

建议从第一天起把仓库切成下面几个一等目录，而不是把一切都塞进 `src/` 和 `docs/`：

```text
/spec
  /src
    /syntax
    /static-semantics
    /dynamic-semantics
    /undefined-behavior
    /implementation-defined
    /diagnostics
  /traceability
    requirements.csv
    tests.csv
/rfcs
  0000-template.md
  0001-...
/conformance
  /parser
  /typing
  /runtime
  /ffi
  /editions
  /feature-gates
/ui
  /diagnostics
  /snapshots
/unsafe-audit
  SAFETY_CASES.md
  geiger-baseline.json
  miri/
  reviews/
/agents
  AGENTS.md
  CLAUDE.md
/.github
  /copilot
    copilot-instructions.md
/scripts
  verify-trace.py
  bless-ui.sh
  gate-release.sh
/src
```

这里最关键的不是目录名，而是权力边界：`spec/` 是唯一规范源；`conformance/` 是唯一语义验收源；`ui/` 只验证诊断与交互表面；`unsafe-audit/` 单独记录所有需要人工承担语义责任的部分；`agents/` 和 `.github/copilot/` 则承载机器可读规则。这个结构分别借鉴了 FLS 的规范分段与追踪、WebAssembly 的 spec+tests 联动、Rust 的 UI 测试和 unsafe 纪律，以及 GitHub/Claude/Codex 的仓库规则文件机制。citeturn15search0turn14search7turn14search17turn18search1turn19search0turn24view1turn32view0turn32view1turn32view3

### 每类变更的流程门

建议只保留三档流程门，而不是全面复制 Rust 的 RFC/MCP/PR 生态。

| 变更类型 | 流程 | 合并门 | 发布门 |
|---|---|---|---|
| 文档措辞、非语义诊断调整、纯重构 | Direct PR | 至少 2 人评审；相关 UI/单元测试通过 | 无额外门 |
| 已有规范内的 bug fix、非公开实现优化、实验性内部开关 | Mini-RFC 或 issue + PR | 必须先有失败测试；PR 中引用规范条款或明确写“未规定行为修复”；相关 conformance/UI 测试通过 | 夜间版观察一段时间 |
| 新语法、新类型规则、新运行时语义、unsafe/FFI 规则、稳定化、破坏性修复、edition 内容 | Full RFC | RFC 合入后才可实现；实现 behind feature gate；跟踪 issue；Reference/spec diff；conformance tests；stabilization report；FCP-lite | 仅在稳定化决议或 edition 窗口发布 |

这套门槛的核心是把“AI 最容易偷跑的地方”全部升级到 full RFC：凡是会改变含义、扩大承诺边界、缩小未定义行为空间、增加 unsafe surface area 的变化，都不允许直接以“实现先行”方式进入主干。Rust 已经证明，feature gate、tracking issue、Reference PR 和 stabilization report 可以有效阻止“实现事实”自动升级为“语言承诺”；你的项目只需把它们缩成更轻量的版本。citeturn40search1turn40search0turn40search4turn37view1

### AI 参与的强制规则清单

下面这份清单建议直接写进 `AGENTS.md`、`CLAUDE.md` 和 PR 模板中：

- **必须标注来源**：凡实质性 AI 生成内容，PR 描述与 squash commit 必须带 `Assisted-by: TOOL:MODEL`，并简述哪些文件或段落受 AI 影响。这个规则直接借鉴 Linux kernel 与 LLVM。citeturn29view1turn29view0turn30view0
- **必须 human in the loop**：AI 不能自动合并、自动发布、自动在 issue/PR 中代表项目发言；只有人类作者能承担 DCO/署名/决策责任。citeturn29view1turn30view0
- **必须先读规范再改代码**：涉及 `src/` 的语义性修改前，agent 先读取对应 `spec/` 子树与相关 RFC；若缺少条款，先回报“规范缺失”，不得擅自补语义。这个规则由仓库 rules 文件承载。citeturn32view1turn32view3turn34search16
- **必须引用条款号**：每个语义性 PR 的描述都要列出受影响的 requirement ID；若没有 requirement ID，则该变更自动升级为“先补 spec 再改 impl”。这个规则是对 FLS traceability 的本地化裁剪。citeturn14search7turn14search17turn15search0
- **必须跑验证**：agent 完成代码改动前，必须运行匹配目录的 conformance/UI/unsafe 验证命令，并把结果贴进 PR；这借鉴了 GitHub“先跑 tests/static analysis”的审查建议，以及 OpenAI 开源仓库的 mandatory verification skills。citeturn33search3turn35view0turn35view1turn35view2
- **必须小步提交**：AI PR 限制为单一语义主题；超过阈值的 PR 自动拆分。LLVM 明确反对把维护者时间浪费在“extractive contribution”上，这条就是工程化落实。citeturn30view0
- **unsafe 必须可解释**：每个 unsafe block 前写 `// SAFETY:`，每个 public unsafe API 写 `# Safety`，并尽量保证一个 block 内只有单一 unsafe 操作。citeturn24view0turn24view1turn22search1
- **AI 不得定义 UB**：凡涉及 UB、内存模型、并发可见性、FFI ABI、优化前提的变更，AI 只能起草说明和测试，最终语义文本必须由人类维护者落笔。这个规则是基于 UCG 现状与 FLS“实现偏置”边界做的风险控制。citeturn21search0turn15search0

## 分级落地路线图

### 第一个月即可执行的最小可行版本

第一个月不要追求完整规范，而要先建立**不会漂移的最小闭环**。最小版本只做六件事：一是在 `spec/` 写出 20–40 条最高频、最高风险的规范条款，每条都带 requirement ID；二是建立 `conformance/` 和 `ui/` 两套基础测试；三是上线 full RFC 与 mini-RFC 模板；四是给所有实验性特性加 feature gate 与 tracking issue；五是启用 `undocumented_unsafe_blocks`/`missing_safety_doc` 风格要求，并用 `cargo-geiger` + Miri 跑 unsafe 基线；六是把 `AGENTS.md`/`CLAUDE.md`/Copilot instructions 放进仓库，强制 agent 先读规范、后跑验证。这样做已经足以显著压低“AI 实现看起来合理但超出规范”的风险。citeturn40search1turn40search0turn24view0turn24view1turn21search2turn21search3turn32view0turn32view1turn32view3

### 第一年的完整版本

到第一年末，治理骨架应升级到完整版本：规范采用 FLS 风格章节模板；每个 requirement ID 自动生成 traceability matrix；所有稳定特性必须有 stabilization report；发布流程中加入一次 crater-like 生态回归跑批；unsafe surface area 建立长期审计台账；准备第一版 edition 机制，用来接纳不得不发生的兼容性修复；AI 规则则从“建议”升级为 CI 门，例如 PR 缺 provenance、缺测试结果、缺 requirement ID 时直接阻断。这个阶段的目标不是模仿 Rust 的组织规模，而是把 Rust/Wasm/Linux/LLVM 已证明有效的**硬约束**浓缩成你们团队能长期维护的制度化工具链。citeturn14search17turn40search0turn17search0turn37view3turn29view0turn29view1turn30view0

### 开放问题与边界

有两点需要明确。其一，Rust 的官方语言规范在 2025–2026 仍处于“吸收 FLS、继续演化”的阶段，因此对内存模型、unsafe 语义等最深水区，不能把“Rust 目前的文档状态”误读成完全定型的标准；你在借鉴时应更保守。其二，关于 AI 贡献政策，Linux 与 LLVM 的官方文本已经足够强，但其他语言核心项目并没有都形成同等成熟的 AI 政策文本，因此本文在 AI 治理部分主要依赖 Linux、LLVM 以及 GitHub/Anthropic/OpenAI 的官方产品与仓库实践，而不是为每个语言项目都找到完全对等的先例。citeturn21search0turn15search0turn29view0turn29view1turn30view0turn32view0turn32view1turn32view3
