## 执行摘要

本报告面向新语言的性能工程体系建设需求，系统调研了 **GPU 基准方法论**、**NVIDIA Nsight 工具链深度集成**、**CUPTI 内建 Profiler**、**编译器侧性能观测**以及**性能回归门禁**五大领域，旨在从语言设计的"第零天"建立可解释、可重复、可对标的性能验收体系。报告的核心主张是：**性能可观测性应当被视为语言运行时的一等公民**，而非事后 bolt-on 的工具链附加功能。

在 GPU 基准方法论层面，现代 GPU 的动态频率调整、WDDM 超时检测恢复（TDR）机制以及云环境的硬件异质性，使得"测量"本身成为一个非平凡的工程问题  [(Fidelity GPU Kernel Benchmarking)](https://standardkernel.com/blog/in-pursuit-of-high-fidelity-gpu-kernel-benchmarking/) 。本报告提炼了可复现 GPU microbenchmark 的六阶段工作流——从环境验证到时钟锁定、warmup、稳态采样、统计聚合到报告归档——每一步均附有具体可执行的参数配置。

在工具链层面，Nsight Compute 提供了从 kernel 级 SOL（Speed of Light）指标到内存层次 Roofline 分析的全栈能力  [(CSDN博客)](https://blog.csdn.net/m0_61864577/article/details/140052386) ，其 CLI 接口 `ncu` 支持 JSON/CSV 输出，可直接接入 CI pipeline。Nsight Systems 的 NVTX 标注体系则允许运行时以零侵入方式标记代码区间，用户无需修改业务代码即可获得完整的时间线视图  [(Libraries.io)](https://libraries.io/pypi/nvtx) 。CUPTI 作为底层接口，提供了 Activity/Callback/Profiling/PC Sampling 四层 API，支持在语言运行时内建低开销的 profiler，但需特别注意与 Nsight 工具的互斥限制  [(wrfranklin.org)](https://wrfranklin.org/wiki/ParallelComputingSpring2015/cuda/nvidia/doc/pdf/CUPTI_Library.pdf) 。

在编译器侧，rustc 的 `self-profile` 机制和 Clang 的 `-ftime-trace` 提供了成熟的编译时间 telemetry 方案  [(fasterthanli.me)](https://fasterthanli.me/articles/why-is-my-rust-build-so-slow) ，可直接借鉴用于新语言编译器的增量编译性能验收。

在回归门禁层面，LLVM compile-time tracker 和 rustc perf.rust-lang.org 的实践表明，基于 **instruction count** 的稳定代理指标配合 **Mann-Whitney U 检验**的非参数统计方法，可有效检测 0.3% 级别的回归  [(nikic's Blog)](https://www.npopov.com/2020/05/10/Make-LLVM-fast-again.html) 。

- **(A)** 新语言"性能验收体系 v0"分层设计（L0–L4）
- **(B)** 运行时内建观测（NVTX+计数器）的 API 草案

## 1. 基准方法论：GPU Microbenchmark 的正确做法

GPU microbenchmark 是性能验收体系的基础层，其方法论的正确性直接决定了后续所有性能数据的可靠性。与 CPU benchmark 相比，GPU benchmark 面临更复杂的测量环境：动态频率调整、多种内存层次、异步执行模型以及驱动层/操作系统层的干扰。本节从 warmup/稳态划分、时钟锁定、多次运行统计、常见测量谬误和业界规范借鉴五个维度展开。

### 1.1 Warmup 与稳态划分

GPU kernel 的首次执行通常呈现"冷启动"特征：驱动可能需要 JIT 编译 PTX 代码、常量缓存未预热、纹理缓存未建立、GPU 频率可能尚未提升至目标水平。因此，warmup 阶段是获取稳定测量结果的必要前提。

业界最佳实践表明，**warmup 迭代次数应当根据 kernel 执行时间的稳定性来判断，而非固定数值**。具体而言，推荐的 warmup 协议如下：

- **最小 warmup 次数**: 10 次迭代。这是 MLPerf Inference v4.0 和 SOL-ExecBench 共同采用的参数  [(arXiv.org)](https://arxiv.org/html/2603.19173v1) 。
- **稳态判定标准**: 连续 5 次迭代的执行时间变异系数（CV = std/mean）低于 **5%**。
- **超时保护**: 单次 warmup 迭代超时 300 秒即终止，防止 hang。
- **L2 缓存清除**: 每次 timed 迭代前清除 L2 缓存（通过清零一个 256MB 的 device buffer），避免跨迭代的缓存命中干扰  [(arXiv.org)](https://arxiv.org/html/2603.19173v1) 。

稳态划分的方法论借鉴自 SPEC CPU 的"training run"哲学  [(Fujitsu)](https://sp.ts.fujitsu.com/dmsp/Publications/public/wp-benchmark-overview-speccpu2017-en.pdf) ：在正式测量前，通过一系列预运行让系统达到热平衡状态。对于 GPU 而言，这还包括让温度稳定到风扇曲线对应的平衡温度，以及让 GPU Boost 频率稳定到功率限制下的可持续频率。

值得注意的是，Nsight Compute 在分析时提供 `--clock-control base` 选项，会自动将 GPU 频率锁定到基础频率以消除频率波动的影响  [(Bing)](https://www.bing.com/ck/a?!=&fclid=3dd8d27b-f3ca-680c-1265-c47af23d6983&hsh=4&ntb=1&p=9e63eb3f077d488c8b22e65439741ca337fa7cb8559fa71c7c1546579153df6bJmltdHM9MTc0ODkwODgwMA&psq=Tensor Core MMA instruction set Ada Lovelace GPU&ptn=3&u=a1aHR0cHM6Ly93d3cuc3BhdHRlcnMuY2EvbW1hLW1hdG11bA&ver=2) 。但在生产环境中进行基准测试时，更推荐手动锁定频率（见 1.2 节），以便完全控制测试条件。

### 1.2 时钟锁定（Locked Clocks）的具体操作

GPU 的动态频率调整是影响测量可重复性的最大干扰因素。现代 NVIDIA GPU 同时存在 SM（Streaming Multiprocessor）时钟和内存时钟两个独立的频率域，两者均可动态调整  [(Fidelity GPU Kernel Benchmarking)](https://standardkernel.com/blog/in-pursuit-of-high-fidelity-gpu-kernel-benchmarking/) 。

#### 1.2.1 时钟锁定命令

时钟锁定通过 `nvidia-smi` 工具完成，具体操作步骤如下  [(youngju.dev)](https://www.youngju.dev/blog/gpu-cuda/2026-03-17-cuda-programming-complete-guide.en) ：

```bash
# Step 1: 启用持久模式（Persistence Mode）
sudo nvidia-smi -pm 1

# Step 2: 锁定 GPU SM 时钟（以 A100 1410 MHz 为例）
sudo nvidia-smi -lgc 1410,1410

# Step 3: 锁定内存时钟（以 A100 1593 MHz 为例）
sudo nvidia-smi -lmc 1593,1593

# 或使用 --lock-gpu-clocks 和 --lock-memory-clocks 语法
sudo nvidia-smi --lock-gpu-clocks=1410
sudo nvidia-smi --lock-memory-clocks=1593
```

#### 1.2.2 频率选择策略

频率锁定的目标值选择需要权衡可重复性与代表性。业界存在三种策略  [(Fidelity GPU Kernel Benchmarking)](https://standardkernel.com/blog/in-pursuit-of-high-fidelity-gpu-kernel-benchmarking/) ：

| 策略 | 频率值 | 优点 | 缺点 |
|------|--------|------|------|
| **Base Clock** | GPU 标称基础频率 | 完全可重复，不受功耗限制影响 | 不代表典型性能 |
| **Boost Clock** | GPU 标称最大 Boost 频率 | 可重复，接近理论峰值 | 可能触发功耗限制导致实际频率下降 |
| **Sustainable Clock** | 实测可持续频率 | 最代表实际工作负载 | 需预先测量，不同 GPU 不同 |

**推荐做法**：对于验收测试，使用 **Boost Clock** 锁定，因为它最接近硬件的理论峰值，便于计算 SOL（Speed of Light）百分比。但前提是测试 workload 的功耗不超过 TDP 限制，否则硬件会 override 手动设置而 throttle  [(Fidelity GPU Kernel Benchmarking)](https://standardkernel.com/blog/in-pursuit-of-high-fidelity-gpu-kernel-benchmarking/) 。验证方法是在锁定频率后运行 workload 并监控 `nvidia-smi dmon` 的输出，确认实际频率等于锁定频率。

#### 1.2.3 环境隔离要求

除了时钟锁定，完整的基准环境还包括  [(Github)](https://github.com/yuhezhang-ai/triton-augment/blob/main/docs/auto-tuning.md) ：

- **专用 GPU 实例**：无其他进程共享 GPU（通过 `nvidia-smi pmon` 验证）
- **禁用 Auto Boost**：`nvidia-smi --auto-boost-default=0`（部分数据中心 GPU 支持）
- **温度稳定**：在正式测量前运行 60 秒 burn-in，让温度达到稳态
- **PCIe 链路确认**：`nvidia-smi topo -m` 确认 GPU 与 CPU 的 PCIe 拓扑

### 1.3 多次运行统计方法

单次 kernel 执行时间的测量受多种噪声源影响：OS 调度中断、GPU 频率微调、内存总线争用、温度相关的 throttle 等。因此，必须通过多次运行和稳健的统计方法来提取信号。

#### 1.3.1 推荐参数

| 参数 | 推荐值 | 来源 |
|------|--------|------|
| Timed iterations per trial | **50** | SOL-ExecBench, CUTLASS |
| Number of trials | **3** | MLPerf, SOL-ExecBench |
| Warmup iterations | **10** | MLPerf, SOL-ExecBench |
| L2 cache clear between iterations | **Yes** (256MB buffer) | SOL-ExecBench  [(arXiv.org)](https://arxiv.org/html/2603.19173v1)  |
| Trial-level统计 | **中位数 (Median)** | 对异常值稳健 |
| 跨 trial 汇总 | **Trimmed Mean (去头尾 20%)** | 平衡稳健性与效率 |
| CV 阈值 | **< 5%** | 可接受的数据质量 |

#### 1.3.2 异常值处理

异常值 rejection 推荐采用 **IQR（Interquartile Range）方法**  [(FasterCapital)](https://fastercapital.com/topics/using-interquartile-range-for-outlier-detection.html) ：

```
Q1 = 第 25 百分位数
Q3 = 第 75 百分位数
IQR = Q3 - Q1
有效范围 = [Q1 - 1.5×IQR, Q3 + 1.5×IQR]
```

任何超出该范围的测量值被视为异常值并剔除。IQR 方法的优势在于不假设数据服从正态分布，且对极端异常值具有鲁棒性。

#### 1.3.3 报告格式

最终报告应当包含以下统计量  [(Fujitsu)](https://sp.ts.fujitsu.com/dmsp/Publications/public/wp-benchmark-overview-speccpu2017-en.pdf) ：

- **Central tendency**: Trimmed mean（去头尾 20%）或中位数
- **Dispersion**: 标准差（std）和变异系数（CV）
- **Absolute range**: Min / Max
- **Confidence**: 95% 置信区间（通过 bootstrap 重采样计算）

### 1.4 常见测量谬误

GPU 基准测试中常见的测量谬误可归纳为以下几类：

| 谬误 | 描述 | 影响 | 避免方法 |
|------|------|------|----------|
| **忽略 warmup** | 首次运行包含 JIT 编译和缓存预热 | 严重低估真实性能 | 至少 10 次 warmup |
| **未锁定时钟** | GPU 动态调整频率 | 运行间差异可达 50%+  [(Fidelity GPU Kernel Benchmarking)](https://standardkernel.com/blog/in-pursuit-of-high-fidelity-gpu-kernel-benchmarking/)  | `nvidia-smi -lgc` |
| **WDDM TDR 超时** | Windows 2 秒 watchdog | Kernel 被强制终止 | 见第 2 节 |
| **CUDA Event vs CPU Timer** | `cudaEvent` 与 CPU timer 混用 | 时序不一致 | 统一使用 CUDA Event |
| **L2 缓存污染** | 前次迭代数据残留 | 高估带宽 | 每次迭代前清缓存 |
| **异步同步点错误** | `cudaDeviceSynchronize()` 位置不当 | 包含启动开销 | 仅环绕测量区域 |
| **云环境隔离不足** | 共享实例资源争用 | 不可预测波动 | 专用实例 + 隔离验证 |

### 1.5 学术与工业基准规范借鉴

#### 1.5.1 MLPerf 的 Run Rules

MLPerf 作为 AI 领域的事实标准基准组织，其规则体系对 GPU 基准测试具有直接借鉴价值。MLPerf Inference v4.0/v4.1 的核心规则包括  [(Github)](https://github.com/mlcommons/inference) ：

- **运行次数**：LoadGen 自动决定运行次数以达到目标统计置信度
- **统计要求**：99th percentile 结果必须满足置信区间要求
- **验证机制**：每个提交必须包含 accuracy 验证日志，确保正确性优先于速度
- **分区（Division）**：Closed division 强制使用相同模型和优化空间，Open division 允许任意优化
- **时钟要求**：无强制时钟锁定，但要求报告 GPU 型号和驱动版本

#### 1.5.2 SPEC 方法论借鉴

SPEC CPU 2026 的方法论强调七个核心原则：可移植性、可重复性、可比较性、正确性验证、代表性、公平披露和公平使用  [(igor´sLAB)](https://www.igorslab.de/en/?p=317730) 。其中对新语言性能验收体系最具借鉴意义的是：

- **正确性验证（Validation）**：SPEC 的 harness 在测量性能的同时验证输出结果的正确性  [(igor´sLAB)](https://www.igorslab.de/en/?p=317730) 。新语言的基准套件应当在每个测试用例中内建 reference output 的校验（golden check）。
- **几何均值聚合**：SPEC 使用几何均值而非算术均值来防止单一 benchmark 过度影响总分  [(jimgray.azurewebsites.net)](https://jimgray.azurewebsites.net/benchmarkhandbook/chapter9.pdf) 。对于多维度性能指标，几何均值提供了更公平的汇总方式。
- **Base vs Peak 双模式**：SPEC 允许保守优化（Base，统一优化标志）和激进优化（Peak，逐 benchmark 优化）两种模式  [(Fujitsu)](https://sp.ts.fujitsu.com/dmsp/Publications/public/wp-benchmark-overview-speccpu2017-en.pdf) 。新语言可以借鉴此设计，分别测量"用户友好优化级别"和"极致优化级别"的性能。
- **Peer Review**：所有 SPEC 结果需经 2 周同行评审后发布  [(Oak Ridge Leadership Computing Facility)](https://www.olcf.ornl.gov/wp-content/uploads/2020/11/A_Overview_and_BenchmarkSelection_2020.pdf) 。新语言的性能回归检测也应包含人工 triage 环节。

## 2. Windows/WDDM 特有问题（专节）

Windows Display Driver Model (WDDM) 下的 CUDA 程序面临一组 Linux 上不存在或表现截然不同的特有问题。这些问题对基准测试的可重复性和正确性构成严重威胁，必须单独成章予以处理。

### 2.1 TDR（Timeout Detection and Recovery）机制

WDDM 内置的 TDR 机制会在 GPU 无响应超过配置的超时时间后强制重置显示驱动  [(NVIDIA Developer Forums)](https://forums.developer.nvidia.com/t/display-driver-stopped-responding-and-has-recovered-wddm-timeout-detection-and-recovery/15056) 。默认超时时间为 **2 秒**（Windows Vista/7/8/10/11 均如此） [(Puget Systems)](https://www.pugetsystems.com/labs/hpc/working-around-tdr-in-windows-for-a-better-gpu-computing-experience-777/) 。

#### TDR 对基准测试的影响

- **Kernel 执行限制**：任何单 kernel 执行时间超过 2 秒（或累计 batch 超过 2 秒）将触发 `cudaErrorLaunchTimeout`（error 702），驱动被重置，所有 CUDA 上下文失效  [(rightnowai.co)](https://www.rightnowai.co/guides/cuda-errors/cuda-error-timeout) 
- **测量中断**：触发 TDR 后，当前 CUDA context 内所有 device memory allocation 变为无效，程序必须重建所有状态才能继续  [(NVIDIA Developer Forums)](https://forums.developer.nvidia.com/t/cudasynchronizedevice-returns-error-code-6/22977) 
- **数据丢失**：正在进行的测量被迫中断，已收集数据可能不完整
- **不可预测性**：驱动可能 batch 多个短 kernel launch 后统一提交，导致累计超时  [(NVIDIA Developer Forums)](https://forums.developer.nvidia.com/t/display-driver-stopped-responding-and-has-recovered-wddm-timeout-detection-and-recovery/15056) 

#### TDR 配置方法

TDR 参数通过 Windows 注册表配置  [(rightnowai.co)](https://www.rightnowai.co/guides/cuda-errors/cuda-error-timeout) ：

```powershell
# 以 Administrator 运行 PowerShell
# 延长 TDR 延迟至 60 秒（推荐用于专用计算 GPU）
Set-ItemProperty -Path "HKLM:\System\CurrentControlSet\Control\GraphicsDrivers" `
  -Name TdrDelay -Type DWord -Value 60 -Force

Set-ItemProperty -Path "HKLM:\System\CurrentControlSet\Control\GraphicsDrivers" `
  -Name TdrDdiDelay -Type DWord -Value 60 -Force

# 完全禁用 TDR（仅推荐用于无显示输出的专用计算 GPU）
Set-ItemProperty -Path "HKLM:\System\CurrentControlSet\Control\GraphicsDrivers" `
  -Name TdrLevel -Type DWord -Value 0 -Force

# 修改后必须重启 Windows 才生效
```

#### 基准测试中的 TDR 规避策略

| 策略 | 适用场景 | 操作 |
|------|----------|------|
| **拆分 Kernel** | 所有场景 | 将长 kernel 拆分为多轮短 kernel，每轮 < 100ms |
| **延长 TDR** | 专用计算 GPU | TdrDelay=60 |
| **禁用 TDR** | 无头服务器 | TdrLevel=0 |
| **使用 TCC 驱动** | Tesla/Quadro GPU | `nvidia-smi -dm 1` 切换至 TCC 模式 |
| **使用非显示 GPU** | 多 GPU 系统 | `CUDA_VISIBLE_DEVICES` 选择无显示输出的 GPU |

### 2.2 WDDM 下的计时特有问题

#### 2.2.1 CUDA Event Timer 的额外开销

在 WDDM 下，`cudaEventElapsedTime()` 返回的时间包含 WDDM 调度和批处理的开销，可能显著高于实际 kernel 执行时间。WDDM 的 User-Mode Driver（UMD）会 batch 多个 CUDA API 调用后统一提交给 Kernel-Mode Driver（KMD），这种批处理机制引入了不可预测的延迟  [(NVIDIA Developer Forums)](https://forums.developer.nvidia.com/t/strannge-behaviour-of-kernel-unspecified-launch-failure/64007) 。

**缓解方法**：
- 使用 `cudaEventBlockingSync` 标志创建 event，减少忙等待
- 在测量区域前后插入 `cudaStreamSynchronize()` 强制刷新 WDDM batch queue
- 考虑使用 TCC 驱动模式（如果硬件支持），完全绕过 WDDM

#### 2.2.2 高精度 CPU Timer 的替代方案

在 WDDM 下，QueryPerformanceCounter（QPC）仍然是可用的最高精度 CPU timer。与 CUDA Event 混用时需注意  [(NVIDIA Developer Forums)](https://forums.developer.nvidia.com/t/any-way-to-measure-the-latency-of-a-kernel-launch/221413) ：

```cpp
// 推荐：仅使用 CUDA Event 进行 GPU 端计时
cudaEvent_t start, stop;
cudaEventCreate(&start);
cudaEventCreate(&stop);

cudaEventRecord(start, stream);
kernel<<<grid, block, 0, stream>>>(...);
cudaEventRecord(stop, stream);
cudaEventSynchronize(stop);

float elapsed_ms;
cudaEventElapsedTime(&elapsed_ms, start, stop);
```

### 2.3 其他 WDDM 限制

| 限制 | 描述 | 影响 |
|------|------|------|
| **Unified Memory 限制** | WDDM 下 `cudaMallocManaged` 性能显著低于 Linux | UM benchmark 结果不可跨平台比较 |
| **Multi-Adapter** | WDDM 的 GPU 调度策略与 Linux 不同 | 多 GPU 负载均衡行为差异 |
| **Driver Store** | 驱动文件由 Windows 统一管理 | 驱动版本控制更复杂 |
| **WSL2 继承** | WSL2 通过 WDDM 访问 GPU，继承所有限制 | WSL2 下同样受 TDR 约束  [(Github)](https://github.com/noonghunna/club-3090/blob/master/docs/HARDWARE.md)  |

## 3. Nsight Compute 深度分析

Nsight Compute（`ncu`）是 NVIDIA 提供的 kernel 级性能分析工具，其指标体系基于硬件性能计数器，提供从 SOL 分析到内存层次 Roofline 的全栈能力。

### 3.1 Kernel 级指标体系

#### 3.1.1 SOL（Speed of Light）分析

SOL 指标衡量 kernel 的实际性能相对于硬件理论峰值的百分比，是 Nsight Compute 的核心诊断指标  [(Note And Save)](https://noteandsave.com/blog/cuda-occupancy-performance) 。

| SOL 类别 | 指标名称 | 含义 | 诊断阈值 |
|----------|----------|------|----------|
| **SM SOL** | `sm__throughput.avg.pct_of_peak_sustained_elapsed` | SM 计算单元利用率 | < 60% 暗示延迟瓶颈 |
| **Memory SOL** | `memory__throughput.avg.pct_of_peak_sustained_elapsed` | 内存子系统利用率 | 接近 100% = 内存受限 |
| **DRAM SOL** | `dram__throughput.avg.pct_of_peak_sustained_elapsed` | HBM/GDDR 带宽利用率 | 接近 100% = DRAM 瓶颈 |
| **L1/TEX SOL** | `l1tex__throughput.avg.pct_of_peak_sustained_elapsed` | L1/Texture cache 利用率 | 低 = cache 未充分利用 |
| **L2 SOL** | `lts__throughput.avg.pct_of_peak_sustained_elapsed` | L2 cache 利用率 | 结合 hit rate 分析 |

**诊断流程**：
1. 若 **Memory SOL 高且 SM SOL 低** → **内存受限（Memory-bound）**：优化内存访问模式（合并、缓存、共享内存 tiling） [(Note And Save)](https://noteandsave.com/blog/cuda-occupancy-performance) 
2. 若 **SM SOL 高且 Memory SOL 低** → **计算受限（Compute-bound）**：减少指令数、使用 Tensor Cores、强度削减
3. 若 **两者均低** → **延迟受限（Latency-bound）**：增加 occupancy、调整 block size、减少寄存器使用

#### 3.1.2 Occupancy 分析

Occupancy（占用率）定义为每个 SM 上实际活跃的 warp 数量与理论最大值的比率。Nsight Compute 提供以下相关指标  [(ornl.gov)](https://info.ornl.gov/sites/publications/Files/Pub148652.pdf) ：

| 指标 | 说明 |
|------|------|
| `launch__occupancy` | 实际占用率 |
| `launch__block_size` | Block 大小 |
| `launch__registers_per_thread` | 每线程寄存器数 |
| `launch__shared_mem_per_block_static` | 静态共享内存/Block |
| `launch__shared_mem_per_block_dynamic` | 动态共享内存/Block |

**关键约束链**：寄存器/共享内存用量 → Block 级资源消耗 → 每 SM 可同时驻留的 Block 数 → Occupancy → 延迟隐藏能力。

#### 3.1.3 内存层次吞吐分析

Nsight Compute 提供了从 L1 到 HBM 的完整内存层次分析  [(ornl.gov)](https://info.ornl.gov/sites/publications/Files/Pub148652.pdf) ：

| 层次 | 关键指标 | 优化方向 |
|------|----------|----------|
| **L1/TEX Cache** | `l1tex__t_sectors_hit_rate` | 提高 hit rate，减少 sector miss |
| **L2 Cache** | `lts__t_sectors_hit_rate` | 数据复用、访问模式优化 |
| **DRAM** | `dram__bytes.sum.per_second` | 带宽最大化，避免 bank conflict |
| **Shared Memory** | `smsp__sass_average_data_bytes_per_sector_mem_shared.pct` | Bank conflict 消除 |

#### 3.1.4 Roofline 分析

Nsight Compute 内置 Roofline 图表功能，基于以下公式构建  [(CSDN博客)](https://blog.csdn.net/m0_61864577/article/details/140052386) ：

- **Peak Work（理论峰值计算性能）**：基于 GPU 标称 FFMA 操作吞吐 × SM 频率
- **Peak Traffic（理论峰值内存带宽）**：基于 HBM 标称带宽
- **Achieved Work（实际计算性能）**：统计实际执行的 FFMA 等效操作数
- **Achieved Traffic（实际内存流量）**：统计各级存储的字节传输量

Roofline 模型的核心洞察是：kernel 的性能受限于计算吞吐或内存带宽两者中的较小者，其比值即为 operational intensity（每字节内存访问的计算操作数）。

### 3.2 CLI 自动化与 CI 集成

Nsight Compute 的 CLI 接口 `ncu` 支持完全自动化的性能数据收集，可直接集成到 CI pipeline。

#### 3.2.1 核心 CLI 命令


#### 3.2.2 JSON/CSV 解析与自动化

`ncu` 支持 `--csv` 和 `--json` 输出格式，便于程序化解析：

```python
import csv
```

### 3.3 Source-Level 关联与编译器要求

Nsight Compute 的 source-level 关联功能需要将 SASS 汇编指令映射回源代码位置，这要求编译器在编译时保留调试信息。

#### 3.3.1 编译器标志要求

| 编译器 | 必需标志 | 说明 |
|--------|----------|------|
| **nvcc** | `--generate-line-info` 或 `-lineinfo` | 生成行号信息，无完整 DWARF |
| **nvcc** | `-G` | 生成完整调试信息（影响优化） |
| **Clang/LLVM** | `-g` + `-fdebug-line-events` | 标准 DWARF 调试信息 |
| **新语言编译器** | 等效 `--generate-line-info` | 输出 ELF/DWARF 或 CUDA 调试段 |

**推荐**：对于性能验收测试，使用 `-lineinfo`（非 `-G`），因为它保留优化后的代码结构，同时提供足够的 source-line 映射。`-G` 会禁用大多数优化，导致测量结果不代表生产性能。

#### 3.3.2 Source Correlation 的数据流

Nsight Compute 通过以下路径实现 source-SASS 关联：

1. 编译器在 CUDA ELF 中嵌入 `.nv_debug_info` 段或标准 DWARF 段
2. Nsight Compute 读取该段，建立 PC（程序计数器）→ source file + line 的映射
3. 性能计数器数据按 PC 聚合后，通过映射表归因到源代码行
4. UI 或 CLI 报告中展示每行代码对应的指令数、stall 原因等

## 4. Nsight Systems 时间线分析

Nsight Systems（`nsys`）提供系统级的时间线视图，展示 CPU 线程、CUDA API 调用、kernel 执行、内存传输和 NVTX 标注区间之间的时序关系。

### 4.1 时间线分析能力

Nsight Systems 的核心价值在于揭示 CPU-GPU 协作的时序特征，包括  [(lxp.lu)](https://docs.lxp.lu/howto/pytorch-profiling-with-nsight/) ：

- **CPU 端瓶颈**：host 端代码是否在 GPU 空闲时未提交足够工作
- **GPU 空闲时间**：kernel 之间的 gap 是否过大，暗示 launch latency 或同步问题
- **内存传输重叠**：H2D/D2H 传输是否与 kernel 执行充分重叠
- **Stream 并行性**：多个 CUDA stream 是否真正并发执行

关键操作命令：

```bash
# 采集完整时间线
nsys profile --trace=cuda,nvtx,osrt,mpi \
    --output=report.nsys-rep \
    ./application

# 导出统计报告
nsys stats --report nvtx_sum report.nsys-rep

# 导出为 JSON（便于程序化解析）
nsys export -t json report.nsys-rep -o report.json
```

### 4.2 NVTX 标注体系与运行时集成

NVTX（NVIDIA Tools Extension）是一个 C-based API，用于在应用程序中标注事件、代码区间和资源。Nsight Systems 自动捕获 NVTX 标注并在时间线中展示  [(Bing)](https://www.bing.com/ck/a?!=&fclid=3ed54746-e2cf-6f31-3bfe-5144e3356e04&hsh=4&ntb=1&p=16a98c5bc699265e81ab9605909b3a1b917e317953156a97f6491a171cb5412cJmltdHM9MTc0OTA4MTYwMA&ptn=3&u=a1aHR0cHM6Ly9naXRodWIuY29tL05WSURJQS9OVlRY&ver=2) 。

#### 4.2.1 NVTX 3 API 概览

NVTX v3 提供了三类核心标注原语：

| 类型 | API | 用途 |
|------|-----|------|
| **Markers** | `nvtxMark(A/W)` | 标注瞬时事件（单点） |
| **Push/Pop Ranges** | `nvtxRangePush(A/W)` / `nvtxRangePop()` | 嵌套区间标注 |
| **Start/End Ranges** | `nvtxRangeStart(A/W)` / `nvtxRangeEnd()` | 任意重叠区间 |

#### 4.2.2 C++ 集成示例

```cpp
#include <nvtx3/nvtx3.hpp>

void training_step(Model& model, DataBatch& batch) {
    NVTX3_FUNC_RANGE();  // 自动标注整个函数
    
    nvtx3::scoped_range forward{"forward_pass"};
    auto output = model.forward(batch.input);
    forward.end();
    
    nvtx3::scoped_range backward{"backward_pass"};
    auto loss = compute_loss(output, batch.target);
    loss.backward();
    backward.end();
    
    nvtx3::scoped_range opt{"optimizer_step"};
    optimizer.step();
}
```

#### 4.2.3 Python 集成示例

```python
import torch.cuda.nvtx as nvtx

# Push/Pop 模式
nvtx.range_push("forward_pass")
output = model(input)
nvtx.range_pop()

# 上下文管理器模式（推荐）
with torch.cuda.nvtx.range("training_step"):
    output = model(input)
    loss = criterion(output, target)
    loss.backward()
    optimizer.step()
```

#### 4.2.4 NVTX 扩展（v3.2 新增）

NVTX 3.2 引入了多项新扩展，对运行时内建观测特别有价值  [(NVIDIA Developer Forums)](https://forums.developer.nvidia.com/t/nvtx-3-2-release-announcement/333012) ：

- **Payload Extension**：向事件附加结构化数据（如 tensor shape、batch size）
- **Counters Extension**：记录可统计的数值量（如 loss 值、learning rate），可在 Nsight Systems 中显示为时间线图表
- **Memory Extension**：标注 CPU/GPU 内存堆和区域，辅助内存分析

#### 4.2.5 零成本集成设计

新语言运行时的 NVTX 集成应当遵循**零成本抽象**原则：当 profiling 未启用时，所有 NVTX 调用应当被编译器完全消除（inline no-op）。这可以通过条件编译或链接时桩函数替换实现。

## 5. CUPTI 与 PM Sampling

CUPTI（CUDA Profiling Tools Interface）是 NVIDIA 提供的动态库，支持创建针对 CUDA 应用的 profiling 和 tracing 工具。它是 Nsight Compute 和 Nsight Systems 的底层基础。

### 5.1 CUPTI API 体系与能力矩阵

CUPTI 提供了多个 API 家族，各自适用于不同场景  [(wrfranklin.org)](https://wrfranklin.org/wiki/ParallelComputingSpring2015/cuda/nvidia/doc/pdf/CUPTI_Library.pdf) ：

| API 家族 | 能力 | 开销 | 适用场景 |
|----------|------|------|----------|
| **Activity API** | 异步收集 CPU/GPU 活动 trace | 低 (1-5%) | Timeline tracing |
| **Callback API** | 拦截 CUDA Runtime/Driver API 调用 | 极低 | API 调用计数、注入 |
| **Profiling API** | 收集硬件性能计数器（metrics） | 中等 (5-20%) | Kernel 级详细分析 |
| **Event API** | 查询/配置/读取硬件事件计数器 | 低 | 简单计数器 |
| **Metric API** | 从事件计算高级指标 | 低 | 利用率、吞吐率 |
| **PC Sampling API** | 程序计数器采样（含 stall reason） | 低-中 (1-15%) | 热点定位 |
| **PM Sampling API** | 性能监控采样（新） | 低 | 细粒度监控 |
| **Checkpoint API** | 自动保存/恢复 device 状态 | 中等 | Kernel replay |

### 5.2 语言运行时内建 Profiler 实现路径

新语言运行时内建 profiler 的推荐实现基于 **CUPTI Activity API + Callback API** 的组合，原因如下：

1. **低开销**：Activity API 的异步缓冲机制将数据收集开销控制在 1-5%  [(eunomia.dev)](https://eunomia.dev/others/cupti-tutorial/pc_sampling_continuous/) 
2. **非侵入**：通过 LD_PRELOAD 或运行时动态加载，无需修改用户代码
3. **全面覆盖**：自动捕获所有 kernel launch、memcpy、memset 等活动
4. **时间戳对齐**：提供 CPU/GPU 统一的时间戳，便于时间线重建

#### 5.2.1 推荐架构

```
┌─────────────────────────────────────────┐
│          User Application               │
├─────────────────────────────────────────┤
│      Language Runtime (lib)             │
│  ┌─────────────────────────────────┐    │
│  │     Built-in Profiler Module    │    │
│  │  ┌─────────┐   ┌─────────────┐ │    │
│  │  │ CUPTI   │   │  NVTX       │ │    │
│  │  │ Activity│   │  Emitter    │ │    │
│  │  │ API     │   │  (opt-in)   │ │    │
│  │  └────┬────┘   └──────┬──────┘ │    │
│  │       │                │         │    │
│  │  ┌────┴────────────────┴────┐    │    │
│  │  │   Ring Buffer (async)    │    │    │
│  │  │   → JSON/Chrome Trace    │    │    │
│  │  └─────────────────────────┘    │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

#### 5.2.2 CUPTI 初始化与事件订阅

```cpp
// CUPTI Activity API 初始化示例
CUpti_SubscriberHandle subscriber;
cuptiSubscribe(&subscriber, 
    [](void* userdata, CUpti_CallbackDomain domain, 
       CUpti_CallbackId cbid, const void* cbdata) {
        // 处理 kernel launch、memcpy 等活动记录
    }, 
    nullptr);

// 启用关注的活动类型
cuptiEnableCallback(1, subscriber, CUPTI_CB_DOMAIN_RUNTIME_API,
                    CUPTI_RUNTIME_TRACE_CBID_cudaLaunchKernel);
cuptiEnableCallback(1, subscriber, CUPTI_CB_DOMAIN_RUNTIME_API,
                    CUPTI_RUNTIME_TRACE_CBID_cudaMemcpy);
```

#### 5.2.3 PC Sampling 集成

CUPTI PC Sampling API 支持 continuous mode，可在不串行化 kernel 执行的情况下收集数据， overhead 控制在 **1-15%** 之间（取决于采样频率） [(eunomia.dev)](https://eunomia.dev/others/cupti-tutorial/pc_sampling_continuous/) 。

| 采样周期 | Overhead | 适用场景 |
|----------|----------|----------|
| 21-31 | 1-5% | 生产环境持续采样 |
| 11-20 | 5-15% | 开发调试 |
| 5-10 | 15-30% | 深度分析 |

### 5.3 Profiler 互斥问题

**关键限制**：CUPTI Profiling API 与 Nsight Compute 不能同时使用。Nsight Compute 自身内部也使用 CUPTI 来收集性能计数器，因此在 Nsight Compute 附加到进程时，尝试初始化 CUPTI Profiling API 会返回错误  [(NVIDIA Developer Forums)](https://forums.developer.nvidia.com/t/profiling-cuda-graph-with-cupti-profiling-api/284956) 。

**应对策略**：

| 场景 | 方案 |
|------|------|
| 开发调试 | 使用 Nsight Compute GUI/CLI，不启用内建 profiler |
| CI 自动化 | Pipeline 分两个 stage：先运行 timeline trace（CUPTI Activity），再运行详细 profiling（Nsight Compute） |
| 生产监控 | 仅使用 CUPTI Activity API（低 overhead），禁用 Profiling API |
| 同时需求 | 使用 PC Sampling API（CUPTI 12.x 支持与 concurrent kernel tracing 同时使用） [(NVIDIA Developer Forums)](https://forums.developer.nvidia.com/t/cuda-profiler-tools-interface-cupti-for-cuda-toolkit-12-8-is-now-available/320718)  |

## 6. 编译器侧观测

新语言编译器的性能是开发者体验的关键组成部分。借鉴 rustc 和 Clang 的成熟实践，编译时间 telemetry 应当被设计为编译器的一等公民功能。

### 6.1 rustc self-profile 体系

Rust 编译器（rustc）内置了 `-Z self-profile` 选项，可输出详细的编译阶段时间分解  [(fasterthanli.me)](https://fasterthanli.me/articles/why-is-my-rust-build-so-slow) 。

#### 6.1.1 输出格式与工具链

rustc self-profile 的输出通过 `measureme` 工具链处理：

```bash
# 生成自分析数据
rustc -Z self-profile ./main.rs

# 生成文本摘要
cargo install --git https://github.com/rust-lang/measureme summarize
summarize summarize main-*.mm_profdata

# 生成火焰图
cargo install --git https://github.com/rust-lang/measureme flamegraph
flamegraph main-*.mm_profdata

# 生成 Chrome Tracing JSON
cargo install --git https://github.com/rust-lang/measureme crox
crox main-*.mm_profdata  # 输出 chrome_profiler.json
```

#### 6.1.2 数据内容

self-profile 输出按 rustc 内部的 **query system** 粒度记录时间  [(fasterthanli.me)](https://fasterthanli.me/articles/why-is-my-rust-build-so-slow) ：

| Query 类型 | 说明 |
|------------|------|
| `parse_crate` | 语法分析 |
| `expand_crate` | 宏展开 |
| `check_crate` | 类型检查 |
| `borrow_check` | 借用检查 |
| `codegen_crate` | 代码生成 |
| `link` | 链接 |

每个 query 记录开始时间、结束时间和调用次数，形成完整的调用树。

### 6.2 Clang -ftime-trace 框架

Clang 的 `-ftime-trace` 选项生成 Chrome Tracing 格式的 JSON 文件，可直接在 `chrome://tracing` 或 Speedscope 中可视化  [(Aras' website)](https://aras-p.info/blog/2019/01/16/time-trace-timeline-flame-chart-profiler-for-Clang/) 。

#### 6.2.1 使用方法

```bash
# 生成时间线 trace
clang++ main.cpp -c -ftime-trace -ftime-trace-granularity=50

# 输出 main.json，用 Chrome 打开 chrome://tracing 加载
```

`-ftime-trace-granularity` 参数控制最小记录时间（默认 500μs），低于此阈值的事件被忽略以减少数据量。

#### 6.2.2 事件层次

Clang 的 time-trace 按编译阶段组织事件  [(izaron.github.io)](https://izaron.github.io/posts/clang-compilation-speed/) ：

| 顶层事件 | 子事件示例 |
|----------|------------|
| `ExecuteCompiler` | Frontend, Backend, Linker |
| `Frontend` | `Parse*`, `ParseClass`, `InstantiateClass` |
| `Backend` | `OptFunction`, `RunPass`, `CodeGen Function` |
| `Source` | 按文件名的 include/parsing 时间 |

**关键洞察**：`-ftime-trace` 直接展示了"编译器在用户代码的哪一部分花了多少时间"，而非"编译器自身的哪个函数耗时"，这对优化用户代码的编译时间具有直接指导意义  [(Aras' website)](https://aras-p.info/blog/2019/01/16/time-trace-timeline-flame-chart-profiler-for-Clang/) 。

### 6.3 增量编译验收指标设计

新语言编译器的增量编译性能应当作为验收指标的一部分。推荐的指标体系如下：

| 指标 | 定义 | 目标 |
|------|------|------|
| **Clean build time** | 完全 clean 后的首次编译时间 | 不劣于 rustc/Clang 同等工作量 |
| **Incremental check time** | 修改单个文件后的 check 时间 | < 5 秒（开发体验阈值） |
| **Incremental build time** | 修改单个文件后的完整编译时间 | < clean build 的 20% |
| **CodeGen reuse ratio** | 复用前次 codegen 结果的比例 | > 80% |
| **Query cache hit rate** | 编译器 query 缓存命中率 | > 90% |

## 7. 性能回归门禁工程

性能回归门禁（Performance Regression Gating）是确保代码变更不引入性能退化的工程机制。本节从统计方法、阈值管理和业界实践三个维度展开。

### 7.1 回归检测的统计方法

#### 7.1.1 变化点检测（Change Point Detection）

变化点检测的目标是在时间序列中识别统计特性发生显著变化的时点。在性能回归场景中，即识别某次 commit 是否导致性能指标的分布发生偏移。

**EDM（EDivisive with Medians）算法**是推荐的变化点检测方法，其特点包括：

- **非参数**：不假设数据服从正态分布
- **鲁棒**：使用中位数而非均值，对异常值不敏感
- **多变化点**：可检测序列中的多个变化点
- **统计检验**：基于能量统计量（energy statistic）进行显著性检验

**Mann-Whitney U 检验**（又称 Wilcoxon rank-sum test）是另一种广泛使用的非参数检验  [(Sitespeed.io)](https://www.sitespeed.io/documentation/sitespeed.io/compare/) ，已被 sitespeed.io 等性能监控工具采用。其优势在于：

- 适用于不服从正态分布的性能数据
- 对小样本（n < 20）仍然有效
- 检测均值偏移和分布形状变化

**推荐流程**：
1. 收集基线 commit 的 30 次测量（通过多次 CI 运行或单次运行内的重复）
2. 收集待测 commit 的 30 次测量
3. 执行 Mann-Whitney U 检验
4. 若 p-value < 0.05 且效应量 > 0.3（Cohen's r），标记为潜在回归
5. 人工 triage 确认是否为真实回归

#### 7.1.2 噪声建模

性能数据的噪声来源包括：OS 调度、CPU 频率调整、缓存状态、后台进程等。有效的噪声建模是回归检测准确性的前提。

| 噪声类型 | 特征 | 处理策略 |
|----------|------|----------|
| **高斯噪声** | 对称分布，随机波动 | 均值/中位数平滑 |
| **异常值** | 偶尔的大幅偏离 | IQR 方法剔除  [(FasterCapital)](https://fastercapital.com/topics/using-interquartile-range-for-outlier-detection.html)  |
| **趋势** | 长期缓慢变化 | 去趋势（diff）后再检验 |
| **周期性** | 日内/周内周期 | 时间标准化 |

### 7.2 阈值管理策略

#### 7.2.1 三种阈值策略对比

| 策略 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **绝对阈值** | 固定数值（如 > 100ms） | 简单直观 | 不随硬件/负载变化 |
| **相对基线** | 与基线比较的百分比变化 | 自适应 | 基线漂移问题 |
| **统计检验** | Mann-Whitney U + p-value | 科学严谨 | 需要足够样本量 |

**推荐**：**混合策略**——使用统计检验作为初筛（ Mann-Whitney U, p < 0.05），结合相对阈值作为实际报警门槛（如变化 > 1%）。

#### 7.2.2 阈值层级设计

```
┌─────────────────────────────────────┐
│  Critical (> 5% regression)         │  → Block merge + notify team
├─────────────────────────────────────┤
│  Warning (1-5% regression)          │  → Flag for triage
├─────────────────────────────────────┤
│  Info (< 1% or improvement)         │  → Record, no action
└─────────────────────────────────────┘
```

### 7.3 业界案例：LLVM 与 rustc 实践

#### 7.3.1 LLVM Compile-Time Tracker

Nikita Popov（nikic）建立的 llvm-compile-time-tracker.com 是编译器性能回归检测的典范  [(nikic's Blog)](https://www.npopov.com/2020/05/10/Make-LLVM-fast-again.html) 。其核心设计包括：

- **测量基础**：CTMark（Compile Time Mark），LLVM test-suite 的子集，包含 7 个代表性程序
- **指标选择**：
  | 指标 | 稳定性 | 用途 |
  |------|--------|------|
  | **instructions** | 高 | 主要指标，proxy for compile time |
  | **max-rss** | 高 | 内存使用 |
  | **task-clock** | 低 | 参考 |
  | **wall-time** | 低 | 参考 |
  | **size-total** | 完全稳定 | 二进制大小 |

- **配置**：O3、ReleaseThinLTO、ReleaseLTO-g 三种 LTO 配置
- **自动化**：每次 commit 后自动运行，结果发布到网站
- **回归阈值**：约 0.3% 的变化即被视为值得关注

#### 7.3.2 rustc perf.rust-lang.org

Rust 编译器团队的性能基础设施是业界最成熟的编译器性能回归系统之一  [(Rust Compiler Development Guide)](https://rustc-dev-guide.rust-lang.org/tests/perf.html) ：

- **自动化 perf run**：每次 PR merge 后自动运行 benchmark suite

- **指标粒度**：
  - instructions:u（用户态指令数，最稳定）
  - wall-time、max-rss、cache-misses 等辅助指标
  - 分场景：check、debug、opt、incr-full、incr-patched 等
- **Triage 流程**：每周二进行 performance triage，汇总上周所有 PR 的性能影响  [(The Mozilla Blog)](https://blog.mozilla.org/nnethercote/) 

**关键经验**：instructions（通过 `perf` 采集）是最稳定的编译时间代理指标，其变异系数远低于 wall-time  [(nikic's Blog)](https://www.npopov.com/2020/05/10/Make-LLVM-fast-again.html) 。

## 交付物 B：运行时内建观测 API 草案

### 设计原则

1. **零成本抽象**：profiling 未启用时，所有调用为无操作（inline no-op）
2. **非侵入**：用户代码无需修改，运行时自动标注关键路径
3. **统一输出**：Chrome Tracing JSON 格式，兼容现有可视化工具
4. **可配置**：通过环境变量或配置文件控制 profiling 粒度和输出目标

```python
    """运行时性能观测器，控制 profiling 会话生命周期"""

        self.config = config
        self._active = False
        self._cupti_session = None
        self._nvtx_enabled = config.nvtx

        if self._active:
            return
        self._cupti_session = CUPTIActivitySession()
        self._cupti_session.enable([
            CUPTI_ACTIVITY_KIND_KERNEL,
```

## 交付物 C：性能回归门禁统计方案

### 方案概述

推荐采用 **"统计检验 + 相对阈值 + 噪声建模"** 的三层检测方案。

### 统计检测算法

```
算法: PerformanceRegressionDetect
输入: baseline_samples[1..N], candidate_samples[1..M]
输出: {status: PASS | WARNING | CRITICAL, details}

1. 异常值剔除:
   baseline_clean = IQR_Filter(baseline_samples)
   candidate_clean = IQR_Filter(candidate_samples)
   
2. 分布检验:
   statistic, p_value = Mann_Whitney_U(baseline_clean, candidate_clean)
   
3. 效应量计算:
   effect_size = Cohen_r(statistic, len(baseline_clean), len(candidate_clean))
   
4. 相对变化:
   median_change = (median(candidate_clean) - median(baseline_clean)) 
                   / median(baseline_clean)
   
5. 决策:
   if p_value >= 0.05:
      return PASS  # 统计不显著
   
   if abs(median_change) < 0.01:  # 1% 阈值
      return PASS  # 变化太小，忽略
   
   if median_change > 0.05:  # 5% 退化
      return CRITICAL
   
   if median_change > 0.01:  # 1-5% 退化
      return WARNING
   
   return PASS  # 改善
```

### 阈值配置

| 阈值 | 值 | 动作 |
|------|-----|------|
| **统计显著性** | p-value < 0.05 | 初筛条件 |
| **最小关注变化** | > 1% | 低于此值忽略 |
| **Warning** | 1% - 5% | 标记待 review |
| **Critical** | > 5% | Block merge |
| **效应量** | Cohen's r > 0.3 | 确认实际影响 |

### 基线管理

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| **Fixed Baseline** | 固定 release 版本作为永久基线 | 跨版本对比 |
| **Rolling Baseline** | 最近 N 个 main branch commit | 日常回归检测 |
| **Dual Baseline** | Fixed + Rolling 同时维护 | 综合场景 |

### 实施架构

