# Windows 原生工具链、分发与许可合规调研

## 结论先行

如果你的新语言**只面向 Windows + NVIDIA**，并且编译产物是标准 `DLL/EXE`（可选 `PYD`），我建议把全链路默认方案定为：

| 链路阶段 | 建议基线 | 原因 |
|---|---|---|
| 编译目标 | 直接产出 **COFF `.obj`**，遵守 **Windows x64 ABI**，最终生成 **PE DLL/EXE/PYD** | Windows 原生链接器与调试器的默认消费对象就是 COFF/PE；Windows x64 上实际只有一套主 ABI 约束，最适合与 MSVC/Windows 调试生态对齐。citeturn43search0turn43search1turn43search12 |
| 链接器 | **开发/调试默认 `link.exe`**；`lld-link` 作为可选“快路径”而不是默认基线 | `link.exe` 对 `/INCREMENTAL`、`.ilk`、PDB、MSBuild/VS 工程集成都是原生道路；在本次资料集中，我核实到了微软对 `link.exe` 的增量链接与输出物文档，但没有完成当前 `lld-link` 与其在 Windows 增量链接语义上的等价性核实，因此保守基线应以 `link.exe` 为主。citeturn43search2turn43search6turn43search22 |
| 调试信息 | 生成 **CodeView + PDB**；调试构建优先保证 VS / WinDbg 可读 | Microsoft 官方 PDB 访问接口是 DIA SDK；Visual Studio 调试器还围绕 debug engine、expression evaluator、symbol provider 组织扩展点，因此“标准 PDB 先跑通”是最低阻力路径。citeturn8search0turn9search12turn9search19turn9search16turn9search18 |
| CUDA 协作 | 以 **MSBuild `.props/.targets` + `$(CUDA_PATH)`** 为主；编译器自行探测 Toolkit、驱动、GPU 能力并给出诊断 | CUDA for Windows 官方文档明确给出 VS/MSVC 支持矩阵、Build Customizations 机制和 `CUDA_PATH`；自 CUDA 13.1 起 Windows 驱动需单独安装，编译器必须把“装了 Toolkit 但没装/没达标驱动”识别出来。citeturn13view0turn14view2turn11view1turn17view2turn18search4 |

| SBOM | 构建时生成 **SPDX**，对外分发时可再导出 **CycloneDX** | 微软自己的 SBOM 工具与 vcpkg 都围绕 SPDX；而 Windows 安全供应链实践也强调 SBOM。CycloneDX 在下游安全工具链里也很常见。citeturn24search2turn19search16turn19search6turn23search1turn23search2 |

对你的具体产品形态，我会建议把“**语言编译器/包管理器**”和“**最终用户运行时**”分开发布：前者做成 **rustup/juliaup 风格的版本管理器 + MSI/winget**，后者做成**最小运行时包**，只在确有必要时再带上可再分发的 NVIDIA 运行库，而不是把整个 CUDA Toolkit 打包进去。这样最有利于 ABI 稳定、签名治理、SBOM 追踪和许可合规。citeturn42search8turn42search1turn35search3turn26search6turn30view0turn34view0 |

## Windows ABI、链接与调试接入

Windows x64 的关键点是：**在 x64 上基本只有一套主调用约定**；前四个参数优先走寄存器，其余参数走栈，ABI 还严格约束了栈、寄存器使用、prolog/epilog 和展开信息。这意味着你的语言如果想无缝接入 Windows 生态，最重要的不是发明自己的宿主 ABI，而是**把默认外部 ABI 固定到 Microsoft x64 ABI**，把语言内部调用约定隐藏在编译器内部。citeturn43search0turn43search4turn43search12 |

Windows 的对象文件与最终镜像分别是 **COFF** 与 **PE**。微软官方 PE/COFF 文档把它们定义为 Windows 下对象文件和可执行镜像的标准结构；`LINK.exe` 负责把 COFF `.obj` 与库文件链接成 `EXE` 或 `DLL`。因此，你的编译器后端只要稳定地产出标准 COFF，就能与 MSVC 生态、Windows SDK import libs、第三方 `.lib`/`.obj` 直接协作。citeturn43search1turn43search10turn43search22 |

对外 C ABI 建议只暴露三种产物面：一是普通 `DLL` 的导出函数；二是自动生成的 import library；三是可选的 `PYD`。在 Windows 上，链接器提供 `/IMPLIB` 来控制 import library 名称，`EXPORTS`/`/EXPORT` 用来定义导出。工程上最稳妥的做法是：**对公共 API 统一用 `.def` 或显式导出属性建模，始终同时产出 `xxx.dll + xxx.lib + xxx.pdb`**；如果要给 Python 扩展模块，则把 `PYD` 视作一种特殊 DLL 产物，签名与调试都按 DLL 处理。citeturn43search2turn43search18turn43search5 |

增量链接方面，微软官方文档明确：`/INCREMENTAL` 控制增量链接，增量模式会生成 `.ilk` 数据库以支持后续更快的重链。对一门新语言来说，这非常适合 **IDE 本地开发/调试构建**；而 CI、可复现构建和正式发行版则更适合关闭增量链接，换取更稳定的二进制布局与更可预测的符号结果。citeturn43search2turn43search6 |

PDB 方面，现状不是“完全黑盒”，但也不是单一、完整、官方语言规范那样的体验。微软提供的是 **DIA SDK**，它把 `.pdb` 作为主要调试数据来源；另一方面，LLVM 社区已经维护 Windows/PDB 支持文档和实现，工程上说明这个格式已经公开到足以被第三方工具链实现读写兼容。对你而言，重要结论不是“PDB 是否学术意义上的完全公开”，而是：**想让 VS / WinDbg / 其他 Windows 调试器原生工作，你就应当输出标准 CodeView/PDB，而不是自造调试格式。** citeturn8search0turn7view4 |

Visual Studio 想做到“像 C++ 一样自然地看变量、单步、监视表达式”，需要分层处理。**第一层**是让调试器能认符号与源位置，这一层靠 PDB；**第二层**是让监视窗口理解你的语言表达式与类型，这通常会触及 Visual Studio 的 **debug engine、expression evaluator、symbol provider** 扩展点。换句话说：**只发 PDB 可以拿到符号化、断点、调用栈、源码行号；但若想把语言级 watch / immediate / 条件断点体验做“原生”，通常还要做 VS 调试扩展。** citeturn9search12turn9search19turn9search16turn9search18 |

Natvis 则是“第三层体验增强”。它不是替代 PDB，而是**在已有符号之上定义可视化规则**。WinDbg 官方文档明确支持加载 NatVis 文件，自定义原生对象显示。对你的语言来说，最值得做的是给**标准库容器、字符串、切片、张量视图、错误对象**提供一套随 SDK 分发的 `.natvis`。这样即便暂时没有完整的 VS 语言级 expression evaluator，开发者在 VS / WinDbg 中也能“看起来像原生对象”。citeturn43search3turn43search7turn43search23 |

## CUDA Toolkit 与 Visual Studio 工程协作

截至当前最新的 Windows 安装文档，CUDA 13.3 在 Windows 上支持的编译器矩阵是：**MSVC 195x / Visual Studio 2026 18.x**、**MSVC 193x / Visual Studio 2022 17.x**、**MSVC 192x / Visual Studio 2019 16.x**；并且 **32 位原生/交叉编译从 CUDA 12.0 起已移除**。如果你的语言只面向 Windows x64 + NVIDIA，那么没有任何理由为 x86 再投入编译器前端、运行时或测试资源。citeturn13view0turn13view1turn13view2turn14view0 |

另一个会直接影响用户安装体验的现实变化是：**从 CUDA 13.1 开始，Windows 显示驱动不再随 Toolkit 打包**，用户必须单独下载并安装合适驱动；CUDA 13.x 兼容文档和 13.3/13.1 发布说明都明确了这一点，并且在对应驱动表中把 Windows Toolkit Driver Version 标成 `N/A`。这意味着你的安装器/编译器再也不能假设“装了 Toolkit 就一定有合格驱动”。citeturn10search0turn11view1turn11view3 |

在 Visual Studio / MSBuild 集成上，Microsoft 的 C++ 项目系统和 MSBuild 允许你通过 `Directory.Build.props`、`Directory.Build.targets`，以及 C++ 专用的 `ForceImportBefore/AfterCppProps/Targets` 扩展点插入平台自定义规则。NVIDIA 的 Windows 安装文档则说明了 CUDA 的 **Build Customizations** 流程：通过 `.props` 文件把 CUDA 规则导入到 `.vcxproj`，并通过 `$(CUDA_PATH)` 指向安装目录。对新语言而言，这意味着你可以把自己的 Windows SDK 适配层做成两层：**一层是独立 CLI/包管理器；另一层是 VS/MSBuild 可导入的 `.props/.targets` 适配包**。citeturn10search2turn10search6turn11view2turn13view0turn14view2 |

值得特别指出的是，**CUDA 13.3 的安装指南正文仍然多处写着 `CUDA 13.2.props`、`CUDA 13.2 Runtime` 模板名称和路径说明**。这不是小事，它说明你不能把版本探测逻辑硬编码成“文档页版本号 = `.props` 文件名版本号”。更稳妥的工程实现是：**枚举 VS 的 `BuildCustomizations` 目录，基于实际文件存在性与 `CUDA_PATH` 环境变量做探测，而不是猜文件名。** citeturn14view1turn14view2 |

建议你的编译器把 CUDA 环境探测拆成三类诊断：

第一类是 **Toolkit 探测**。优先检查 `CUDA_PATH`，其次检查 VS `BuildCustomizations` 目录，最后允许用户通过显式命令行参数覆盖。NVIDIA 文档已经说明 `$(CUDA_PATH)` 由安装器设置，样例工程也依赖它。citeturn13view0turn14view2 |

第二类是 **驱动与设备探测**。官方 `nvidia-smi` 文档明确说：其输出格式不保证向后兼容，而 **NVML** 与其 Python 绑定则保证向后兼容，并且是更适合长期维护工具的首选；NVML 还提供 `nvmlSystemGetDriverVersion`、`nvmlSystemGetCudaDriverVersion_v2`、`nvmlDeviceGetCount_v2`、`nvmlDeviceGetCudaComputeCapability` 之类的系统与设备查询接口。你的编译器/语言服务器最好**优先走 NVML / CUDA Runtime API，`nvidia-smi` 仅作为人工诊断后备**。citeturn17view2turn17view0turn17view1turn18search12turn18search4 |

第三类是 **兼容性判定**。不要只输出“找不到 CUDA”；要把错误区分成“没装驱动、驱动版本不够、找到 Toolkit 但找不到对应 `nvrtc`/`libdevice`、检测到 GPU 但 compute capability 不满足目标架构、Windows 上装了 13.x Toolkit 却还以为驱动会被自动装上”。NVIDIA 的兼容文档、驱动发布说明和 compute capability 表都足以支撑这种细粒度诊断。citeturn17view3turn11view1turn16search14 |

## 发布工程：签名、SBOM 与安装器

在 Windows 生态里，**Authenticode 代码签名不是锦上添花，而是上线门槛**。微软文档明确：Authenticode 通过证书链验证发布者身份与代码完整性；SignTool 是官方签名与时间戳工具。因此，你至少要把以下对象纳入签名范围：编译器主程序、语言运行时 DLL、每个可执行工具、SDK 内的 Python 扩展 `PYD`、MSI/MSIX/安装器 EXE。时间戳也应视为强制项，这样证书过期后历史构件仍可继续验证。citeturn22search8turn19search13turn19search3turn19search22 |

证书策略上，2026 年的微软官方口径已经和很多旧经验不同了。微软在“SmartScreen reputation for Windows app developers”和“Code signing options”文档里明确写到：**2024 之后 EV 证书已经不再自动绕过 SmartScreen；EV 与 OV 都要经历同样的信誉积累过程。** 因此，如果你不是做驱动/HW Dev Center 那类仍有 EV 特定流程要求的场景，那么**买 EV 证书的核心理由不再是 SmartScreen，即时信誉这件事已经不存在了。** citeturn20search1turn20search2turn20search14 |

对公开分发的 Windows 软件，微软还明确指出：**自签名证书不适合公共发布**，因为默认不被信任，并会触发强烈的 SmartScreen 阻断。对于你的项目，比较现实的选项只有两条：其一是传统 **OV 代码签名证书**；其二是微软推荐给非 Store 分发开发者的 **Azure Artifact Signing**。后者是托管式签名服务，仍通过 SignTool 接入，但当前仍处于公开预览阶段，而且微软也明确说它**不会给你即时 SmartScreen 信任**。citeturn19search5turn22search3turn22search4turn22search7turn19search14 |

成本方面，公开页面能核实到的**示例公开标价**大致是：GlobalSign 代码签名证书页面显示 **1 年期从 434 美元/年起**；DigiCert 的公开购买页与对比页显示标准代码签名与 EV 代码签名的公开订阅价大致在 **696 美元/年、972 美元/年** 量级，而带 KeyLocker/HSM 托管方案更高。另一个你必须计入预算的现实变化是，DigiCert 文档指出**代码签名私钥必须保存在符合要求的硬件中**，所以即使买的是证书，也往往会连带 token/HSM/托管签名流程成本。由于证书价格会变，下面应理解为**2026-06-10 的公开页面量级**，不是报价承诺。citeturn21search2turn21search1turn21search7turn21search19turn21search4 |

SBOM 方面，如果你面向企业客户或要做供应链合规，建议把 **SPDX 作为主格式、CycloneDX 作为兼容输出**。原因很简单：微软自己的 vcpkg SBOM 与 `microsoft/sbom-tool` 都围绕 SPDX；SPDX 本身已是 ISO/IEC 5962:2021 国际标准；CycloneDX 则是 OWASP 侧很主流的 BOM 标准。最实用的做法不是二选一，而是**构建时先生成 SPDX，发布时转换/补充 CycloneDX，并把二者作为安装包旁路产物上传与归档**。citeturn19search16turn24search2turn23search1turn23search4turn23search2turn23search5 |

安装器选择上，建议不要追求“一个格式打天下”，而是按场景分层：

| 场景 | 推荐 | 原因 |
|---|---|---|
| 编译器/SDK/CLI 工具链 | **MSI + winget** | Windows Installer 仍是桌面/企业部署标准通道，支持事务与回滚；winget 能直接分发 MSI/EXE/MSIX/Inno/NSIS 等多种安装器，适合版本化发布。citeturn25search0turn25search4turn26search0turn26search6 |
| 终端用户桌面应用 | **EXE/MSI 为主**，必要时补 **MSIX** | MSIX 的优势是 package identity、干净安装、更新和 Windows 特性接入；但很多本地原生工具链、驱动联动、复杂 PATH/脚本/后装步骤场景仍更自然地落在 EXE/MSI。citeturn19search2turn25search6turn25search2 |
| 面向个人用户的轻量安装器 | **Inno Setup** 可作为引导层 | Inno Setup 官方仍在积极维护，对 x64/Arm64 与较广泛 Windows 版本支持很好，适合做“下载器/引导器/一键安装”。citeturn25search1turn25search5turn25search13 |

综合下来，我对你的项目的发布建议是：**工具链采用 MSI + winget；最终应用采用签名 EXE/MSI；只有明确需要 package identity、自动更新或商店/企业统一分发时，再提供 MSIX 变体。** 这比“全部 MSIX 化”更契合原生编译器、PATH、SDK、插件和命令行工作流。citeturn25search0turn19search2turn26search0 |

## NVIDIA 再分发许可矩阵

先给出这部分最关键的原文抓手。CUDA Toolkit EULA 里，最重要的是四个短语：**§2.2** 的 **“listed in Attachment A”**、**§1.1** 的 **“incorporated in object code format”**、**§1.2** 对 **“stand-alone product”** 的禁止；而 cuDNN 补充条款里，**§2** 明写可分发的是 **“the runtime files .so and .dll”**。这四句几乎决定了你能不能把某个 NVIDIA 组件塞进自己的安装包。citeturn30view0turn32view3turn32view1turn34view0 |

把通用条件先固定下来，后面的矩阵就容易读了。对你要分发的应用，至少要满足：**应用本身必须有实质附加功能**、**可分发的 SDK 部分只能由你的应用访问**、**除非文档明确标为可分发，否则开发者工具仅限内部使用**、**你的分发条款要与 NVIDIA 协议一致**、并且**不能把 SDK 当作独立产品单卖或单独转授权**。这些要求在通用 SDK 协议与 cuDNN 页面都写得很直白。citeturn34view0turn32view1 |

下面的矩阵只回答一个工程问题：**一门第三方语言/运行时，在 Windows 上把 NVIDIA 组件打进自己的分发包，边界在哪里。**

| 组件 | Windows 侧是否可再分发 | 条件 | 依据 |

| `cudart.dll` | 可以 | 受 CUDA EULA 通用条件约束；它在 Attachment A 的 CUDA Runtime/Windows 列表中。citeturn30view0 | CUDA EULA §2.2、§2.6 Attachment A. citeturn30view0 |
| `cudart_static.lib` / `cudadevrt.lib` | 可以 | 仅在你的应用目标代码里使用并随应用分发，不应把它们当独立 SDK 再卖。citeturn30view0turn32view1 | CUDA EULA §1.1、§1.2、§2.6. citeturn30view0turn32view3turn32view1 |

基于这些条款，我对“**第三方语言把 NVIDIA 库打进自己的分发包**”的工程化合规判断是：

## 工具链分发先例、风险成本与建议落地

Windows 上原生工具链分发，大体有四种范式。

**Rust** 是“**版本管理器优先**”范式：官方安装页让 Windows 用户下载 `rustup-init.exe`，Rust 书与 Cargo 文档都说明 rustup 是安装器兼版本管理器；在 Windows/MSVC 路线上，它还会提示安装 Visual Studio，因为需要 linker 和原生库。与此同时，Rust 官方仍保留单版本的独立 MSI 作为离线/固定版本安装路径。对你的语言来说，这是最值得借鉴的模式。citeturn42search1turn42search8turn42search6turn42search13turn42search4 |

**Zig** 是“**便携压缩包优先**”范式：官方直接提供 Windows zip，建议用户解压后把目录加到 PATH。这种模式非常适合单一自举编译器和可移动工具链，但它的企业治理、签名、升级、依赖发现和与 Visual Studio 的工程集成通常都比 rustup/msi 路线更弱。换句话说，Zig 适合“一个独立工具”，不太适合“一个希望完全进入 Windows 企业工具链与安装治理体系的语言平台”。citeturn40search1turn40search5turn40search13 |

**Julia** 的官方页现在把 **Juliaup** 作为主安装入口；而官方社区项目负责人在 Windows 安装讨论里明确说过 Juliaup 的 Windows 安装器基于 **MSIX / App Installer / Microsoft Store** 思路。对你来说，Julia 的启发不是“学它用 Store”，而是“**把语言本体与版本管理器合并为一个可升级的安装入口**”，然后再按企业环境提供非 Store 路径。citeturn35search3turn35search8turn35search7 |

**Mojo** 则属于“**环境/包管理器优先**”范式：官方文档明确推荐 `pixi` 或 `uv`，并且当前 **Windows 仅支持 WSL，不支持原生 Windows**。因此它对你的参考价值主要在“包与环境管理体验”，而不是 Windows 原生工具链集成。你的目标既然是原生 Windows + NVIDIA，那么 Mojo 不是等价先例。citeturn41search0turn41search1turn41search3turn41search5 |

## 开放问题与局限

本轮资料中，**`link.exe` 的增量链接/PDB/MSBuild 路线**、**CUDA/ cuDNN 再分发边界**、**SmartScreen 现行行为**都已经能给出高置信结论；但有两点我建议你在工程立项前再做一次定向验证：一是当前版本 **`lld-link` 在你的目标场景里对 PDB、增量构建和 VS 调试体验的实际 parity**；二是你是否要为 **VS Code** 单独投入“语言级 watch / evaluator / DAP”体验，而不仅仅依赖标准 native 符号。另有一点要特别说明：**Mojo 当前并不支持原生 Windows，仅支持 WSL**，因此它只能作为“包管理/自举方式”的参考，不是“Windows 原生工具链”的对照组。citeturn41search5turn43search2turn43search6turn9search12turn9search16 |
